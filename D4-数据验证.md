# ODPlatform D4: data_validation 子系统

> **课程定位**: 实训第 4 天讲义
>
> **核心叙事**: 从一个干净的 D3 仓库出发, 构建一个完整的【数据质检子系统】 `data_validation/`——把"D3 产出的 YOLO yaml + 一行命令"变成"**14 项检查 + 一份可签字的质量报告 + 一张逐实例整改清单 + 一个 CI 能直接消费的退出码**"。
>
> **五面真撞墙**:
> 1. **一个 validate() 函数串到底, 第 N 个 check fail 阻断后面所有 check** → 推出注册表 + `@check` 装饰器 + CheckResult 聚合
> 2. **每个 check 自己扫一遍盘, 14 个 check 把磁盘扫成筛子** → 推出 DatasetSnapshot 一次扫描多次复用 + "顺手统计"
> 3. **跑完只剩 `[PASS] yaml_schema` 这种干瘪日志, 没人知道数据集长什么样** → 推出 ValidationReport(纯数据) + render(纯展示)分离
> 4. **全部检查绿灯, 训练照样废掉**——"每个字段都合法 ≠ 整体合法"的几何盲区 → 推出第二批 check (出图框 / 退化框 / 重复标注)
> 5. **想把统计塞进 report.json, 6000 个实例直接把报告撑爆** → 推出画像层 profiler 双轨设计 (报告放聚合, CSV 放明细)
>
> **配套上一节**: 《D3: data_pipeline 子系统》(D3 生产数据, D4 质检数据)
>
> **配套下一节**: 《D5: 训练子系统》(D4 守好数据这一关, D5 才能开始训)
>
> **两条设计原则, 全文反复回响**:
> 1. **只检测, 不修复**——发现问题归 data_validation, 修复问题归 data_pipeline 重跑产线。两件事混在一起, "我刚跑了一次 validate, 数据是不是变了?"这种基本问题都说不清楚。
> 2. **判定与画像分离**——check 回答"能不能训"(产 severity), profiler 回答"长什么样"(产统计, 永不阻断)。混在一起, 统计代码的一个 bug 就能把放行决策搞挂。
>
> **建议时长**: 10-14 课时 (体量约为 D3 的 1.5 倍, 但后半段 check 高度同构, 越写越快)

D4 的产物有两类: **代码**和**对代码的解释**。

**强烈建议**: **跟着文档自己敲一遍**——D4 是注册表模式 + 多角色协作的综合演练, 不亲手过一遍, 看到第 6 个 check 时就会糊涂"谁调谁、数据从哪来"; 看到 profiler 时就会糊涂"这跟 check 有什么区别"。

## 开篇: 为什么企业一定要做数据验证

写第一行代码之前, 先把这件事在企业里的**位置和价钱**讲清楚——不讲清楚这个, D4 全程都像在"为了架构而架构"。

### 三个真实事故形态 (全部是"静默错误")

**事故一: 出图框。** 标注外包交付 5 万张图, 其中 8% 的框是 `cx=0.93, w=0.30`——四个坐标各自都在 [0,1] 里, 任何"逐字段范围检查"都放行。训练框架加载时悄悄把框 clip 回图内, **框被改了形状, 没有任何报错**。模型上线后小目标召回率比 POC 掉 11 个点, 算法团队查了两周模型结构, 最后发现是数据。两周 × 3 个算法工程师, 这就是一次静默错误的价钱。

**事故二: 划分泄露。** 同一段视频抽帧, 相邻两帧一张进了 train 一张进了 val。验证集指标虚高 7 个点, 大家按这个指标拍板上线——真实场景一上就翻车。**最贵的不是训练费, 是基于假指标做出的所有决策。**

**事故三: 类别在 train 里压根没出现。** 类别表声明了 `head`, val 里有 30 个 head 实例, train 里 0 个。评估报告里 head 的 AP 是 0, 评审会上结论是"模型对小目标不行, 换 backbone"。换了三个 backbone 之后才有人去数了一遍训练集。**数据缺口被误读成模型缺陷, 是企业里最常见、最贵的误诊。**

三个事故的共同点: **没有一个会报错**。代码全绿, 日志全绿, 训练曲线正常下降——错的只有最后的指标和基于指标的决策。这类错误业内叫 **silent failure (静默错误)**, 它是数据验证子系统存在的全部理由。

### 算一笔钱的账

一次中等规模的检测训练: 8 卡 × 30 小时。按云价折算加上 3 个算法工程师两天的排查时间, 一次"废训练 + 误诊断"的直接成本进万元很轻松——而**间接成本(基于假指标的产品决策、给客户的承诺)往往是直接成本的十倍**。

D4 这个子系统跑一次: 540 张图的数据集 **0.2 秒**, 5 万张图分钟级。**这是整个平台里投入产出比最高的子系统**——它不产生任何模型能力, 它只负责让别的投入不打水漂。

### 数据验证在交付流程中的位置: 质量闸门

```
  D3 data_pipeline          D4 data_validation              D5 训练
┌─────────────────┐      ┌─────────────────────┐      ┌──────────────┐
│ 转换/划分/落盘   │ ───▶ │  14 项检查 + 报告    │ ───▶ │  ultralytics │
│ (生产数据)       │      │  (质检数据)          │      │  (消费数据)   │
└─────────────────┘      └──────────┬──────────┘      └──────────────┘
                                    │ exit code
                          0 放行 / 1 有条件放行(人工确认) / 2 阻断
```

企业语境里它有个正式名字: **质量闸门 (quality gate)**。闸门的三个企业属性, 决定了 D4 后面所有的架构选择:

1. **闸门必须给出可执行的决策, 不是一堆数字。** 所以 D4 的产出不是"统计报表", 是 **PASS/INFO/WARNING/ERROR 四级判定 + 0/1/2 退出码 + A/B/C/D 评级**——CI 读退出码, 审批人读评级, 各取所需。
2. **闸门必须可审计。** 三个月后客户问"当时这批数据验过吗? 验的是哪份数据? 谁放行的?"——所以报告里要有**数据指纹、工具版本、操作人、签字栏**。审计字段不是官僚主义, 是"出了事能定位到环节"的工程保险。
3. **闸门拦下的问题必须可整改。** 报告说"有 37 个出图框"而给不出是哪 37 个, 标注团队就只能全量重审——所以要有 **instances.csv 逐实例明细**, 报告里任何数字都能下钻到具体文件和行号。

> **金句**: **"质检子系统的客户从来不止算法工程师一个——CI 要退出码, 审批人要结论, 标注团队要整改清单, 审计要指纹。D4 的全部架构, 就是同时伺候好这四个客户。"**

### D4 跟 D3 的关系: "生产数据" → "质检数据"

| | D3 data_pipeline | D4 data_validation |
|---|---|---|
| 职责 | **生产**: 转换标注格式 + 划分数据集 | **质检**: 检查产出物能不能进训练 |
| 副作用 | 写盘 (生成 YOLO 数据集) | **零写数据** (只读数据集, 只写报告) |
| 失败语义 | 互斥分发: 选错 converter 立刻抛 | 聚合执行: 单项失败**绝不阻断**其余检查 |
| 注册表 | `@register(format)` 按格式分发 | `@check(name)` 全量聚合 |
| 产物 | 数据集目录 | report.json / report.md / *.csv + 退出码 |

同一个注册表 idiom, 两种完全不同的调度承诺——这是 D4 最重要的模式课, 阶段 2 展开。

**"只检测不修复"再强调一次**: D4 对数据集是**只读**的。发现出图框, D4 报告它, 但绝不顺手 clip——质检员改产品, 这条产线的责任边界就碎了。修复永远走 data_pipeline 重新生产, 或独立的清理工具 (TODO D4.10)。

# 阶段 0: 立规矩 (paths/constants 增量)

跟 D3 阶段 0 一样, 本阶段不写业务代码, 只立规矩。

## 0.1 子系统目录约定: 框架 vs 实现 物理分离

```
data_validation/
├── registry.py / snapshot.py / service.py / ...   ← 框架 (改它 = 大事)
└── checks/                                         ← 实现 (加文件 = 日常)
```

⚠️ **设计点 (为什么物理目录反映开闭原则)**: 不是为了"目录好看", 是为了**让代码审查的人一眼看出"这次提交在改框架还是加实现"**。框架变动影响所有 check, 实现变动只影响一个 check——放不同目录, 一行 `git diff --stat` 就能区分。D4 终态 14 个 check, 这条约定的价值会被验证 14 次。

## 0.2 paths.py 增量 (本节只新增 1 项)

```python
# ============================================================
# D4 增量: 数据验证运行目录
# ============================================================
VALIDATION_RUNS_DIR: Path = RUNS_DIR / "data_validation"

def validation_run_dir(run_id: str) -> Path:
    """单次验证运行的产出目录: runs/data_validation/<run_id>/"""
    return VALIDATION_RUNS_DIR / run_id
```

⚠️ **设计点 (按需创建 vs 启动创建)**: 不加进 `get_dirs_to_initialize()`——运行目录"不存在"本身就是信息 ("还没跑过验证")。谁要写, 谁 `mkdir(parents=True, exist_ok=True)`。

**为什么每次运行一个独立目录?** 这是企业可追溯性的物理基础: 三个月内的每一次验证结果都原样躺在盘上, 第 14 阶段的"趋势对比"功能就是直接读上一次运行目录实现的——**先把目录规矩立对, 后面的功能才有地方长**。

## 0.3 constants.py 增量: 阈值集中管理的纪律

D4 是个**阈值密集型**子系统——"缺多少标签算严重""失衡到几倍要警告"全是数字。规矩一句话:

> **所有判定阈值进 `constants.py`, check 代码里零硬编码数字。**

为什么这条纪律在企业里值钱:

1. **阈值是业务决策, 不是实现细节。** "缺 5% 标签算 WARNING"是质量负责人定的, 不是写代码的人定的——它该待在一个非工程角色也能 review 的地方, 而不是埋在某个函数第 73 行。
2. **阈值要进报告。** 第 14 阶段的企业报告要求每个结论标注口径 ("失衡比 ≥ 50x → WARNING")——阈值散在各文件里, 这件事就做不到。
3. **为配置化留位。** TODO D4.1 计划支持"数据集级阈值覆盖"(医疗数据集容忍度和安防不同)——前提是阈值有唯一出处可覆盖。

本阶段先立第一批 (pair_existence 的分级阈值), 后续每个阶段引入新 check 时**随用随立**, 终态 18 个, 附录 B 有全表速查:

```python
# ============================================================
# D4 增量: 数据验证 (data_validation 子系统)
# ============================================================

# ---- pair_existence 缺失比例分级 ----
PAIR_MISSING_ERROR_RATIO: float = 0.5   # >= 50%: ERROR (流程级问题)
PAIR_MISSING_WARN_RATIO:  float = 0.05  # 5% ~ 50%: WARNING
```

❌ **反模式**: 在 check 里写 `if ratio >= 0.5:`——三个月后没人记得 0.5 是谁定的、能不能改、改了影响谁。

**重要**: 任务类型 (`Task.DETECT/SEGMENT`) / 图像扩展名 (`IMAGE_EXTENSIONS`) D3 已经立了, **D4 直接复用, 不重立**——SSoT 原则。不要手痒在 data_validation 里再立一个 `TaskType`。

# 阶段 1: 朴素第一版——一个 validate() 函数串到底

## 1.1 任务设定

先做最小闭环: 给一个 YOLO 数据集 yaml, 检查 4 件事——

1. yaml 字段一致 (nc / names / path 齐不齐)
2. 每张图都有对应标签文件
3. 标签行格式合法 (5 字段 / 坐标在 [0,1])
4. train/val/test 之间没有重名图像 (数据泄露)

⚠️ **写在代码前——朴素 != 偷工减料**: 阶段 1 的代码是**故意写歪的教学样本**, 它会在 1.4 节撞墙。但"先写朴素版再撞墙"不是表演——**它是你在任何新领域的真实第一版**。区别只在: 受过训练的工程师撞墙撞得早 (写 4 个检查就闻到味), 没受过的撞得晚 (写到 14 个才发现重构不动)。

拿一个真实的脏数据集跑 (缺 3 个标签 + 2 行格式错 + 1 对跨划分重名):

**就这一行。** 然后进程退出, 返回 False。

## 1.4 🧱 撞墙①: 串到底的三宗罪

**第一宗罪: 早 return 阻断。** 数据集里明明还有 2 行格式错和 1 对泄露, 但用户只看到第一个问题。修一个 → 重跑 → 看到下一个 → 再修……**修一轮跑一轮, 5 个问题跑 5 遍**。企业里这叫"挤牙膏式质检", 标注团队最恨的就是它——返工单一次给全, 和一次给一条, 整改周期差一个量级。

**第二宗罪: bool 装不下严重程度。** "缺 1 个标签"和"缺 60% 标签"在这个函数里都是 `return False`。可前者该放行 (标注遗漏, 工程可容忍), 后者该阻断 (流程级事故)——**同样是"不通过", 紧张度天差地别**, bool 没有这个维度。

**第三宗罪: 加检查 = 改这个函数。** 第 5 个检查往哪加? 继续往函数里塞。第 14 个呢? 这个函数会长到 600 行, 每次改动都要 review 全函数, 多人并行加检查必然 merge 冲突——**它把"加一项检查"这种日常小事, 变成了"动核心函数"这种大事**。

## 1.5 想清楚解药的形状

三宗罪对应三味药, 这就是阶段 2 的设计输入:

| 罪 | 药 |
|---|---|
| 早 return 阻断 | **聚合执行**: 每个检查独立函数, 调度层跑全部、收全部, 谁也阻断不了谁 |
| bool 太瘦 | **四级 severity**: PASS / INFO / WARNING / ERROR + 统一返回结构 CheckResult |
| 加检查改核心 | **注册表 + 装饰器**: 加检查 = 加一个文件, 框架零改动 (开闭原则) |

阶段 1 的代码不 commit, 直接进阶段 2。

# 阶段 2: 注册表机制——D4 子系统的骨架

## 2.1 先想清楚: registry.py 要装什么?

registry.py 是 D4 的**合同层**——它定义"什么是一个 check"。终态五样东西:

1. `CheckSeverity` —— 四级严重程度 + 排序
2. `CheckResult` —— 单个 check 的统一返回类型
3. `ValidationOptions` —— 一次运行的开关集合
4. `CheckContext` —— check 函数的入参合同 (扩展位)
5. `@check` 装饰器 + 模块级注册表 + 自动 import

**合同先行, 一次立全**: 五样东西现在全部写完, 哪怕 `ValidationOptions` 要到阶段 12 才有第一个消费者。理由很企业: **合同层改一次, 全体 check 作者都要重读一遍合同**——把"改合同"的次数压到 1, 是对 14 个 check 作者(可能是 14 个人)最大的尊重。实现可以分批兑现, 合同不行。

## 2.2 五个设计选择 (每个都有反面)

### 选择 1: `severity` 是四级字符串常量类, 不是 `bool passed`

阶段 1 已经撞过墙: "不通过"至少分三种紧张度——

- **ERROR**: 训练绝不能继续 (数据泄露 / 无训练数据)
- **WARNING**: 能继续但要人工 review (缺 30% 标签)
- **INFO**: 知道一下 (缺 0.1% 标签——影响极小)

加上 PASS, 四级一次到位。**这四级直接映射企业放行决策**: ERROR=阻断, WARNING=有条件放行(需负责人确认), INFO=放行附关注项, PASS=放行——阶段 14 的报告评级 D/C/B/A 就是它的另一套衣服。

⚠️ **为什么是字符串常量类不是 `enum.Enum`?** 字符串值跟 JSON 报告 / 日志 / 命令行最自然地对接, 不用到处 `.value`。`rank()` 方法补上排序能力 (`max(results, key=rank)` 一行算总评级)——enum 的全部好处里我们只要这一个, 那就只造这一个。

⚠️ **为什么保留 INFO 不砍成三级?** "缺 0.1% 标签"既不该 PASS (数据**确实**有问题), 也不该 WARNING (影响极小, 不应闹眼前)。砍掉 INFO, 这类"可容忍但该留痕"的信号无处表达。对齐 syslog / k8s events 的成熟分级, 后面接监控告警也不别扭。

### 选择 2: `details` 是 `Dict[str, Any]`, 不是 `List[str]`

check 发现的问题要带结构 (哪个 split / 哪个文件 / 第几行 / 越界多少)。`List[str]` 把结构碾成句子, 下游 (JSON 报告 / Markdown 表格 / 监控) 全都要反向解析字符串——**结构化数据降级成文本是单向门, 进去就回不来了**。Dict 进 JSON 零成本, 阶段 7 的 well-known keys 渲染模式也靠它。

### 选择 3: `passed` 是 `@property`, 不是存值

`passed` 完全由 severity 推导 (`severity in (PASS, INFO)`)。存两份就有第二份真相——总会有人只改一个。**派生属性永不存值**, 这条纪律在 ValidationReport 里还会用四次。

### 选择 4: 入参是 `CheckContext` 包装, 不是裸 `yaml_path`

所有 check 的签名钉死为 `(ctx: CheckContext) -> CheckResult`。为什么不直接传 `yaml_path`? 因为阶段 4 要加 snapshot, 阶段 12 要加 options——**裸参数每加一个, 全部 check 签名跟着改一遍**; 包一层 Context, 资源随便加, 签名永不动。这层包装就是给"未来一定会变"买的保险, 而且保费极低。

### 选择 5: `ValidationOptions` 是 frozen dataclass, 全字段带默认值

一次运行的行为开关 (要不要逐张解码图像 / 要不要生成画像)。两条硬约束:

- **frozen**: 运行中不可变——杜绝某个 check 偷偷改开关影响后续 check。14 个 check 共享一个 options 对象, 可变就是定时炸弹。
- **全默认值**: 不传 options 的老调用方行为零变化——向后兼容是合同层的义务。

⚠️ **开关与阈值分家**: options 只放**行为开关** (做不做), 阈值仍在 constants (做到什么程度算坏)。两者变更的角色不同——开关是使用者每次运行时选的, 阈值是质量负责人定期定的。

## 2.3 registry.py 完整代码

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  :registry.py
# @Time      :2026/6/10 10:21:42
# @Author    :雨霓同学
# @Project   :ODPlatform
# @Function  :
# apps/platform/src/od_platform/data_validation/registry.py
"""data_validation 注册表 + 数据契约 (CheckResult / CheckSeverity / CheckContext)。

跟 D3 的 data_pipeline/registry.py 是【同一模式的两种用法】:
    - D3: 互斥分发 (一次调用一个 converter)
    - D4: 聚合执行 (一次调用全部 check, 收集结果)

公开 API:
    @check(name)                 装饰器, 自动注册一个 check 函数
    CheckResult                  统一返回类型
    CheckSeverity                严重程度 (四级)
    CheckContext                 check 函数的入参合同 (扩展位)
    get_all_checks()             返回全部注册的 check (供 service 调度)
    get_check(name)              按名查询单个 check (供测试)
    list_check_names()           返回注册的 check 名列表
"""
from __future__ import annotations

import logging
from dataclasses import dataclass, field
from pathlib import Path
from typing import Any, Callable, Dict, List

from od_platform.data_validation.snapshot import DatasetSnapshot

logger = logging.getLogger(__name__)


# ============================================================
# 1. CheckSeverity — 严重程度 (四级一次到位)
# ============================================================

class CheckSeverity:
    """Check 结果的严重程度。

    跨级关系 (供 ValidationReport.overall_severity 比较):
        ERROR > WARNING > INFO > PASS

    四级而不是两级 (passed: bool) 的理由——见讲义阶段 2.2 选择 1。
    """
    ERROR   = "ERROR"     # 阻塞级 (CI 必须停, 训练绝不能继续)
    WARNING = "WARNING"   # 关注级 (能继续, 但需要人工 review)
    INFO    = "INFO"      # 告知级 (工程上"知道一下", 不阻断)
    PASS    = "PASS"      # 通过

    _ORDER = {"PASS": 0, "INFO": 1, "WARNING": 2, "ERROR": 3}

    @classmethod
    def rank(cls, level: str) -> int:
        """供聚合层比较 severity 用 — ERROR.rank() > PASS.rank()。"""
        return cls._ORDER.get(level, 0)


# ============================================================
# 2. CheckResult — 单个 check 的统一返回类型
# ============================================================

@dataclass
class CheckResult:
    """单个 check 的完整产出。

    Args:
        name:     check 名 (= 注册时的字符串, 跟 @check("name") 一致)
        severity: ERROR / WARNING / INFO / PASS (见 CheckSeverity)
        summary:  一句话总结, 供终端日志 / 报告头部使用 (给人看)
        details:  结构化详情字典 (给机器看 — JSON 报告 / 监控趋势)
    """
    name:     str
    severity: str
    summary:  str
    details:  Dict[str, Any] = field(default_factory=dict)

    @property
    def passed(self) -> bool:
        """便于聚合层判断是否算"通过"。

        PASS / INFO 都算通过 — INFO 的语义是"知道一下但不阻断"。
        """
        return self.severity in (CheckSeverity.PASS, CheckSeverity.INFO)


# ============================================================
# 3. ValidationOptions — 一次验证运行的开关集合
# ============================================================

@dataclass(frozen=True)
class ValidationOptions:
    """验证运行级开关 — 经由 CheckContext.options 下发给所有 check / profiler。

    设计:
        - frozen: 运行中不可变, 杜绝某个 check 偷偷改开关影响后续 check
        - 全部带默认值: 老调用方 (不传 options) 行为零变化 — 向后兼容
        - 这里只放"行为开关", 阈值仍在 constants.py (D4.1 配置化阈值的留位不变)

    Args:
        check_images:       True 时 image_integrity 逐张解码验证 (重型, 默认关)
        profile:            True 时生成实例画像 + instances.csv (轻型, 默认开)
        read_image_headers: True 时 profiler 读图像头取分辨率 (仅读 header 不解码,
                            单图 <1ms; 超大数据集可关以省 I/O)
        top_n_anomalies:    画像层异常锚点清单长度 (最小面积/极端宽高比 Top-N)
    """
    check_images:       bool = False
    profile:            bool = True
    read_image_headers: bool = True
    top_n_anomalies:    int  = 20


# ============================================================
# 4. CheckContext — check 函数的入参合同 (扩展位)
# ============================================================

@dataclass
class CheckContext:
    """Check 函数的入参 — 所有 check 函数签名都是 (ctx: CheckContext) -> CheckResult。

    本阶段先放最简单的字段:
        yaml_path : 数据集 yaml 文件路径
        snapshot: DatasetSnapshot - 扫描的产物，供给所有check使用
        options:  ValidationOptions - 运行级开关 (D4.x 增量, 带默认值保持兼容)
    阶段 4 引入 DatasetSnapshot 时, 这里会加 snapshot 字段 — 所有 check 改成消费
    snapshot, 不再自己扫盘。届时函数签名仍然不变, 只是 ctx 里的可用资源变多了 —
    这就是为什么我们把它做成一个对象, 而不是几个并列的位置参数。

    未来扩展(D4.x / D5+):
        - previous_snapshot: 历史比较
        - config_overrides:  数据集级阈值覆盖
        - executor:          并发上下文
    永远不改 check 函数签名 —— 这是 CheckContext 这层包装的全部意义。
    """
    yaml_path: Path
    snapshot: "DatasetSnapshot"
    options: ValidationOptions = field(default_factory=ValidationOptions)


# ============================================================
# 4. 注册表条目
# ============================================================

@dataclass(frozen=True)
class CheckEntry:
    """注册表里的一条记录。frozen — 注册后不可改, 一锤子买卖。"""
    name: str
    func: Callable[[CheckContext], CheckResult]


# ============================================================
# 5. 模块级注册表 + @check 装饰器
# ============================================================

_REGISTRY: Dict[str, CheckEntry] = {}
_LAZY_INITIALIZED: bool = False


def check(name: str) -> Callable:
    """注册一个 check 函数到全局注册表。

    用法:
        @check("yaml_schema")
        def validate_yaml_schema(ctx: CheckContext) -> CheckResult:
            ...

    设计:
        - 装饰器不包装函数 — 直接 return func。
          理由: 让被装饰的函数堆栈干净, 单元测试可以直接调用它而不经过装饰器,
          失败时的 traceback 也不会多一层"check_wrapper"的迷惑帧。

        - 重复注册立刻报 ValueError, 不是覆盖。
          理由: 重复注册要么是手误(两个 check 取了同名), 要么是 import 链异常
          导致同一个 check 被加载了两次 — 两种都是 bug, 应该立刻可见。

    Args:
        name: check 的字符串名 (会出现在 JSON 报告 / 日志里, 选稳定的)

    Returns:
        装饰器函数 — 注册副作用发生后, 原函数被原样返回。
    """
    def decorator(func: Callable[[CheckContext], CheckResult]) -> Callable:
        if name in _REGISTRY:
            raise ValueError(
                f"check '{name}' 重复注册 — 第二次出现在 {func.__module__}.{func.__name__}"
            )
        _REGISTRY[name] = CheckEntry(name=name, func=func)
        return func
    return decorator


# ============================================================
# 6. 自动 import — 加新 check 不改框架代码的物理基础
# ============================================================

def _lazy_init() -> None:
    """首次调用 get_all_checks() / get_check() 时, 自动 import checks/ 子包下
    所有非下划线开头的模块, 让其中的 @check 装饰器副作用发生。

    设计:
        - 用 pkgutil.iter_modules 自动扫盘, 不维护硬编码 import 列表。
        理由: 框架的 registry.py 维护一个手工 import 列表违反开闭 —
        加新 check 要改两处(checks/ 加文件 + registry.py 加 import)。
        自动 import 5 行解决, 加新 check 只动 checks/。

        - 跳过下划线开头的模块(_placeholder, _internal_helpers 等)。
        理由: 给框架留一个"内部辅助模块不被自动注册"的逃生口。

        - _LAZY_INITIALIZED 放在 import 全部成功之后——任何 import 失败
        都不会污染状态, 下次调用可以重试。
    """
    global _LAZY_INITIALIZED
    if _LAZY_INITIALIZED:
        return

    import importlib
    import pkgutil
    from od_platform.data_validation import checks

    for _, name, _ in pkgutil.iter_modules(checks.__path__):
        if name.startswith("_"):
            continue
        importlib.import_module(f"{checks.__name__}.{name}")

    _LAZY_INITIALIZED = True


# ============================================================
# 7. 查询 API
# ============================================================

def get_all_checks() -> List[CheckEntry]:
    """返回全部注册的 check (按注册顺序)。供 service.run_all_checks 调度用。"""
    _lazy_init()
    return list(_REGISTRY.values())


def get_check(name: str) -> CheckEntry:
    """按名查询单个 check。供测试 / CLI 调试用。

    Raises:
        KeyError: 未注册的 check 名 — 立刻报错, 不返回 None 让调用方一脸懵
    """
    _lazy_init()
    if name not in _REGISTRY:
        raise KeyError(f"check '{name}' 未注册 — 已注册的: {list(_REGISTRY)}")
    return _REGISTRY[name]


def list_check_names() -> List[str]:
    """返回已注册 check 的名字列表。"""
    _lazy_init()
    return list(_REGISTRY.keys())```

## 2.4 自动 import——开闭原则的物理基础

`checks/__init__.py` 用 `pkgutil` 扫描并 import 包内所有非下划线模块:

```python
# apps/platform/src/od_platform/data_validation/checks/__init__.py
"""checks 包 — 自动 import 所有 check 模块, 触发 @check 注册。

加新 check = 在本目录加一个 .py 文件。不需要改这里, 不需要改 registry,
不需要改 service —— import 即注册, 这是开闭原则的物理基础。
"""
import pkgutil
import importlib

for _finder, _name, _ispkg in pkgutil.iter_modules(__path__):
    if not _name.startswith("_"):
        importlib.import_module(f"{__name__}.{_name}")
```

**这 5 行是整个开闭原则的支点**: 没有它, 每加一个 check 都要去某处手写一行 `import`——那一处就会变成 14 个人排队改的热点文件, merge 冲突重灾区。有了它, "加 check"对版本库的扰动 = 新增一个文件, **diff 干净到 reviewer 扫一眼就能批**。

⚠️ **设计点 (注册顺序)**: `iter_modules` 按文件名字母序——所以 check 的执行顺序 = 文件名字母序。我们**故意不提供优先级机制**: check 之间如果有执行顺序依赖, 说明它们共享了状态, 那是设计错误, 不该用排序去掩护。

## 3.1 为什么第一个是它

挑第一个 check 有讲究: **它要小到能聚焦"check 写作范式", 又真到能立刻抓住生产问题**。yaml_schema 两者全占——

- 它不碰图像不碰标签, 只读一个 yaml, 范式最纯净
- 它抓的问题在企业里**高频且致命**: `nc=5` 但 `names` 写了 6 个 (训练时类别 id 错位, 全部标签静默错配); 两个类别同名 (报告与训练统计混淆)。配置错误是所有错误里"最便宜也最该零容忍"的——一个文件几十个字节, 没有任何理由放过

**后面 13 个 check 全是这一个的同构变体**——把这一个吃透, 后面的代码量会越写越像填表。

## 3.2 范式: 前置错早返回, 业务错收集

check 内部有两类错, 处理方式截然不同:

- **前置错** (yaml 压根读不出来): 后续检查全部失去意义 → **早返回** ERROR
- **业务错** (nc 与 names 不一致 / 类别重名): 互相独立 → **全部收集进 problems 列表**, 一次报全

阶段 1 撞墙①教的"不许挤牙膏", 在单个 check 内部同样适用。

## 3.3 完整代码

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  :yaml_schema.py
# @Time      :2026/6/10 10:43:48
# @Author    :雨霓同学
# @Project   :ODPlatform
# @Function  :
# apps/platform/src/od_platform/data_validation/checks/yaml_schema.py
"""yaml_schema check — 验证数据集 yaml 文件的字段完整性和一致性。

检查项 (任何一项失败都标 ERROR):
    1. yaml 文件存在且可解析
    2. yaml 顶层是 dict (不是 list / scalar) — 仅指文件根节点
    3. 包含 'nc' 字段, 正整数
    4. 包含 'names' 字段, list[str] 或 dict[int,str], 元素非空
    5. len(names) == nc

不检查的项 (后续 check 负责):
    - train / val / test 路径是否存在 → pair_existence
    - 类别 ID 是否越界 → label_format
"""
from __future__ import annotations
from typing import List

from od_platform.data_validation.registry import (
    check, CheckContext, CheckResult, CheckSeverity,
)

@check("yaml_schema")
def validate_yaml_schema(ctx: CheckContext) -> CheckResult:
    snap = ctx.snapshot

    # ---------- 前置错: yaml 加载阶段就出问题了 ----------
    if snap.yaml_load_error is not None:
        return CheckResult(
            name="yaml_schema",
            severity=CheckSeverity.ERROR,
            summary=snap.yaml_load_error,
            details={
                "reason":          "yaml_load_error",
                "yaml_path":       str(snap.yaml_path),
                "yaml_load_error": snap.yaml_load_error,
            },
        )

    # ---------- 业务错: 字段一致性 (收集所有, 一次报齐) ----------
    problems: List[str] = []

    nc = snap.nc
    if nc is None or nc <= 0:
        problems.append(f"nc 缺失或不是正整数: {snap.yaml_data.get('nc')!r}")

    names = snap.class_names
    if not names:
        # _normalize_names 返回空 tuple 表示"非法或缺失"
        raw = snap.yaml_data.get("names")
        problems.append(f"names 缺失或不是合法的 list[str] / dict[int,str]: {type(raw).__name__}")

    if nc is not None and nc > 0 and names and len(names) != nc:
        problems.append(f"nc ({nc}) 跟 names 长度 ({len(names)}) 不一致")

    if names and len(set(names)) != len(names):
        seen, dups = set(), []
        for n in names:
            if n in seen and n not in dups:
                dups.append(n)
            seen.add(n)
        problems.append(f"names 存在重复类别名: {dups} — 报告与训练统计会混淆")

    if problems:
        return CheckResult(
            name="yaml_schema",
            severity=CheckSeverity.ERROR,
            summary=f"yaml 字段不一致: {len(problems)} 处问题",
            details={
                "reason":      "field_inconsistency",
                "problems":    problems,
                "nc":          nc,
                "names_count": len(names) if names else 0,
            },
        )

    # ---------- 全部通过 ----------
    return CheckResult(
        name="yaml_schema",
        severity=CheckSeverity.PASS,
        summary=f"yaml 字段一致 (nc={nc}, names_count={len(names)})",
        details={
            "nc":          nc,
            "names_count": len(names),
        },
    )
```

⚠️ **设计点 (names 查重为什么值得单列一条)**: 两个类别同名时, 一切"按类别名聚合"的下游 (分布表 / 失衡比 / 报告) 都会把两个 id 的数字悄悄合并——又是一个静默错误。查重的代码三行, 抓住一次就回本。

⚠️ **设计点 (summary 一行话的纪律)**: summary 是给"检查项一览表"的一行话, **必须自带关键数字** (`nc=5, names_count=6`)——让人不点开 details 也能判断要不要管。"yaml 有问题"这种 summary 等于没写。

data = yaml.safe_load(ctx.yaml_path.read_text(...))   # 读 yaml — 第 2 次
for split in ("train", "val", "test"):

for split in ...:

## 4.2 🧱 撞墙②: I/O 的乘法

算一笔账, 这是企业里说服别人重构时唯一好使的语言——**量化**:

| | 单个 check 自己扫 | 14 个 check 都自己扫 |
|---|---|---|
| 读 yaml | 1 次 | **14 次** |
| 列目录 + stat 图像 | 1 遍 (N 个文件) | **~10 遍** (凡碰文件的 check) |
| 读标签内容 | 1 遍 | **~8 遍** (凡碰标签内容的 check) |

540 张图时无所谓; **50 万张图、标签在 NFS 上时, 每多扫一遍是几分钟**——质量闸门跑 40 分钟, CI 里没人会开它, 闸门形同虚设。更糟的是**正确性问题**: 14 个 check 各扫各的, 如果扫描期间有人动了数据, 不同 check 看到的是**不同版本的数据集**——报告内部自相矛盾, 审计噩梦。

## 4.3 解药: DatasetSnapshot——扫一次, 给所有人用

**一次扫描, 产出一个不可变的"数据集快照", 所有 check 消费同一个快照。** 三条铁律:

1. **只装数据, 不做判断。** snapshot 里没有任何 severity / 没有任何"好坏"结论——判断是 check 的事。装数据的和下结论的混在一起, 两者都没法单独测试。
2. **frozen 不可变。** 14 个 check 共享一个对象, 任何一个能改它, 其他 13 个就活在流沙上。`@dataclass(frozen=True)` 把这条铁律变成解释器强制执行的法律。
3. **顺手统计——同一个循环里能白拿的数字, 现在就拿。** 扫描循环反正要读每个标签文件, 顺手数一下"几张有标注 / 总共几行 / 每个类别几个实例"几乎零成本; 但**事后**想要这些数字, 就得整盘重读一遍。这是 snapshot 最值钱的红利, 后面会兑现两次 (阶段 10 的 annotation_coverage, 阶段 11 的 class_presence/class_balance——它们因此都是**零额外 I/O** 的 check)。

⚠️ **铁律 3 的边界——"顺手"不是"顺路搬家"**: 顺手统计只拿**同一循环内边际成本≈0**的计数。bbox 面积分布 / 宽高比直方图这类要逐行解析几何的重统计**不进 snapshot**——那是阶段 13 画像层的事。判别标准一句话: **"不拿白不拿"的拿, "要专门算"的不拿。** snapshot 一旦开始膨胀, 它就会变成谁都想塞一勺的杂物间。

## 4.4 snapshot.py 完整代码 — 上半: 两个数据类

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  :snapshot.py
# @Time      :2026/6/10 11:25:21
# @Author    :雨霓同学
# @Project   :ODPlatform
# @Function  :
# apps/platform/src/od_platform/data_validation/snapshot.py
"""DatasetSnapshot — 一次扫描, 多次复用。

设计原则:
    - 装数据, 不做判断 (业务判断属于 check 函数)
    - frozen=True 防止 check 串改造成调试灾难
    - best-effort: yaml 解析失败 / split 目录不存在都不抛, 装进 snapshot
      让 check 自己拿去判定 — 调度层永远不该自己 fail
"""
from __future__ import annotations

import logging
from dataclasses import dataclass, field
from pathlib import Path
from typing import Any, Dict, List, Optional, Tuple

import yaml

from od_platform.common.constants import IMAGE_EXTENSIONS, Task
from od_platform.common.performance_utils import time_it

logger = logging.getLogger(__name__)


# ============================================================
# SplitStats — 轻量统计 (3 个数字, 供报告展示)
# ============================================================

@dataclass(frozen=True)
class SplitStats:
    """单个 split 的轻量统计 — 用于 ValidationReport 数据集摘要。

    基础 3 个数字: 够"看一眼知道数据集啥规模"。
    D4.4 增量 (class_instances / class_images): 在同一次标签读取循环里顺手累加,
    零额外 I/O — 这是 snapshot"顺手统计"红利的第二次兑现, 供 class_presence /
    class_balance / profiler 共享消费, 它们因此都不必再扫盘。
    更重的统计 (bbox 面积分布等) 仍然不进 snapshot — 归 profiler。
    """
    image_count:     int    # 该 split 的图像总数
    annotated_count: int    # 有非空标签的图像数
    total_instances: int    # 全部 bbox 实例数 (label 文件总行数)
    class_instances: Dict[int, int] = field(default_factory=dict)
    """{class_id: 实例数} — 仅统计能解析出整数 class_id 的行 (坏行归 label_format 报)。"""
    class_images:    Dict[int, int] = field(default_factory=dict)
    """{class_id: 含该类的图像数} — 同一图内同类多实例只计 1。"""


# ============================================================
# DatasetSnapshot — 不可变数据载体
# ============================================================

@dataclass(frozen=True)
class DatasetSnapshot:
    """对一个数据集做一次完整快照, 供所有 check 共享消费。

    所有路径已经解析为绝对路径; 调用方不需要再考虑相对路径 / cwd 问题。

    Args:
        yaml_path:        数据集 yaml 文件绝对路径
        yaml_data:        原始 yaml 解析结果 (dict, 供 yaml_schema 直接判定)
        yaml_load_error:  yaml 解析过程中的错误信息 (None = 解析成功)
                        注意: 解析失败时 yaml_data 是 {}, 但 snapshot 仍然构造
                        成功 — 让 yaml_schema check 拿着 yaml_load_error 报错,
                        而不是 build_snapshot 自己抛异常
        data_root:        数据集根目录 (yaml 里 'path' 字段, 或 yaml 同级目录兜底)
        nc:               类别数 (yaml.nc; 缺失或非法时 None)
        class_names:      类别名 (按 ID 升序的 tuple; 缺失时空 tuple)
        task_type:        'detect' / 'segment'
        images_per_split: {split_name: (图像 Path, ...)} 绝对路径, 已排序
        labels_per_split: {split_name: (预期标签 Path, ...)} 按 stem 一对一对应
                        注意: 此处只是"按图像名构造期望的 .txt 路径", 不保证
                        文件存在 — pair_existence check 才负责存在性
        stats_per_split:  {split_name: SplitStats} 每个 split 的 3 个数字
        scan_warnings:    扫描过程中的非致命警告 (split 目录不存在等)
    """
    yaml_path:        Path
    yaml_data:        Dict[str, Any]
    yaml_load_error:  Optional[str]
    data_root:        Path
    nc:               Optional[int]
    class_names:      Tuple[str, ...]
    task_type:        str
    images_per_split: Dict[str, Tuple[Path, ...]]
    labels_per_split: Dict[str, Tuple[Path, ...]]
    stats_per_split:  Dict[str, SplitStats]
    label_files_per_split: Dict[str, Tuple[Path, ...]] = field(default_factory=dict)
    """{split: labels 目录下【实际存在】的 .txt 文件 (绝对路径, 已排序)}。

    与 labels_per_split 的区别 (一字之差, 方向相反):
        labels_per_split      = 按图像名【构造的期望】标签路径 (图 → 标签方向)
        label_files_per_split = 磁盘上【真实存在】的标签文件   (标签 → 图方向)
    orphan_labels check 用两者的 stem 差集抓"有标签无图像"的反向孤儿。
    """
    scan_warnings:    Tuple[str, ...] = field(default_factory=tuple)

    # ---------- 派生属性 (不存值) ----------

    @property
    def splits(self) -> Tuple[str, ...]:
        """返回快照里实际有图像的 split 名 (按 train/val/test 顺序)。"""
        order = ("train", "val", "test")
        return tuple(s for s in order if s in self.images_per_split)

    @property
    def total_images(self) -> int:
        return sum(len(imgs) for imgs in self.images_per_split.values())

```

⚠️ **设计点 (labels_per_split vs label_files_per_split, 一字之差方向相反)**: 前者是**按图像名构造的期望路径** (图→标签方向, 给 pair_existence 用), 后者是**磁盘上真实存在的文件** (标签→图方向, 给阶段 10 的 orphan_labels 用)。两个方向都要有, 否则"有图没标签"和"有标签没图"只能查一半——这是**镜像检查**思想第一次出现, 阶段 10 正式展开。

⚠️ **设计点 (scan_warnings 是张欠条)**: 扫描中发现"yaml 声明了 test 但目录不存在"这类事, snapshot 按铁律 1 不能下结论, 只能**记下来**。注意: **收集了没人消费的数据等于没收集**——这张欠条必须有 check 来兑付, 阶段 10 的 split_presence 就是债主。先记住这句话, 它是企业代码审查里抓"假功能"的利器。

## 4.5 snapshot.py 完整代码 — 下半: 辅助函数 + build_snapshot

```python
# ============================================================
# 内部辅助 — IO / 解析 (失败不抛)
# ============================================================

def _load_yaml(yaml_path: Path) -> Tuple[Dict[str, Any], Optional[str]]:
    """加载 yaml, 返回 (data, error_str)。失败时返回 ({}, err)。"""
    if not yaml_path.exists():
        return {}, f"yaml 文件不存在: {yaml_path}"
    try:
        with open(yaml_path, "r", encoding="utf-8") as f:
            data = yaml.safe_load(f) or {}
        if not isinstance(data, dict):
            return {}, f"yaml 顶层不是 dict: {type(data).__name__}"
        return data, None
    except yaml.YAMLError as e:
        return {}, f"yaml 解析失败: {e}"
    except OSError as e:
        return {}, f"yaml 读取失败: {e}"


def _resolve_data_root(yaml_path: Path, yaml_data: Dict[str, Any]) -> Path:
    """三种合法写法:
       1. yaml.path 是绝对路径 → 用它
       2. yaml.path 是相对路径 → 相对 yaml 文件所在目录
       3. yaml.path 缺失 → 用 yaml 文件所在目录
    """
    path_str = yaml_data.get("path")
    if not path_str:
        return yaml_path.parent.resolve()
    p = Path(path_str)
    return p.resolve() if p.is_absolute() else (yaml_path.parent / p).resolve()


def _resolve_split_dir(data_root: Path, split_field: Any) -> Optional[Path]:
    """yaml 里的 train/val/test 字段, 只支持字符串形式 (ultralytics 主流写法)。
    其他形式 (list / None) 返回 None, 让 yaml_schema 后续负责报错。
    """
    if not isinstance(split_field, str) or not split_field.strip():
        return None
    p = Path(split_field)
    return p.resolve() if p.is_absolute() else (data_root / p).resolve()


def _list_images(split_dir: Path) -> List[Path]:
    """返回 split_dir 下所有支持扩展名的图像 (绝对路径, 已排序)。"""
    if not split_dir.exists() or not split_dir.is_dir():
        return []
    images: List[Path] = []
    for ext in IMAGE_EXTENSIONS:
        images.extend(split_dir.glob(f"*{ext}"))
        images.extend(split_dir.glob(f"*{ext.upper()}"))   # Windows 大小写
    return sorted(set(images))


def _label_path_for_image(image_path: Path) -> Path:
    """YOLO 默认布局: images/<split>/foo.jpg → labels/<split>/foo.txt

    倒着找最后一个 'images' 替换为 'labels' (路径里可能多次出现 'images',
    取最后一个最稳妥)。
    """
    parts = list(image_path.parts)
    for i in range(len(parts) - 1, -1, -1):
        if parts[i] == "images":
            parts[i] = "labels"
            break
    return Path(*parts[:-1]) / (image_path.stem + ".txt")


def _normalize_names(names_raw: Any) -> Tuple[str, ...]:
    """把 yaml.names 归一化成"按 ID 升序的 tuple"。

    输入合法形式:
        - list[str]:      ['a','b','c']        → ('a','b','c')
        - dict[int,str]:  {0:'a',1:'b',2:'c'}  → ('a','b','c')

    非法或缺失返回空 tuple — 由 yaml_schema 负责报错。
    """
    if isinstance(names_raw, list):
        if all(isinstance(n, str) for n in names_raw):
            return tuple(names_raw)
        return ()
    if isinstance(names_raw, dict):
        if all(isinstance(k, int) for k in names_raw.keys()) \
                and all(isinstance(v, str) for v in names_raw.values()):
            return tuple(v for _, v in sorted(names_raw.items()))
        return ()
    return ()


def _build_split_stats(labels: List[Path]) -> SplitStats:
    """顺手算统计 — 在 build_snapshot 已经在循环图像了, 多读一次 label 也免不了。

    D4.4 增量: 同一循环里顺手累加按类别计数 (class_instances / class_images),
    解析失败的行静默跳过 — 行格式的合法性判定是 label_format 的职责, 这里只装数据。
    """
    image_count     = len(labels)
    annotated_count = 0
    total_instances = 0
    class_instances: Dict[int, int] = {}
    class_images:    Dict[int, int] = {}

    for lbl in labels:
        if not lbl.exists():
            continue
        try:
            content = lbl.read_text(encoding="utf-8")
        except OSError:
            continue
        lines = [l for l in content.splitlines() if l.strip()]
        if not lines:
            continue   # 空 label 不算 annotated
        annotated_count += 1
        total_instances += len(lines)

        seen_in_image: set = set()
        for line in lines:
            parts = line.split()
            if not parts:
                continue
            try:
                cls_id = int(parts[0])
            except ValueError:
                continue   # 坏行不计数 — label_format 负责报
            class_instances[cls_id] = class_instances.get(cls_id, 0) + 1
            if cls_id not in seen_in_image:
                seen_in_image.add(cls_id)
                class_images[cls_id] = class_images.get(cls_id, 0) + 1

    return SplitStats(
        image_count=image_count,
        annotated_count=annotated_count,
        total_instances=total_instances,
        class_instances=class_instances,
        class_images=class_images,
    )


def _list_label_files(labels: List[Path]) -> Tuple[Path, ...]:
    """返回 labels 目录下【实际存在】的全部 .txt 文件。

    labels 目录从期望标签路径反推 (labels[0].parent) — 不重复造路径规则。
    目录不存在返回空 tuple; 大小写双扫兼容 Windows 来源的数据。
    """
    if not labels:
        return ()
    labels_dir = labels[0].parent
    if not labels_dir.exists() or not labels_dir.is_dir():
        return ()
    found = set(labels_dir.glob("*.txt")) | set(labels_dir.glob("*.TXT"))
    return tuple(sorted(found))


# ============================================================
# 公开 API: build_snapshot
# ============================================================

@time_it(name="构建数据集快照",logger_instance=logger, iterations=1)
def build_snapshot(
    yaml_path: Path,
    task_type: Optional[str] = None,
) -> DatasetSnapshot:
    """构造数据集快照, 一次扫描提供后续 check 全部所需素材。

    错误处理原则: best-effort —— yaml 解析失败 / split 目录不存在都只装进
    snapshot 的 yaml_load_error / scan_warnings, 不抛异常。由对应 check
    负责把这些信号转成 CheckResult。

    Args:
        yaml_path: 数据集 yaml 路径
        task_type: 'detect' / 'segment'。优先级:
                显式参数 > yaml.task 字段 > Task.DETECT 兜底

    Returns:
        DatasetSnapshot (永远不抛异常)
    """
    yaml_path = yaml_path.resolve()
    warnings: List[str] = []

    yaml_data, yaml_err = _load_yaml(yaml_path)
    if yaml_err:
        warnings.append(yaml_err)

    data_root = _resolve_data_root(yaml_path, yaml_data)

    nc          = yaml_data.get("nc") if isinstance(yaml_data.get("nc"), int) else None
    class_names = _normalize_names(yaml_data.get("names"))

    # task_type 优先级: 显式 > yaml.task > 兜底 detect
    resolved_task = task_type or yaml_data.get("task") or Task.DETECT
    if resolved_task not in (Task.DETECT, Task.SEGMENT):
        warnings.append(f"未知 task_type '{resolved_task}', 回退到 '{Task.DETECT}'")
        resolved_task = Task.DETECT

    images_per_split: Dict[str, Tuple[Path, ...]] = {}
    labels_per_split: Dict[str, Tuple[Path, ...]] = {}
    stats_per_split:  Dict[str, SplitStats]       = {}
    label_files_per_split: Dict[str, Tuple[Path, ...]] = {}

    for split in ("train", "val", "test"):
        split_dir = _resolve_split_dir(data_root, yaml_data.get(split))
        if split_dir is None or not split_dir.exists():
            if split in yaml_data:
                warnings.append(f"split '{split}' 目录不可用: {split_dir}")
            continue

        images = _list_images(split_dir)
        if not images:
            warnings.append(f"split '{split}' 目录下无图像: {split_dir}")
            continue

        labels = [_label_path_for_image(img) for img in images]

        images_per_split[split] = tuple(images)
        labels_per_split[split] = tuple(labels)
        stats_per_split[split]  = _build_split_stats(labels)
        label_files_per_split[split] = _list_label_files(labels)

    snapshot = DatasetSnapshot(
        yaml_path=yaml_path,
        yaml_data=yaml_data,
        yaml_load_error=yaml_err,
        data_root=data_root,
        nc=nc,
        class_names=class_names,
        task_type=resolved_task,
        images_per_split=images_per_split,
        labels_per_split=labels_per_split,
        stats_per_split=stats_per_split,
        label_files_per_split=label_files_per_split,
        scan_warnings=tuple(warnings),
    )
    logger.info(
        f"snapshot 构建完成: {snapshot.total_images} 张图像, "
        f"splits={list(snapshot.splits)}, task={resolved_task}"
    )
```

三个值得停一秒的细节:

1. **`_label_path_for_image` 倒着找最后一个 `images` 替换成 `labels`**——路径里可能多次出现 "images" (比如用户把数据放在 `/data/images_backup/images/train/`), 取最后一个最稳。路径规则只此一处实现, 所有消费者引用——又是 SSoT。
2. **`_build_split_stats` 里坏行静默跳过**——`int(parts[0])` 失败就 continue。不是偷懒: **行格式合法性的判定是 label_format 的职责**, snapshot 越俎代庖去报错, 同一个问题会被报两遍且口径可能不一致。装数据的不下结论, 铁律 1 落到了第 217 行这种具体地方。
3. **`@time_it(name="构建数据集快照")`**——扫盘是 D4 最重的 I/O, 必须可量化。当用户抱怨"验证慢"时, 第一眼看日志里这一行: 是扫描慢还是 check 慢, 责任立刻二分。

# 阶段 5: 结构性三连 — pair_existence / label_format / split_uniqueness

snapshot 就位, 现在把阶段 4 推翻的两个草稿正式重写, 再补一个防泄露的——它们消费 snapshot, 自己不碰 yaml 不列目录。本阶段还要立两个新模式: **按比例分级 severity** 和 **details 预览上限**。

## 5.1 pair_existence — 按比例分级的第一课 (完整代码)

每张 `images/train/foo.jpg` 都应有 `labels/train/foo.txt` (允许空文件 = 无目标背景图; 但不能**不存在**)。问题是: **缺多少算严重?**

- 缺 0.1% (1000 张缺 1 张) → 标注遗漏, 工程上能容忍 → INFO
- 缺 5% ~ 50% → 显著遗漏, 影响精度, 人工 review → WARNING
- 缺 ≥ 50% → 流程级事故 (多半是路径配错/拷贝漏盘) → ERROR

**一刀切 ERROR 太严, 一刀切 WARNING 太松——按比例分级**。这就是企业判定的常态: 同一种问题, 量变引起定性变。阶段 0 立的两个阈值在此兑付。

```python
# apps/platform/src/od_platform/data_validation/checks/pair_existence.py
"""pair_existence check — 验证每张图都有对应的 .txt 标签文件。

Severity 按缺失比例分级:
    missing_ratio == 0.0           → PASS
    missing_ratio <  WARN_RATIO    → INFO    (个别遗漏, 可容忍)
    missing_ratio <  ERROR_RATIO   → WARNING (显著遗漏, 影响精度)
    missing_ratio >= ERROR_RATIO   → ERROR   (流程级问题, 不允许训练)

注: 空 .txt 文件算"存在"且合法 (YOLO 接受空标签表示无目标图像)。
    这里只判定"文件是否存在", 不判定"是否为空"。
"""
from __future__ import annotations

from typing import Any, Dict, List

from od_platform.common.constants import (
    PAIR_MISSING_ERROR_RATIO,
    PAIR_MISSING_WARN_RATIO,
)
from od_platform.data_validation.registry import (
    check, CheckContext, CheckResult, CheckSeverity,
)


# 详情里展示的"前 N 个缺失文件"上限 — 避免 details 爆炸
DETAILS_PREVIEW_LIMIT = 20


@check("pair_existence")
def validate_pair_existence(ctx: CheckContext) -> CheckResult:
    snap = ctx.snapshot

    if not snap.images_per_split:
        return CheckResult(
            name="pair_existence",
            severity=CheckSeverity.INFO,
            summary="无任何 split 可检查 (snapshot 为空)",
            details={"reason": "empty_snapshot"},
        )

    # 收集每个 split 的孤儿图像 (有图无标签)
    orphan_per_split: Dict[str, List[str]] = {}
    total_images  = 0
    total_missing = 0

    for split, images in snap.images_per_split.items():
        labels = snap.labels_per_split.get(split, ())
        missing_in_split: List[str] = []
        # build_snapshot 保证两者长度一致 + 顺序一致 (按图像名构造标签路径)
        for img, lbl in zip(images, labels):
            total_images += 1
            if not lbl.exists():
                total_missing += 1
                missing_in_split.append(str(img))
        if missing_in_split:
            orphan_per_split[split] = missing_in_split

    missing_ratio = total_missing / max(total_images, 1)

    # ---------- 按比例分级 severity (从最严到最松写) ----------
    if total_missing == 0:
        severity = CheckSeverity.PASS
        summary  = f"全部 {total_images} 张图像都有对应标签"
    elif missing_ratio >= PAIR_MISSING_ERROR_RATIO:
        severity = CheckSeverity.ERROR
        summary = (
            f"缺标签比例 {missing_ratio:.1%} ≥ {PAIR_MISSING_ERROR_RATIO:.0%} "
            f"({total_missing}/{total_images} 张图无标签)"
        )
    elif missing_ratio >= PAIR_MISSING_WARN_RATIO:
        severity = CheckSeverity.WARNING
        summary = (
            f"缺标签比例 {missing_ratio:.1%} ≥ {PAIR_MISSING_WARN_RATIO:.0%} "
            f"({total_missing}/{total_images} 张图无标签)"
        )
    else:
        severity = CheckSeverity.INFO
        summary = (
            f"少量标签缺失 ({total_missing}/{total_images} = {missing_ratio:.2%})"
        )

    # ---------- 构造 details (含 preview 截断防止报告爆炸) ----------
    details: Dict[str, Any] = {
        "total_images":  total_images,
        "total_missing": total_missing,
        "missing_ratio": round(missing_ratio, 4),
        "thresholds": {
            "error_at": PAIR_MISSING_ERROR_RATIO,
            "warn_at":  PAIR_MISSING_WARN_RATIO,
        },
        "missing_per_split": {
            split: len(orphans) for split, orphans in orphan_per_split.items()
        },
    }

    # 详细清单截断 — 前 N 条供日志展开, 完整列表通过 JSON 报告查看
    if orphan_per_split:
        details["missing_examples"] = {
            split: orphans[:DETAILS_PREVIEW_LIMIT]
            for split, orphans in orphan_per_split.items()
        }

    return CheckResult(
        name="pair_existence",
        severity=severity,
        summary=summary,
        details=details,
    )```

⚠️ **设计点 (DETAILS_PREVIEW_LIMIT = 20)**: 缺 3 万个标签时, details 里塞 3 万条路径会把 report.json 撑成垃圾场。**计数给全量, 样例给前 20**——"多严重"用数字说, "长什么样"用样例说。这条上限模式后面每个 check 都沿用。"那全量清单去哪看?" 阶段 13 的 instances.csv / 各 check 的明细——**报告层永远放摘要, 明细层另有去处**, 这是双轨思想的第一次预告。

## 5.2 label_format — 设计卡

| | |
|---|---|
| **事故形态** | 字段数不对 / 数字解析不了 / class_id ≥ nc / 坐标出 [0,1] / segment 多边形点数不够——标注工具导出 bug 与人工编辑事故的集合 |
| **企业损失** | ultralytics 对坏行的行为是**跳过并 console 刷一行 warning**——5 万张图的训练里没人看 console, 坏行悄悄变成"少了一批训练样本" |
| **判定规则** | 任何坏行 → ERROR (格式错是结构性问题, 没有"坏一点点"的说法); 错误按 KIND 分类计数, 样例前 20 |
| **依赖** | `snap.task_type` 决定走 detect (5 字段) 还是 segment (1+2N 字段) 校验分支 |

核心是单行校验器 `_validate_one_line`——**把"一行合不合法"做成纯函数**, 主循环只管遍历与收集。错误类型用机器可读常量 (不是中文句子), `details["error_kinds"]` 里能按类型计数, 下游报告直接画表:

```python
KIND_FIELD_COUNT_MISMATCH  = "field_count_mismatch"
KIND_PARSE_ERROR           = "parse_error"
KIND_CLASS_ID_OUT_OF_RANGE = "class_id_out_of_range"
KIND_COORD_OUT_OF_RANGE    = "coord_out_of_range"
KIND_POLYGON_TOO_FEW       = "polygon_too_few_points"

def _validate_one_line(line: str, task_type: str, nc: int):
    """返回 (kind, detail) 或 None(合法)。纯函数 — 单测在这层打, 不用造文件。"""
    parts = line.split()
    if task_type == Task.DETECT:
        if len(parts) != 5:
            return KIND_FIELD_COUNT_MISMATCH, f"detect 任务要求 5 字段, 实际 {len(parts)}"
        try:
            cls_id, coords = int(parts[0]), [float(x) for x in parts[1:5]]
        except ValueError as e:
            return KIND_PARSE_ERROR, f"字段类型错: {e}"
        if not (0 <= cls_id < nc):
            return KIND_CLASS_ID_OUT_OF_RANGE, f"cls_id={cls_id} 不在 [0,{nc}) 内"
        bad = [c for c in coords if not (0.0 <= c <= 1.0)]
        if bad:
            return KIND_COORD_OUT_OF_RANGE, f"坐标越界 [0,1]: {bad}"
        return None
    # segment 分支: 1+2N 字段、N>=3、逐坐标 [0,1] — 同构, 见完整文件
```

⚠️ **设计点 (task_type 从哪来)**: 走 `snap.task_type`, 它在 build_snapshot 里按"调用方显式指定 > yaml 的 task 字段 > 默认 detect"三级解析**一次**——14 个 check 谁都不要自己再去猜任务类型, 否则两个 check 对同一份数据用不同任务口径校验, 报告自相矛盾。

**先记下一个伏笔**: 这个 check 验证 cx/cy/w/h **各自** ∈ [0,1]。"各自合法"和"整体合法"是两回事——阶段 9 会在这里撞第四面墙。

## 5.3 split_uniqueness — 设计卡

| | |
|---|---|
| **事故形态** | 同一 stem 的图像出现在两个划分 (视频抽帧/增强副本/手工拷贝), 开篇事故二的主角 |
| **企业损失** | 验证集指标虚高 → 基于假指标的上线决策。**数据泄露是质检里唯一"零容忍"的类别** |
| **判定规则** | 任何跨划分 stem 重复 → **硬性 ERROR**, 不按比例分级——泄露 1 张和泄露 100 张对"指标可信度"的破坏是同一种性质 |
| **边界** | 按 stem 比对, **不做内容哈希**——`a.jpg` 改名成 `b.jpg` 抓不到, 那是内容级近重复的活 (TODO D4.9 pHash, 重型) |

```python
@check("split_uniqueness")
def validate_split_uniqueness(ctx: CheckContext) -> CheckResult:
    snap = ctx.snapshot
    stems = {s: {p.stem for p in imgs} for s, imgs in snap.images_per_split.items()}

    overlaps = []
    splits = list(stems)
    for i in range(len(splits)):
        for j in range(i + 1, len(splits)):
            common = stems[splits[i]] & stems[splits[j]]
            if common:
                overlaps.append({
                    "split_a": splits[i], "split_b": splits[j],
                    "count": len(common),
                    "preview": sorted(common)[:DETAILS_PREVIEW_LIMIT],
                })
    # overlaps 为空 → PASS; 否则 ERROR(数据泄露) — 完整文件见仓库
```

⚠️ **设计点 (为什么不分级)**: pair_existence 的"缺"是**量的问题**(比例分级), split_uniqueness 的"漏"是**质的问题**(一票否决)。给每个 check 选分级策略时先问这一句: **这个问题是越多越糟, 还是有就完蛋?**

# 阶段 6: 调度层 service — 聚合模式的承诺

## 6.1 互斥分发 vs 聚合执行

D3 和 D4 用同一个注册表 idiom, 但调度层的**承诺**完全不同:

| | D3 (互斥分发) | D4 (聚合执行) |
|---|---|---|
| 一次调用 | 选 **1 个** converter 跑 | 跑**全部** check |
| 失败语义 | 选错/转换失败 → **立刻抛**, 流程停 | 单个 check 异常 → **接住包装**, 其余继续 |
| 给调用方的话 | "我帮你干这一件事, 干不成就喊" | "**我保证没有任何一个 check 能阻断其他 check**" |

第二句承诺就是 D4 调度层存在的意义——撞墙①的"挤牙膏"病, 在框架层根治。

## 6.2 唯一一处 `except Exception`

整个 D4 子系统只有 `_safe_run_one` 里有一个宽泛的 `except Exception`, 这是**有意为之并写进 docstring 的纪律**:

- **为什么这里必须宽**: check 是开闭原则下的扩展点, 谁都能加, 调度层无法预知第 15 个 check 会抛 KeyError 还是自定义异常。接不住任何一种, 聚合承诺就破了。
- **为什么别处必须窄**: 其他地方的异常类型是可预知的 (读文件 OSError / 解析 yaml.YAMLError), 宽捕获会把 bug 当业务错误吞掉。**"宽捕获只许出现在扩展点边界"**——代码审查时拿这条卡, 一卡一个准。

被接住的异常包装成 **ERROR 级 CheckResult** (check 自己崩了 = 这一项没验成 = 不能放行), `exc_info=True` 把完整 traceback 留在日志里供排查——summary 给人看一句话, traceback 给工程师 grep。

## 6.3 日志分级哲学: PASS 走 DEBUG

每个 check 跑完即时打一行进度日志, severity → log level 直接映射, 唯独 **PASS 走 DEBUG**:

> PASS 不是"事件", 是"非事件"。健康数据集每次跑 14 行 `[PASS]` 刷屏, 噪声盖过信号。日志只记"有变化"的事 (ERROR/WARNING/INFO), PASS 的全貌走阶段 7 报告的"检查项一览"集中展示, 想看就 `-v`。

这是企业日志的通用纪律: **日志是告警通道, 不是成绩单**。

## 6.4 代码 (service.py 上半)

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  :service.py
# @Time      :2026/6/10 10:31:05

# @Project   :ODPlatform
# @Function  :
# apps/platform/src/od_platform/data_validation/service.py
"""data_validation 调度层 — run_all_checks。

聚合模式核心承诺: 任何 check 抛异常都不能阻断其他 check。
这条承诺通过 _safe_run_one 里的 try/except Exception 兑现 —— 整个 D4 子系统
里仅此一处用到 Exception 这种宽泛的捕获, 因为这里是"开闭原则下的扩展点"
(谁都能加 check), 调度层无法预知 check 会抛什么。

其他地方应该用具体的 except OSError / except yaml.YAMLError 等。
"""
from __future__ import annotations

from typing import List
from pathlib import Path
from typing import Optional

from od_platform.common.performance_utils import time_it
from od_platform.data_validation.registry import (
    CheckContext, CheckEntry, CheckResult, CheckSeverity, ValidationOptions,
    get_all_checks,

import json
import time
from datetime import datetime, timezone

@time_it(name="所有检测耗时总计",logger_instance=logger, iterations=1)
def run_all_checks(ctx: CheckContext) -> List[CheckResult]:
    """跑全部注册的 check, 收集结果。

    聚合模式承诺:
        任何 check 自身抛异常都被本函数接住, 包装成 ERROR 级 CheckResult,
        不阻断其他 check。 这是 D4 跟 D3 (互斥模式) 最大的区别 —— D3 service.convert
        失败立刻抛, D4 service.run_all_checks 失败也继续。

    Returns:
        List[CheckResult] — 按 check 注册顺序, 一条不漏

    for entry in entries:
        result = _safe_run_one(entry, ctx)
        _log_check_result(result)
        results.append(result)

    _log_summary(results)
    return results

@time_it(name=lambda entry, ctx: f"检查:【{entry.name}】", logger_instance=logger, iterations=1)
def _safe_run_one(entry: CheckEntry, ctx: CheckContext) -> CheckResult:
    """跑单个 check, 异常包装成 ERROR — 聚合模式承诺的兑现处。

    设计:
        - try/except Exception 是【唯一】合理使用宽泛捕获的地方。
        理由: check 是开闭原则下的扩展点, 谁都能加新 check, 调度层无法预知
        它会抛什么(KeyError / TypeError / 自定义异常 / ...)。 接住一切,
        包装成 ERROR 级 CheckResult, 让其他 check 继续。

        - exc_info=True: 把完整 traceback 写到日志, 方便事后排查 — 这条信息
        会出现在 CheckResult.summary 里(避免污染用户看到的简短摘要),
        但保留在日志文件里, 调试时 grep 一下就有。
    """
    try:
        return entry.func(ctx)
    except Exception as e:
        logger.exception(f"check '{entry.name}' 抛异常, 已捕获为 ERROR 级结果")
        return CheckResult(
            name=entry.name,
            severity=CheckSeverity.ERROR,
            summary=f"check 内部异常: {type(e).__name__}: {e}",
            details={
                "exception_type": type(e).__name__,
                "exception_msg":  str(e),
            },
        )


def _log_check_result(r: CheckResult) -> None:
    """单个 check 跑完即时打一条日志, 让用户看到进度。

    Severity → log level 映射:
        ERROR   → logger.error
        WARNING → logger.warning
        INFO    → logger.info
        PASS    → logger.debug   (默认不显示, -v 才看)

    为什么 PASS 走 DEBUG:
        PASS 不是"事件", 是"非事件" — 健康数据集每次跑 4 行 [PASS] 刷屏,
        噪声盖过信号。 日志只记 WARN/ERROR/INFO 这种"有变化"的事件,
        PASS 信息走 ValidationReport 的最终摘要 (阶段 8 的 render.py),
        集中展示。
    """
    log_method = {
        CheckSeverity.ERROR:   logger.error,
        CheckSeverity.WARNING: logger.warning,
        CheckSeverity.INFO:    logger.info,
        CheckSeverity.PASS:    logger.debug,
    }.get(r.severity, logger.info)

    log_method(f"[{r.severity:7s}] {r.name}: {r.summary}")


def _log_summary(results: List[CheckResult]) -> None:
    """所有 check 跑完后, 打一行总览。"""
    counts = {}
    for r in results:
        counts[r.severity] = counts.get(r.severity, 0) + 1
    parts = [f"{n} {s}" for s, n in counts.items()]
    logger.info(f"check 执行完毕: {' / '.join(parts)}")

```

⚠️ **设计点 (@time_it 的 callable name)**: `_safe_run_one` 的计时名是 `lambda entry, ctx: f"检查:【{entry.name}】"`——每个 check 的耗时在日志里**按名字**单列。当 14 个 check 总耗时变慢, 一眼定位是哪个在拖 (剧透: 永远是开了 `--check-images` 的 image_integrity, 阶段 12 见)。**框架级计时是性能问题的"谁的锅"裁判**, 缺了它, 慢就只能靠猜。

# 阶段 7: 🧱 撞墙③ + ValidationReport / render 分离

## 7.1 撞墙③: 跑通之后, 发现"没人看得懂"

把 7 个 check 串起来跑一个健康数据集:

```
check 执行完毕: 7 PASS
```

**就这一行** (PASS 走 DEBUG, 都没上 INFO)。跑一个脏数据集, 好一点——几行 WARNING。但四个客户全都不满意:

- 算法工程师: "数据集**长什么样**? 几张图几个类别什么分布? 我得自己再写脚本数一遍?"
- CI: "给我一个机器可读的产物, 我要把结果贴到 PR 评论里。"
- 标注团队: "WARNING 说缺 37 个标签, **哪 37 个?**"
- 三个月后的审计: "**当时**那次验证的结果在哪? 日志早滚没了。"

本能的修法是在 service 里继续堆 `logger.info(...)` 把信息打全——**打住**。那会让 service 同时长出两个变化轴: 检查逻辑变 (业务) 和展示格式变 (审美)。下个月有人要 Markdown 报告, 再下个月要 HTML——每次都去改调度层? 调度层会变成格式化代码的垃圾场。

## 7.2 解药: 数据与展示物理分离

```
service ──产出──▶ ValidationReport (纯数据, 只装不渲染)
                        │
            ┌───────────┼─────────────────┐
            ▼           ▼                 ▼
    render_to_logger  report.json   render_markdown
    (给工程师)       (给机器/CI)    (给审批流, 阶段 14)
```

**一份数据, N 个 renderer, 互不知晓。** 加一种展示 = 加一个消费 report 的函数, 数据层零改动。这跟 D3 的"registry 装实现"是同一族思想: **把"会一直变的"(展示) 和"基本不变的"(数据结构) 物理隔开**。

ValidationReport 自身的三条纪律:

1. **派生属性永不存值**——overall_severity / exit_code / counts 全部从 results 现算。存了值就有第二份真相, 测试时还得测"两份有没有同步"这种本不该存在的问题。
2. **to_dict 是 schema 承诺**——report.json 会被监控/趋势/BI 读, 字段**只增不改不删** (additive schema)。今天随手改个 key 名, 三个月后下游图表集体变灰, 没人知道为什么。
3. **先立字段, 分批填值**——fingerprint / profile / operator 等字段现在就立 (默认 None), 阶段 12-14 的生产者逐个接入。合同先行的纪律, 在数据层重演一遍。

## 7.3 report.py 完整代码

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  :report.py.py
# @Time      :2026/6/10 15:27:21
# @Author    :雨霓同学
# @Project   :ODPlatform
# @Function  :
# apps/platform/src/od_platform/data_validation/report.py
"""ValidationReport — 一次验证的完整产出 (纯数据)。

设计原则: 只装数据 + 派生属性 + to_dict。
所有"怎么展示给人看"的逻辑归 render.py — 这是 D4 撞墙③ 的解药。

派生属性 (永不存值):
    - overall_severity:   全部结果里最严重的级别
    - counts_by_severity: 各级别计数
    - exit_code:          0/1/2 — Unix 退出码语义
    - failed_results:     非 PASS 的 results 子集
    - report_path:        run_dir/report.json (run_dir 未设则 None)
"""
from __future__ import annotations

from dataclasses import dataclass
from pathlib import Path
from typing import Any, Dict, List, Optional

from od_platform.data_validation.registry import CheckResult, CheckSeverity
from od_platform.data_validation.snapshot import DatasetSnapshot


@dataclass
class ValidationReport:
    """一次验证的完整产出。

    Args:
        run_id:            形如 '20260516_184523' 的时间戳 ID
        yaml_path:         验证的 yaml 文件路径
        snapshot:          一次扫描产物 (含 stats_per_split / class_names 等)
        results:           各 check 的 CheckResult 列表 (按注册顺序)
        duration_seconds:  整体耗时
        started_at_iso:    起始时间 ISO 8601 字符串 (UTC)
        run_dir:           本次运行的产出目录 (None = 不写盘模式)
        fingerprint:       数据集指纹 dict (fingerprint.compute_fingerprint 产物)
        profile:           画像聚合 dict (profiler.run_profile 产物; None = 未开启)
        operator:          操作人 (审计字段; CLI 默认取系统用户名, 可 --operator 覆盖)
        tool_version:      工具版本 (od_platform.__version__)
        environment:       运行环境 {platform, python, hostname}

    schema 承诺: 新字段只增不改 — 老消费者读 report.json 不受影响 (additive)。
    """
    run_id:           str
    yaml_path:        Path
    snapshot:         DatasetSnapshot
    results:          List[CheckResult]
    duration_seconds: float
    started_at_iso:   str
    run_dir:          Optional[Path] = None
    fingerprint:      Optional[Dict[str, Any]] = None
    profile:          Optional[Dict[str, Any]] = None
    operator:         Optional[str] = None
    tool_version:     Optional[str] = None
    environment:      Optional[Dict[str, str]] = None

    # ---------- 派生属性 (从 results 算, 不存值) ----------

    @property
    def overall_severity(self) -> str:
        """全部结果里最严重的 severity。

        无结果时返回 PASS — 'nothing went wrong' 的合理默认。
        """
        if not self.results:
            return CheckSeverity.PASS
        return max(
            self.results,
            key=lambda r: CheckSeverity.rank(r.severity),
        ).severity

    @property
    def counts_by_severity(self) -> Dict[str, int]:
        """{ERROR: 0, WARNING: 1, INFO: 2, PASS: 1} 之类。"""
        counts: Dict[str, int] = {}
        for r in self.results:
            counts[r.severity] = counts.get(r.severity, 0) + 1
        return counts

    @property
    def exit_code(self) -> int:
        """Unix 退出码:
            0 — PASS / 只有 INFO
            1 — 有 WARNING
            2 — 有 ERROR
        """
        sev = self.overall_severity
        if sev == CheckSeverity.ERROR:
            return 2
        if sev == CheckSeverity.WARNING:
            return 1
        return 0

    @property
    def failed_results(self) -> List[CheckResult]:
        """非 PASS / 非 INFO 的结果 (即 WARNING + ERROR)。"""
        return [r for r in self.results if not r.passed]

    @property
    def report_path(self) -> Optional[Path]:
        """JSON 报告路径 — run_dir 未设时返回 None。"""
        return (self.run_dir / "report.json") if self.run_dir else None

    @property
    def markdown_path(self) -> Optional[Path]:
        """Markdown 报告路径 — run_dir 未设时返回 None。"""
        return (self.run_dir / "report.md") if self.run_dir else None

    # ---------- JSON 序列化 ----------

    def to_dict(self) -> Dict[str, Any]:
        """供 JSON 报告序列化。结构稳定 — 监控 / 趋势分析可以读这份。

        D4 增量字段 (tool_version / operator / environment / fingerprint /
        profile / class_instances) 全部为【新增键】, 不动旧键 — additive schema。
        """
        return {
            "run_id":           self.run_id,
            "yaml_path":        str(self.yaml_path),
            "task_type":        self.snapshot.task_type,
            "started_at":       self.started_at_iso,
            "duration_seconds": round(self.duration_seconds, 3),
            "overall_severity": self.overall_severity,
            "exit_code":        self.exit_code,
            "counts":           self.counts_by_severity,
            "tool_version":     self.tool_version,
            "operator":         self.operator,
            "environment":      self.environment,
            "dataset_summary": {
                "nc":          self.snapshot.nc,
                "class_names": list(self.snapshot.class_names),
                "stats_per_split": {
                    split: {
                        "image_count":     stat.image_count,
                        "annotated_count": stat.annotated_count,
                        "total_instances": stat.total_instances,
                        "class_instances": {
                            (
                                self.snapshot.class_names[cid]
                                if cid < len(self.snapshot.class_names)
                                else f"id_{cid}"
                            ): n
                            for cid, n in sorted(stat.class_instances.items())
                        },
                    }
                    for split, stat in self.snapshot.stats_per_split.items()
                },
            },
            "fingerprint": self.fingerprint,
            "results": [
                {
                    "name":     r.name,
                    "severity": r.severity,
                    "summary":  r.summary,
                    "details":  r.details,
                }
                for r in self.results
            ],
            "profile": self.profile,
        }
```

⚠️ **设计点 (exit_code 的三档为什么长这样)**: `0 = PASS/INFO`, `1 = WARNING`, `2 = ERROR`——这是给 CI 的 API。INFO 归 0 是刻意的: INFO 的定义就是"知道一下, 不拦你", 如果它返回非零, 团队很快会学会无脑加 `|| true`, 整个闸门就废了。**告警疲劳是质量闸门的头号死因, 分级的全部意义就是防它。**

## 7.4 render_to_logger — 给工程师的三段式

渲染层第一位客户是终端前的工程师。三段式: **报告头 (run/task/耗时/总评级) → 数据集摘要 (类别 + 各划分三个数字) → 检查项一览 (一行一个) → 失败详情 (仅当有非 PASS)**。前三段全量代码平铺直叙, 见仓库 `render.py`; 值得讲的是第四段的 **well-known keys 模式**:

> ⚠️ **设计点 (终端表格对齐复用 D1 的 `string_utils`, 别用 `str.ljust`)**: "数据集摘要"和"检查项一览"这两段是**定宽终端表格**, 列里常有中文类别名 (`head` / `反光衣` 混排)。终端里中文占 2 个显示宽度, 直接 `f"{name:<20}"` / `name.ljust(20)` 数的是字符数, 中英混排必然错位 (这正是 D1 撞墙⑤)。所以这两段的列对齐**一律走 `od_platform.common.string_utils` 的 `pad_to_width` / `format_table_row`**——和 D3 的 `report.py` 排版用的是同一个工具。注意区分: 这是**终端**输出 (要手动补空格对齐); 而 `render_markdown.py` 产出的是 **Markdown 源码** (`| col | col |`), 由 Markdown 渲染器自己对齐, **绝不能**手动 pad——两个渲染器, 两套对齐规则, 别混。

```python
from od_platform.common.string_utils import pad_to_width, format_table_row  # 终端段对齐, 同 D3 report.py

        for p in det["problems"]:
            logger.info(f"        - {p}")
    elif r.name == "pair_existence":
        mps = det.get("missing_per_split", {})
        ...
    # ... 每个想要定制展开的 check 一个分支 ...
    else:
        if "reason" in det:          # 通用 fallback: 至少展示 reason
            logger.info(f"        reason: {det['reason']}")
```

⚠️ **设计点 (展示层的 if/elif 为什么是合理的)**: 阶段 1 我们刚批判过 if/elif, 这里怎么又用上了? 区别在**变化频率和影响面**——调度层的 if/elif 每加一个 check 必改且影响全体 (所以用注册表消灭); 展示层的分支**只在某个 check 想定制展开时**才加一条, 不加就走 fallback, 互相零影响。**模式是用来解决问题的, 不是用来供着的**: if/elif 在"低频、局部、有兜底"的场景里就是最朴素正确的写法。

```bash

## 8.1 端到端函数 (主干版)

把"yaml 路径 → 报告"串成一个函数。**注意: 下面是主干版**——阶段 12-14 会往标注的三个位置接入指纹/画像/Markdown, 终态全文在阶段 14。先把主干读熟:

```python
def validate_dataset(yaml_path, task_type=None, run_id=None,
                     run_dir=None, write_report=True, ...):
    resolved_run_id  = run_id or datetime.now().strftime("%Y%m%d_%H%M%S")
    resolved_run_dir = run_dir or (validation_run_dir(resolved_run_id)
                                   if write_report else None)
    if write_report and resolved_run_dir is not None:
        resolved_run_dir.mkdir(parents=True, exist_ok=True)

    t0 = time.perf_counter()
    started_iso = datetime.now(timezone.utc).isoformat()

    snapshot = build_snapshot(yaml_path=yaml_path, task_type=task_type)
    ctx      = CheckContext(yaml_path=yaml_path, snapshot=snapshot)   # ◀ 阶段12: +options
    results  = run_all_checks(ctx)
                                                                      # ◀ 阶段13: 指纹+画像
    report = ValidationReport(run_id=..., snapshot=snapshot,
                              results=results, duration_seconds=..., ...)

    if write_report and resolved_run_dir is not None:
        (resolved_run_dir / "report.json").write_text(
            json.dumps(report.to_dict(), indent=2, ensure_ascii=False), "utf-8")
                                                                      # ◀ 阶段14: +report.md
    return report
```

⚠️ **设计点 (为什么是函数不是 Validator 类)**: 它没有需要跨调用维持的状态——一次调用, 一次产出。没有状态就不要类, "面向对象"不是把每个动词都包成名词。等真出现"复用扫描结果连验三个数据集"的需求再谈 (TODO D4.8)。

⚠️ **设计点 (run_id 让调用方可注入)**: 默认时间戳, 但测试要传固定值 (产物路径可断言), CI 要传 pipeline 号 (跟构建系统对账)。**一切"默认自动生成"的标识, 都留显式注入口**——这条小习惯让阶段 15 的测试少受很多罪。

## 8.2 CLI: 退出码是给机器的第一公民

```python
# cli/validate_data.py 骨架 (终态参数见阶段 14)
from od_platform.common.constants import Task           # 复用 D3 的任务类型 (--task choices 用)
from od_platform.common.refs import resolve_yaml   # ← 复用 D3 阶段 8.1 留给"验证子系统"的接缝

def main() -> int:
    args = _build_parser().parse_args()
    logger = get_logger(base_path=LOGGING_DIR, log_type="validate",
                        log_level=logging.DEBUG if args.verbose else logging.INFO)
    try:
        # D3 的 refs.resolve_yaml 同时吃"裸名字"和"路径":
        #   --dataset helmet → configs/datasets/helmet.yaml (裸名字走约定目录)
        #   --yaml ./a.yaml  → 原样解析该路径
        # 两条入口收敛到同一个解析器, 不再自己手搓 .resolve()。
        ref = args.dataset if args.dataset else args.yaml
        yaml_path = resolve_yaml(ref)
        report = validate_dataset(yaml_path=yaml_path, task_type=args.task,
                                  write_report=not args.no_report)
        render_to_logger(report, logger, report_path=report.report_path)
        return report.exit_code
    except KeyboardInterrupt:
        return 3
    except Exception:
        logger.exception("未预期异常 — CLI bug, 不是数据问题")
        return 3
```

> ⚠️ **设计点 (复用 D3 的 `resolve_yaml`, 别再手搓路径解析)**: D3 阶段 8.1 当初留了四个 `resolve_*` 接缝, 其中 `resolve_yaml` 的注释白纸黑字写着"给【验证】子系统的接缝"——就是为今天这个 CLI 准备的。它已经把"裸名字 → `configs/datasets/<name>.yaml`、带分隔符/绝对路径 → 原样"的纯语法判别做好了。D4 直接调它, 既省一段重复逻辑, 又让"该去哪找 yaml"这件事**全项目只有一个定义处** (SSoT)。自己写 `args.yaml.resolve()` 等于在 D3 已经铺好的路旁边又踩一条小道——两条路早晚会因为"名字还是路径"的判别规则不一致而打架。

| 退出码 | 含义 | CI 的反应 |
|---|---|---|
| 0 | PASS / 仅 INFO | 放行 |
| 1 | 有 WARNING | 标黄, 需负责人确认 (或团队约定直接拦) |
| 2 | 有 ERROR | 拦下 |
| 3 | Ctrl-C / CLI 自身异常 | 拦下并报 infra 问题——**这是工具的锅, 不是数据的锅** |

⚠️ **设计点 (3 为什么必须独立)**: 把"工具崩了"和"数据有 ERROR"混成同一个码, 值班的人会按"数据问题"流程去找标注团队——找错门。**退出码是错误责任的第一次路由**, 路由错了, 后面全错。

`--dataset NAME` (日常: 按约定找 `configs/datasets/<name>.yaml`) 与 `--yaml PATH` (调试: 直接给路径) 互斥二选一——约定优于配置, 但永远留逃生门。

## 9.2 bbox_within_image — 完整代码 (本阶段精讲)

分级先想清楚。出图框**可训** (加载器 clip 后还是个框), 所以不是无脑 ERROR; 但**占比是流程健康度的信号**:

- 个别出图 (< 10%) → 标注员手抖, WARNING, 修了再训更好
- 成片出图 (≥ 10%) → **坐标系换算/增强导出的系统性 bug**, ERROR, 必须修流水线——这种比例下 clip 出来的数据集和标注意图已是两个东西

还有一个容易翻车的细节: 浮点。标注工具导出 `0.5000000001` 这种尾差很常见, 直接拿 `> 1.0` 判会把数值噪声当事故——**留 `1e-6` 容差**, 阈值照例进 constants:

```python
# constants.py 增量
BBOX_EDGE_TOLERANCE: float = 1e-6   # 吞浮点尾差, 不把数值噪声当事故
BBOX_OUT_ERROR_RATIO: float = 0.10  # 出图占比 >= 10% → 系统性错误, ERROR
```

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  :bbox_within_image.py
# @Project   :ODPlatform
# @Function  :框体出图检测 — label_format 坐标范围检查的盲区补位
# apps/platform/src/od_platform/data_validation/checks/bbox_within_image.py
"""bbox_within_image check — 检测"四个坐标各自合法、但框体伸出图像"的框。

盲区成因 (这是 label_format 抓不住的):
    label_format 验证 cx / cy / w / h 各自 ∈ [0,1], 但 YOLO 格式的框边缘是
    组合量: left = cx - w/2, right = cx + w/2 (top/bottom 同理)。
    例如 cx=0.9, w=0.4 → 四值全合法, 但 right=1.1 — 框有 1/4 在图像外。
    这是标注工具坐标系换算 / 增强后未裁剪导出的最高频错误之一。
    ultralytics 加载时会静默 clip, 框被悄悄改形, 训练"看着正常精度上不去"。

Severity 分级:
    无出图框                          → PASS
    出图框占比 <  BBOX_OUT_ERROR_RATIO → WARNING (个别标注误差, 可 clip 后训练)
    出图框占比 >= BBOX_OUT_ERROR_RATIO → ERROR   (系统性导出错误, 必须修流水线)

边界容差 BBOX_EDGE_TOLERANCE (1e-6) 吞掉浮点尾差, 不把数值噪声当事故。
segment 任务: 多边形每个顶点坐标已被 label_format 限制在 [0,1], 顶点即边界,
不存在"组合量越界"问题 → 直接 PASS (标注语义不适用)。
"""
from __future__ import annotations

from typing import Any, Dict, List

from od_platform.common.constants import (
    BBOX_EDGE_TOLERANCE,
    BBOX_OUT_ERROR_RATIO,
    Task,
)
from od_platform.data_validation.registry import (
    check, CheckContext, CheckResult, CheckSeverity,
)

PREVIEW_LIMIT = 20


def _overflow_amount(coords: List[float]) -> float:
    """coords = [cx, cy, w, h]。返回框边缘越出 [0,1] 的最大量 (0.0 = 未越界)。"""
    cx, cy, w, h = coords
    left, right  = cx - w / 2.0, cx + w / 2.0
    top, bottom  = cy - h / 2.0, cy + h / 2.0
    overflow = max(
        0.0 - left,
        right - 1.0,
        0.0 - top,
        bottom - 1.0,
    )
    return overflow if overflow > BBOX_EDGE_TOLERANCE else 0.0


@check("bbox_within_image")
def validate_bbox_within_image(ctx: CheckContext) -> CheckResult:
    snap = ctx.snapshot

    if snap.task_type != Task.DETECT:
        return CheckResult(
            name="bbox_within_image",
            severity=CheckSeverity.PASS,
            summary=f"task={snap.task_type}, 顶点坐标已由 label_format 覆盖, 本检查不适用",
            details={"reason": "task_not_applicable", "task_type": snap.task_type},
        )

    out_boxes: List[Dict[str, Any]] = []
    total_boxes = 0
    total_out   = 0
    max_overflow = 0.0

    for split, labels in snap.labels_per_split.items():
        for lbl in labels:
            if not lbl.exists():
                continue
            try:
                content = lbl.read_text(encoding="utf-8")
            except OSError:
                continue
            for line_no, line in enumerate(content.splitlines(), 1):
                parts = line.strip().split()
                if len(parts) != 5:
                    continue   # 归 label_format
                try:
                    coords = [float(x) for x in parts[1:5]]
                except ValueError:
                    continue   # 归 label_format
                total_boxes += 1
                overflow = _overflow_amount(coords)
                if overflow > 0.0:
                    total_out += 1
                    max_overflow = max(max_overflow, overflow)
                    if len(out_boxes) < PREVIEW_LIMIT:
                        out_boxes.append({
                            "split":    split,
                            "label":    str(lbl),
                            "line_no":  line_no,
                            "overflow": round(overflow, 6),
                        })

    if total_out == 0:
        return CheckResult(
            name="bbox_within_image",
            severity=CheckSeverity.PASS,
            summary=f"全部 {total_boxes} 个框完整落在图像内",
            details={"total_boxes": total_boxes},
        )

    out_ratio = total_out / max(total_boxes, 1)
    severity = (
        CheckSeverity.ERROR
        if out_ratio >= BBOX_OUT_ERROR_RATIO
        else CheckSeverity.WARNING
    )
    return CheckResult(
        name="bbox_within_image",
        severity=severity,
        summary=(
            f"{total_out}/{total_boxes} ({out_ratio:.1%}) 个框伸出图像边界 "
            f"(最大越界量 {max_overflow:.4f}) — 加载器会静默 clip, 建议修正标注"
        ),
        details={
            "total_boxes":  total_boxes,
            "total_out":    total_out,
            "out_ratio":    round(out_ratio, 4),
            "max_overflow": round(max_overflow, 6),
            "thresholds": {
                "edge_tolerance": BBOX_EDGE_TOLERANCE,
                "error_at_ratio": BBOX_OUT_ERROR_RATIO,
            },
            "out_boxes_preview": out_boxes,
        },
    )
```

三个值得停一秒的细节:

1. **`_overflow_amount` 是纯函数且返回越界量而不是 bool**——报告里"最大越界量 0.08"比"有越界"有用十倍: 0.0001 的越界和 0.25 的越界, 整改优先级完全不同。**质检的产出要尽量带"量", 量才能排优先级。**
2. **segment 任务直接 PASS 并写明理由**——多边形逐顶点已被 label_format 限制在 [0,1], 顶点即边界, 不存在组合量越界。"不适用"必须**显式声明**而不是沉默跳过: 报告里 14 项永远是 14 项, 审计不用猜"segment 数据集怎么少了一项"。
3. **字段数错/解析错的行直接 continue**——那是 label_format 的案子。**一个问题只许一个 check 报**, 否则同一坏行在报告里出现两次, 计数对不上, 用户开始怀疑整份报告。

## 9.3 box_geometry — 完整代码

退化框是出图框的孪生兄弟: 坐标全合法, 但 `w=0.0005` 在 640px 图上是 **0.32 像素宽**——训练时要么被框架静默丢弃, 要么贡献一个 NaN loss。同一个 check 顺手再抓**极端宽高比** (w/h > 20 或 < 1/20): 拉框手抖出的"细线框", 即便是真目标也对 anchor 匹配极不友好——它复用 D3 立过但一直没有消费者的 `STATS_MAX_ASPECT_RATIO`。

⚠️ **设计点 (孤儿常量是债)**: 一个 constants 里躺着没有任何代码引用的阈值, 说明当初"想做的检查"没落地——**常量表就是功能欠账的台账**。本阶段还掉一笔, 阶段 11 的 class_balance 还另一笔 (`STATS_MAX_IMBALANCE_RATIO`)。

```python
# constants.py 增量
DEGENERATE_BOX_MIN_SIZE: float = 1e-3   # 归一化宽/高低于此 = 退化 (640px 图 ≈ 0.64px)
```

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  :box_geometry.py
# @Project   :ODPlatform
# @Function  :退化框与极端宽高比检测 (讲义阶段 11.2 + 宽高比增量增强)
# apps/platform/src/od_platform/data_validation/checks/box_geometry.py
"""box_geometry check — 检测"坐标合法但几何退化/形态异常"的框。

label_format 已经保证坐标 ∈ [0,1]; 本 check 补它的两个盲区:
    1. 退化框: w≈0 / h≈0 / 面积≈0 — 训练时 ultralytics 静默丢框或 NaN loss
    2. 极端宽高比: w/h 或 h/w > STATS_MAX_ASPECT_RATIO — 大概率是标注误操作
       (拉框时手抖出的细线), 即便是真实目标也对 anchor 匹配极不友好

只对 detect task 检查 (segment 的退化几何是另一回事, 留扩展)。
判定: 有问题框 → WARNING (能训, 但该清理; 不像格式错是结构性 ERROR)。
"""
from __future__ import annotations

from collections import Counter
from typing import Any, Dict, List, Optional, Tuple

from od_platform.common.constants import (
    DEGENERATE_BOX_MIN_SIZE,
    STATS_MAX_ASPECT_RATIO,
    Task,
)
from od_platform.data_validation.registry import (
    check, CheckContext, CheckResult, CheckSeverity,
)

PREVIEW_LIMIT = 20

KIND_ZERO_WIDTH   = "zero_width"
KIND_ZERO_HEIGHT  = "zero_height"
KIND_TINY_AREA    = "tiny_area"
KIND_EXTREME_AR   = "extreme_aspect_ratio"


def _geometry_issue(coords: List[float]) -> Optional[Tuple[str, str]]:
    """coords = [cx, cy, w, h] (已知合法 5 字段)。返回 (kind, detail) 或 None。

    判定顺序: 先退化后宽高比 — 退化框的宽高比本身没有意义, 不重复报。
    """
    _cx, _cy, w, h = coords
    if w < DEGENERATE_BOX_MIN_SIZE:
        return KIND_ZERO_WIDTH, f"w={w:.6f}"
    if h < DEGENERATE_BOX_MIN_SIZE:
        return KIND_ZERO_HEIGHT, f"h={h:.6f}"
    if w * h < DEGENERATE_BOX_MIN_SIZE ** 2:
        return KIND_TINY_AREA, f"area={w * h:.8f}"
    ar = w / h
    if ar > STATS_MAX_ASPECT_RATIO or ar < 1.0 / STATS_MAX_ASPECT_RATIO:
        return KIND_EXTREME_AR, f"w/h={ar:.3f} (阈值 {STATS_MAX_ASPECT_RATIO:g}x)"
    return None


@check("box_geometry")
def validate_box_geometry(ctx: CheckContext) -> CheckResult:
    snap = ctx.snapshot

    if snap.task_type != Task.DETECT:
        return CheckResult(
            name="box_geometry",
            severity=CheckSeverity.PASS,
            summary=f"task={snap.task_type}, 退化几何检查暂仅支持 detect (留扩展)",
            details={"reason": "task_not_applicable", "task_type": snap.task_type},
        )

    issues: List[Dict[str, Any]] = []
    issue_kinds: Counter = Counter()
    total_boxes = 0

    for split, labels in snap.labels_per_split.items():
        for lbl in labels:
            if not lbl.exists():
                continue   # pair_existence 报
            try:
                content = lbl.read_text(encoding="utf-8")
            except OSError:
                continue
            for line_no, line in enumerate(content.splitlines(), 1):
                parts = line.strip().split()
                if len(parts) != 5:
                    continue   # 字段数错归 label_format
                try:
                    coords = [float(x) for x in parts[1:5]]
                except ValueError:
                    continue   # 解析错归 label_format
                total_boxes += 1
                issue = _geometry_issue(coords)
                if issue is not None:
                    kind, detail = issue
                    issue_kinds[kind] += 1
                    if len(issues) < PREVIEW_LIMIT:
                        issues.append({
                            "split":   split,
                            "label":   str(lbl),
                            "line_no": line_no,
                            "kind":    kind,
                            "detail":  detail,
                        })

    if not issue_kinds:
        return CheckResult(
            name="box_geometry",
            severity=CheckSeverity.PASS,
            summary=f"全部 {total_boxes} 个框几何正常 (无退化框/极端宽高比)",
            details={"total_boxes": total_boxes},
        )

    total_issues = sum(issue_kinds.values())
    return CheckResult(
        name="box_geometry",
        severity=CheckSeverity.WARNING,
        summary=(
            f"{total_issues}/{total_boxes} 个框几何异常 "
            f"(退化框/极端宽高比) — 训练前建议清理"
        ),
        details={
            "total_boxes":    total_boxes,
            "total_issues":   total_issues,
            "issue_kinds":    dict(issue_kinds),
            "issues_preview": issues,
            "thresholds": {
                "degenerate_min_size": DEGENERATE_BOX_MIN_SIZE,
                "max_aspect_ratio":    STATS_MAX_ASPECT_RATIO,
            },
        },
    )
```

⚠️ **设计点 (判定顺序: 先退化后宽高比)**: `w=0.0005, h=0.2` 的框, 宽高比是 0.0025——同时命中两条规则。但**退化框的宽高比本身没有意义**, 报两条只会让计数虚高。`_geometry_issue` 用 if 链的先后天然实现"一行只判一种 kind", 顺序即优先级。

## 9.4 duplicate_annotations — 设计卡

| | |
|---|---|
| **事故形态** | ① 完全重复行 (标注平台导出 bug / 文件手工合并); ② 同类近重复框, IoU≥0.95 (标注员对同一目标重复打框) |
| **企业损失** | 重复框让该目标在 loss 里**双倍加权**; 评估时 NMS 留一个、另一个成虚假 FP——**mAP 被悄悄拉低**, 又是"看着对、指标差" |
| **判定规则** | 任何重复 → WARNING (可训但必须清理)。完全重复双任务通用 (纯字符串 Counter); 近重复仅 detect (segment 多边形 IoU 重型, 留扩展) |
| **复杂度账** | IoU 两两比较按【图内同类】分组, 单图同类通常 <50 框, O(n²) 量级可忽略——**先量化再决定要不要上空间索引**, 不为想象中的规模买单 |

```python
DUPLICATE_BOX_IOU_THRESHOLD = 0.95   # constants.py — 同图同类 IoU 阈值

def _iou_xywh(a, b) -> float:
    """两个 (cx, cy, w, h) 归一化框的 IoU。"""
    ax1, ay1 = a[0] - a[2] / 2, a[1] - a[3] / 2
    ax2, ay2 = a[0] + a[2] / 2, a[1] + a[3] / 2
    bx1, by1 = b[0] - b[2] / 2, b[1] - b[3] / 2
    bx2, by2 = b[0] + b[2] / 2, b[1] + b[3] / 2
    inter_w = max(0.0, min(ax2, bx2) - max(ax1, bx1))
    inter_h = max(0.0, min(ay2, by2) - max(ay1, by1))
    inter = inter_w * inter_h
    if inter <= 0.0:
        return 0.0
    union = a[2] * a[3] + b[2] * b[3] - inter
    return inter / union if union > 0 else 0.0

# 主体: ① Counter(lines) 抓完全重复;
#       ② 按 (图, 类) 分组两两 IoU >= 阈值抓近重复, 坐标完全相同的对
#          跳过 (已被①计) — 一个问题只许一个口径计一次。完整文件见仓库。
```

# 阶段 10: 覆盖与资产四连 — 数据管理失控的探测器

几何盲区抓"标错的框"; 这一批抓的是**数据资产管理层面的失控**: 该有的没有 (annotation_coverage / split_presence)、不该有的有 (orphan_labels)、有但说不清归属 (stem_collision)。它们单看都不致命, 但**全是流水线出 bug 的早期信号**——企业质检的价值一半在拦事故, 另一半在**让失控趁早暴露**。

## 10.1 annotation_coverage — 完整代码 (顺手统计第一次兑现)

成品数据集里有多少图最终没有有效标注 (空 txt 或缺 txt)? 少量是正常的 (有意保留的背景图能抑制误检), **大量则通常是标注流程漏了一批**——而"漏了一批没标的"在训练日志里**零症状**, 只会默默稀释正样本。

分级: ≥30% WARNING (该 review 数据来源了), 5%~30% INFO (确认是不是有意的背景图), <5% PASS。

**这个 check 一行盘都不扫**——`image_count` 和 `annotated_count` 在阶段 4 的扫描循环里早就"顺手"算好了, 它只做减法和除法。**铁律 3 的第一笔分红到账**: 写它花了十分钟, 跑它花零毫秒。

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  :annotation_coverage.py
# @Project   :ODPlatform
# @Function  :成品数据集"无有效标注图"占比检测 (讲义阶段 11.3)
# apps/platform/src/od_platform/data_validation/checks/annotation_coverage.py
"""annotation_coverage check — 成品端"无有效标注图"占比。

D3 的覆盖率撞墙看的是【划分前原始数据】的图/标注比; 本 check 看的是
【落盘后成品数据集】里有多少图最终没有有效标注 (空 txt 或缺 txt)。
视角不同, 互补。

零额外 I/O: 直接消费 snapshot.stats_per_split 里已经算好的
image_count / annotated_count — 这正是 snapshot"顺手统计"红利的兑现处。

Severity 分级:
    ratio >= UNLABELED_WARN_RATIO (30%) → WARNING (大量负样本拖垮训练, 该 review)
    ratio >= UNLABELED_INFO_RATIO (5%)  → INFO    (知道一下, 可能是有意的背景图)
    其余                                 → PASS
"""
from __future__ import annotations

from typing import Any, Dict

from od_platform.common.constants import UNLABELED_INFO_RATIO, UNLABELED_WARN_RATIO
from od_platform.data_validation.registry import (
    check, CheckContext, CheckResult, CheckSeverity,
)


@check("annotation_coverage")
def validate_annotation_coverage(ctx: CheckContext) -> CheckResult:
    snap = ctx.snapshot

    total_images    = 0
    total_unlabeled = 0
    per_split: Dict[str, Dict[str, int]] = {}

    for split, st in snap.stats_per_split.items():
        unlabeled = st.image_count - st.annotated_count
        per_split[split] = {"images": st.image_count, "unlabeled": unlabeled}
        total_images    += st.image_count
        total_unlabeled += unlabeled

    if total_images == 0:
        return CheckResult(
            name="annotation_coverage",
            severity=CheckSeverity.INFO,
            summary="无图像可统计",
            details={"reason": "no_images"},
        )

    ratio = total_unlabeled / total_images
    details: Dict[str, Any] = {
        "total_images":    total_images,
        "total_unlabeled": total_unlabeled,
        "unlabeled_ratio": round(ratio, 4),
        "per_split":       per_split,
        "thresholds": {
            "warn_at": UNLABELED_WARN_RATIO,
            "info_at": UNLABELED_INFO_RATIO,
        },
    }

    if ratio >= UNLABELED_WARN_RATIO:
        severity = CheckSeverity.WARNING
        summary  = (
            f"无有效标注图占比 {ratio:.1%} >= {UNLABELED_WARN_RATIO:.0%} "
            f"({total_unlabeled}/{total_images}) — 建议 review 数据来源"
        )
    elif ratio >= UNLABELED_INFO_RATIO:
        severity = CheckSeverity.INFO
        summary  = (
            f"无有效标注图占比 {ratio:.1%} "
            f"({total_unlabeled}/{total_images}) — 若为有意保留的背景图可忽略"
        )
    else:
        severity = CheckSeverity.PASS
        summary  = f"标注覆盖正常 (无标注图 {total_unlabeled}/{total_images} = {ratio:.2%})"

    return CheckResult(
        name="annotation_coverage",
        severity=severity,
        summary=summary,
        details=details,
    )
```

⚠️ **设计点 (跟 D3 覆盖率检查不重复吗?)**: 不重复——D3 在**划分前**看原始图/标注比 (产线投料检查), 本 check 在**落盘后**看成品 (产线出货检查)。投料对、出货错, 恰恰说明产线中段有问题。**同一指标在产线不同位置各测一次, 是制造业质检的常识**, 软件没理由例外。

## 10.2 orphan_labels — 设计卡 (镜像检查)

| | |
|---|---|
| **事故形态** | labels 目录里躺着没有对应图像的 .txt——删图不删标签 / 转换工具多吐文件 / 增量同步脚本漏删 |
| **企业损失** | 对训练**零直接伤害** (加载器按图找标签)。但它让"实例总数"等审计口径虚高, 更是**数据管理失控的信号**——今天多 2 个孤儿, 下个月就是图和标签整体对不上号 |

| **数据来源** | 阶段 4 备好的 `snap.label_files_per_split` (磁盘真实文件) 与图像 stem 集合做差集——零额外 I/O |

## 10.3 stem_collision — 设计卡 (归属歧义)

| | |
|---|---|
| **事故形态** | 同一目录里 `a.jpg` 和 `a.png` 共存 (多来源合并/格式转换残留)。YOLO 路径规则下两张图映射到**同一个** `a.txt` |
| **企业损失** | 两图内容相同 → 训练样本静默重复 (轻); 内容不同 → **至少一张图的标签是错配的** (重, 且全程无报错)。工具读不了人心也不读像素 (轻量原则), 分不清是哪种 |

```python
by_stem: Dict[str, List[str]] = defaultdict(list)
for img in images:

collided = [{"stem": s, "files": sorted(n)} for s, n in by_stem.items() if len(n) > 1]

## 10.4 split_presence — 设计卡 (兑付阶段 4 的欠条)

| | |
|---|---|
| **事故形态** | yaml 声明了 `test: images/test` 但目录不存在/为空——此前它只会变成 snapshot 里一条**没人看的 scan_warning**, 然后划分凭空蒸发而 exit code 还是 0 |
| **企业损失** | "我以为我有测试集"是最危险的状态之一; 另外 val 只有 8 张图时, 任何评估指标的方差都大到没有统计意义, 拿它拍板等于掷骰子 |
| **判定规则** | 取最严: train 完全缺失 → **ERROR** (无训练数据); 任何 scan_warning (声明了但不可用) → WARNING; val 未声明 → WARNING (训练盲飞); test 未声明 → INFO; 可用划分 < `SPLIT_MIN_IMAGES`(30) 张 → INFO (统计意义提示) |

```python
findings: List[Dict[str, str]] = []
for w in snap.scan_warnings:           # ← 阶段 4 的欠条, 在此兑付
    if "split" in w:
        add(CheckSeverity.WARNING, w)
if "train" not in present:
    add(CheckSeverity.ERROR, "train 划分不可用 — 无训练数据, 流程必须终止")
...
severity = max((f["severity"] for f in findings), key=CheckSeverity.rank)
```

⚠️ **设计点 (一个 check 多条 findings 怎么定总级)**: 每条 finding 各带 severity, 总级 = `max(rank)`。details 里**逐条保留各自级别**——报告层能区分"这项 WARNING 里其实混着一条 ERROR 苗子"。一个 check 内部, 复刻了整个 report 的聚合逻辑——**分形结构是好架构的体味**。

# 阶段 11: 类别语义双连 — class_presence / class_balance

前面的 check 都在问"文件和框对不对"; 这一批开始问**"类别学得起来吗"**——这是离算法最近的一层质检, 也是开篇事故三 ("head 类 AP=0 被误诊为模型差") 的正解。两个 check 全部消费阶段 4 顺手统计的 `class_instances`——**红利第二次兑现, 仍然零额外 I/O**。

## 11.1 class_presence — 设计卡

| | |
|---|---|
| **事故形态** | ① val/test 有、train 没有 → 模型从未见过该类, **评估指标必然为 0, 且极易被误读成"模型差"**; ② train 有、val 没有 → 该类训练效果不可评估, 盲区; ③ yaml 声明、全集 0 实例 → 死类别, 类别表与标注早就脱节 |
| **企业损失** | 事故三的两周三人排查; 死类别则让 nc 虚高, 报告/失衡比的分母全部失真 |
| **判定规则** | ①② → WARNING; ③ → INFO。**注意责任边界**: class_id ≥ nc 的越界归 label_format, 这里只看 [0, nc) 内的"出现性"——一个问题一个 check |

```python
for cls_id in range(snap.nc):
    in_train, in_val, in_test = count("train", cls_id), count("val", cls_id), count("test", cls_id)
    if in_train + in_val + in_test == 0:
        findings.append({..., "kind": "declared_unused", "severity": INFO})
    elif has_train and in_train == 0:
        findings.append({..., "kind": "missing_in_train", "severity": WARNING,
            "message": f"类别 '{name}' 在 val/test 出现 {in_val+in_test} 次但 train 为 0 "
                       f"— 模型无法学习, 评估必然失真"})
    elif has_train and has_val and in_train > 0 and in_val == 0:
        findings.append({..., "kind": "missing_in_val", "severity": WARNING})
```

⚠️ **设计点 (message 要替读者把"后果"说出来)**: 不是 "class 0: train=0, val=30" 这种裸数字, 而是**直接写明 "评估必然失真"**——质检报告的读者里有不写代码的角色, 数字 + 后果才是完整的信息。开篇事故三如果当年报告里有这一句, 三个 backbone 就不用白换。

## 11.2 class_balance — 设计卡 (还掉第二枚孤儿常量)

| | |
|---|---|
| **事故形态** | 长尾: train 里 person 4000 实例, head 80 实例——模型必然偏科 |
| **定位 (克制)** | 长尾的 action 在**数据集设计层** (定向采集/重采样/class_weights), 本 check **只量化分级提醒, 不修复也不建议具体方案细节**——质检越界开药方, 出了事说不清 |
| **判定规则** | 失衡比 = max(实例数) / min(**非零类**实例数)。≥50x → WARNING (复用孤儿常量 `STATS_MAX_IMBALANCE_RATIO`); ≥10x → INFO (新常量 `CLASS_IMBALANCE_INFO_RATIO`); 每个划分各算, 总级取最严 |
| **边界** | 零实例类**不进比值**——否则除零, 且它的语义是"缺类"(class_presence 的案子) 不是"失衡"。**同一现象按语义分案, 比按表象合案更经得起审计** |

```python
nonzero = {c: n for c, n in stat.class_instances.items() if n > 0}
if len(nonzero) >= 2:
    max_cls, min_cls = max(nonzero, key=nonzero.get), min(nonzero, key=nonzero.get)
    ratio = nonzero[max_cls] / nonzero[min_cls]
    # >= 50x WARNING / >= 10x INFO / else PASS; details 里带 max/min 类名与实例数
```

报告里这一项的 summary 长这样 (真实输出):

```
[INFO] class_balance: train 失衡比 29.0x (>= 10x) — 长尾提醒, 训练时建议关注 class_weights / 重采样
```

# 阶段 12: ValidationOptions 启用 + image_integrity (重型可选)

## 12.1 为什么这个 check 不敢默认开

前 13 个 check 有一个共同的隐形前提: **它们都不解码图像**。读 yaml、读标签文本、stat 文件——单位成本是微秒级, 50 万张图也敢秒级跑完。

image_integrity 不一样: 它要**逐张打开并解码图像**——抓损坏文件、JPEG 截断、EXIF 旋转标记。单张毫秒级到几十毫秒, 50 万张就是**几十分钟**。把它混进默认全家桶, 质量闸门从 0.2 秒变 40 分钟, 阶段 8 接好的 CI 会被第一个用户关掉——**一个慢的闸门, 等于没有闸门**。

所以它是 D4 第一个 (也是本课唯一一个) **重型可选** check: `--check-images` 显式开启。什么时候开? **接入第三方/外包数据时、交付验收时**——这两个场景里"图像本身是好的"恰恰是最不能假设的。

## 12.2 阶段 2 的伏笔回收: options 走 CheckContext, 签名一行不动

开关怎么传到 check 手里? 阶段 2 立合同时 `ValidationOptions` 和 `CheckContext.options` 字段就已就位, 现在只接两根线:

```python
# service.validate_dataset 签名增参 + 透传 (终态全文见阶段 14)
def validate_dataset(..., options: Optional[ValidationOptions] = None, ...):
    resolved_options = options or ValidationOptions()
    ctx = CheckContext(yaml_path=yaml_path, snapshot=snapshot,
                       options=resolved_options)
```

**14 个 check 的函数签名零改动。** 如果阶段 2 当初图省事让 check 直接收 `yaml_path`, 此刻就是 14 个文件的签名大手术 + 14 次 review——CheckContext 那层"看似多余"的包装, 保费在这一天全额赔付。

## 12.3 关着也要报: 审计语义

被关闭的检查项该怎么表现? 本能答案是"跳过, 别出现"——**企业答案恰好相反**:

> 报告必须如实记录"哪些项【没有】被验证"。

所以关闭时它返回一条 **INFO**: "图像完整性检查未开启 (--check-images)"。三个月后审收这份报告的人, 看到的是"这批数据没验过图像可解码性"这个**事实**, 而不是一片看似全面的绿色。**沉默的跳过是在替未来的读者做"这不重要"的决定——质检工具没有这个权力。**

## 12.4 完整代码

检测项五件套: 0 字节 (传输中断典型形态) / verify+load 双段解码 (verify 抓结构损坏, **load 才抓得住 JPEG 截断**——只 verify 是经典假阴性坑) / EXIF Orientation≠1 (部分加载器按它旋转、部分不旋转, **框和像素错位**——"标注没错、训练全错"的经典) / 通道模式分布 / 分辨率分布 (顺手为阶段 14 的图像级画像供数)。

分级: 损坏占比 ≥10% → ERROR (交付级问题); 任何损坏 → WARNING (个别坏图, 加载器会跳过); 仅 EXIF/模式提示 → INFO。

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  :image_integrity.py
# @Project   :ODPlatform
# @Function  :图像完整性检测 (重型可选, D4.2) — 损坏/0字节/EXIF方向/通道模式
# apps/platform/src/od_platform/data_validation/checks/image_integrity.py
"""image_integrity check — 逐张解码验证图像完整性 (重型, 默认关闭)。

为什么默认关: 必须逐张打开并解码图像, I/O 与 CPU 成本跟数据集体量线性
增长 — 与其它 check"复用 snapshot 零额外 I/O"的量级完全不同。
依据讲义 D4.2 的定位: 重型可选, 痛点驱动, --check-images 显式开启。

为什么关着也要"报一句 INFO": 企业审计语义 — 报告必须如实记录
"哪些项【没有】被验证", 而不是沉默 (旧版报告的"图片可读性验证: 未开启"
正是这个用法)。接入第三方数据源时强烈建议开启。

检测项 (开启后):
    1. 0 字节文件                       → 损坏 (传输中断典型形态)
    2. 无法解码 / 截断 (verify + load)  → 损坏 (decode 才能抓 JPEG 截断)
    3. EXIF Orientation != 1            → 风险: 部分加载器按 EXIF 旋转、
       部分不旋转, 框与像素错位 — 经典的"标注没错、训练全错"
    4. 通道模式分布 (RGB/L/P/RGBA...)   → 画像信息; P 模式带透明在部分
       加载器下产生告警/失真
    5. 分辨率分布 (min/max/Top 常见)    → 供报告层做图像级画像与像素口径换算

Severity:
    损坏占比 >= IMAGE_CORRUPT_ERROR_RATIO → ERROR   (交付级问题)
    任何损坏                              → WARNING (个别坏图, 加载器会跳过)
    仅 EXIF 方向 / 模式提示               → INFO
    全部正常                              → PASS
"""
from __future__ import annotations

from collections import Counter
from typing import Any, Dict, List

from od_platform.common.constants import IMAGE_CORRUPT_ERROR_RATIO
from od_platform.data_validation.registry import (
    check, CheckContext, CheckResult, CheckSeverity,
)

PREVIEW_LIMIT = 20
_EXIF_ORIENTATION_TAG = 274   # PIL ExifTags.Base.Orientation


@check("image_integrity")
def validate_image_integrity(ctx: CheckContext) -> CheckResult:
    if not ctx.options.check_images:
        return CheckResult(
            name="image_integrity",
            severity=CheckSeverity.INFO,
            summary="图像完整性检查未开启 (--check-images) — 接入第三方数据时建议开启",
            details={"reason": "disabled", "enable_flag": "--check-images"},
        )

    try:
        from PIL import Image
    except ImportError:
        return CheckResult(
            name="image_integrity",
            severity=CheckSeverity.WARNING,
            summary="已请求图像完整性检查但 Pillow 未安装 (pip install pillow)",
            details={"reason": "pillow_missing"},
        )

    snap = ctx.snapshot

    corrupt: List[Dict[str, Any]] = []
    exif_rotated: List[str] = []
    total_images   = 0
    total_corrupt  = 0
    total_exif     = 0
    mode_counter:  Counter = Counter()
    size_counter:  Counter = Counter()
    min_dim: Any = None
    max_dim: Any = None

    for split, images in snap.images_per_split.items():
        for img_path in images:
            total_images += 1

            # ---- 1. 0 字节 ----
            try:
                if img_path.stat().st_size == 0:
                    total_corrupt += 1
                    if len(corrupt) < PREVIEW_LIMIT:
                        corrupt.append({
                            "split": split, "image": str(img_path),
                            "kind": "zero_byte",
                        })
                    continue
            except OSError as e:
                total_corrupt += 1
                if len(corrupt) < PREVIEW_LIMIT:
                    corrupt.append({
                        "split": split, "image": str(img_path),
                        "kind": "stat_failed", "error": str(e),
                    })
                continue

            # ---- 2. verify (结构) + load (截断) ----
            try:
                with Image.open(img_path) as im:
                    im.verify()
                with Image.open(img_path) as im:
                    im.load()
                    w, h = im.size
                    mode = im.mode
                    try:
                        orientation = im.getexif().get(_EXIF_ORIENTATION_TAG, 1)
                    except Exception:
                        orientation = 1
            except Exception as e:
                total_corrupt += 1
                if len(corrupt) < PREVIEW_LIMIT:
                    corrupt.append({
                        "split": split, "image": str(img_path),
                        "kind": "decode_failed",
                        "error": f"{type(e).__name__}: {e}",
                    })
                continue

            # ---- 3/4/5. 画像信息 ----
            mode_counter[mode] += 1
            size_counter[(w, h)] += 1
            area = w * h
            if min_dim is None or area < min_dim[0] * min_dim[1]:
                min_dim = (w, h)
            if max_dim is None or area > max_dim[0] * max_dim[1]:
                max_dim = (w, h)
            if orientation != 1:
                total_exif += 1
                if len(exif_rotated) < PREVIEW_LIMIT:
                    exif_rotated.append(str(img_path))

    corrupt_ratio = total_corrupt / max(total_images, 1)
    details: Dict[str, Any] = {
        "total_images":     total_images,
        "total_corrupt":    total_corrupt,
        "corrupt_ratio":    round(corrupt_ratio, 4),
        "exif_rotated_count": total_exif,
        "mode_distribution": dict(mode_counter),
        "resolution_top": [
            {"size": f"{w}x{h}", "count": n}
            for (w, h), n in size_counter.most_common(10)
        ],
        "resolution_min": f"{min_dim[0]}x{min_dim[1]}" if min_dim else None,
        "resolution_max": f"{max_dim[0]}x{max_dim[1]}" if max_dim else None,
        "corrupt_preview":      corrupt,
        "exif_rotated_preview": exif_rotated,
        "thresholds": {"corrupt_error_at_ratio": IMAGE_CORRUPT_ERROR_RATIO},
    }

    if total_corrupt > 0:
        severity = (
            CheckSeverity.ERROR
            if corrupt_ratio >= IMAGE_CORRUPT_ERROR_RATIO
            else CheckSeverity.WARNING
        )
        summary = (
            f"{total_corrupt}/{total_images} ({corrupt_ratio:.1%}) 张图像损坏/不可解码"
            + (f"; 另有 {total_exif} 张带 EXIF 旋转标记" if total_exif else "")
        )
    elif total_exif > 0:
        severity = CheckSeverity.INFO
        summary = (
            f"图像全部可解码; {total_exif} 张带 EXIF 旋转标记 "
            f"— 不同加载器旋转行为不一, 存在框/像素错位风险, 建议落盘前归一化"
        )
    else:
        severity = CheckSeverity.PASS
        summary = f"全部 {total_images} 张图像完整可解码, 无 EXIF 旋转标记"

    return CheckResult(
        name="image_integrity",
        severity=severity,
        summary=summary,
        details=details,
    )
```

⚠️ **设计点 (Pillow 是可选依赖, 缺了优雅降级)**: 用户开了 `--check-images` 但环境没装 Pillow → 返回 **WARNING** "已请求但无法执行"——不是 ImportError 崩掉 (违反聚合承诺), 也不是默默 PASS (撒谎)。**"你要的我做不了"本身是个需要如实上报的结果。** 可选依赖在 check 函数内部 lazy import, 不开就不付导入成本。

⚠️ **设计点 (EXIF 为什么只到 INFO)**: 带旋转标记不等于数据错——取决于标注时用的工具怎么处理 EXIF。它是**风险信号**不是缺陷判定, 升 WARNING 会制造大批假阳性, 告警疲劳又来了。报告里给出数量 + 建议 ("落盘前归一化"), 决定权留给人。

## 13.1 撞墙⑤: "把统计塞进报告"的两次失败

14 项判定齐了, 但四个客户里还有两个没伺候好: 算法工程师要**分布** ("各类别框多大? 每图多密? 小目标占比?"), 标注团队要**整改清单** ("出图的是哪 37 个框?")。本能方案一: 加一个 `dataset_statistics` check, 把统计塞进它的 details。

跑一下 540 图 / 6649 实例的安全帽数据集——details 里光"每实例几何"就是 6649 条记录, **report.json 从 30KB 涨到 3MB**; 换 50 万图的数据集, 报告直接 GB 级, 没有任何下游打得开。方案一卒。

本能方案二: 那就只放聚合, 明细砍掉。报告瘦回来了, 但标注团队的问题回到原点——"哪 37 个框?" 答不出, 他们就得全量重审 5 万张图。方案二卒。

两次失败指向同一个结论: **"给人看的"和"给机器/团队用的"是两种产物, 强行塞进一个容器, 必然顾此失彼。**

## 13.2 解药: 双轨设计 + 判定/画像分离

```
                    ┌─ 轨道 A (明细): instances.csv / per_image.csv
profiler ──单趟──▶ │   每个 bbox 一行 × 20 列含全部异常标记
                    │   客户: 标注团队(整改清单) / BI / 主动学习选样
                    └─ 轨道 B (聚合): profile dict → report.json + report.md

                        客户: 报告读者 — 永远只看聚合 + 锚点
```

同时立一条与 check 的分工铁律——本课第二条贯穿性设计原则在此落地:

| | check | profiler |
|---|---|---|
| 回答 | **能不能训** (判定) | **长什么样** (统计) |
| 产出 | severity | 数字, **永不产 severity** |
| 失败影响 | 决定 exit_code | **绝不阻断**——profiler 崩了, 验证照常出结论 |
| 异常口径 | constants | **同一份 constants** |

⚠️ **设计点 (口径同源是审计生命线)**: profiler 给每个实例打 degenerate / out_of_image 等标记, 判定逻辑与对应 check **引用同一批常量** (`_instance_flags` 里逐条对照)。否则迟早出现"报告说 3 个退化框、CSV 标了 5 个"——两个数字一打架, 用户对整份报告的信任清零。**同一个口径, 只许有一个定义处。**

## 13.3 工程约束: 单趟 + 流式 + 堆

50 万图的约束倒推三条实现纪律:

1. **单趟扫描**: 解析一次标签, 同时写 CSV + 累加聚合——不再扫第二遍盘
2. **流式写 CSV**: 边解析边 writerow, **内存里永不囤实例列表**——百万实例占用恒定
3. **Top-N 用堆**: "最小面积 Top-20"维护一个 20 元素的堆, 不是排序全量

def _parse_line_geometry(
    parts: List[str], task_type: str
) -> Optional[Tuple[int, float, float, float, float]]:
    """从一行标签解析出 (class_id, cx, cy, w, h)。

    detect : 直接取 5 字段
    segment: 取多边形外接框 (画像口径; 分割的逐点几何归未来扩展)

        cls_id = int(parts[0])
    except (ValueError, IndexError):
        return None

    if task_type == Task.DETECT:
        if len(parts) != 5:
            return None
        try:
            cx, cy, w, h = (float(x) for x in parts[1:5])
        except ValueError:
            return None
        return cls_id, cx, cy, w, h

    if task_type == Task.SEGMENT:
        if len(parts) < 7 or (len(parts) - 1) % 2 != 0:
            return None
        try:
            coords = [float(x) for x in parts[1:]]
        except ValueError:
            return None
        xs, ys = coords[0::2], coords[1::2]
        x1, x2, y1, y2 = min(xs), max(xs), min(ys), max(ys)
        return cls_id, (x1 + x2) / 2, (y1 + y2) / 2, x2 - x1, y2 - y1

    return None

## 13.5 关键代码 (二): 分布一致性的量化 — JSD

旧式报告写"train 与 val 分布存在轻微偏差"——**"轻微"是谁的轻微? 没有口径的形容词在评审会上一文不值。** 量化方案: 类别实例占比的 **Jensen-Shannon 散度** (log2 底, 值域 [0,1], 对称且对零占比稳健——KL 散度在某类占比为 0 时直接无穷大, 不适合长尾数据):

```python
def _jsd(p: List[float], q: List[float]) -> float:
    """Jensen-Shannon 散度 (log2, 值域 [0,1]) — 类别分布一致性量化。"""
    def _kl(a: List[float], b: List[float]) -> float:
        return sum(
            x * math.log2(x / y) for x, y in zip(a, b) if x > 0 and y > 0
        )
    m = [(x + y) / 2 for x, y in zip(p, q)]
    return 0.5 * _kl(p, m) + 0.5 * _kl(q, m)


# ============================================================
# 公开 API
```

## 13.7 fingerprint — 审计的第一问 (完整代码)

报告做得再好, 审计第一问就能问倒: **"这份报告对应的到底是哪份数据?"** 报告归档了, 数据后来被人动过没有? 没有指纹, 答案只能靠赌咒发誓。

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  :fingerprint.py
# @Project   :ODPlatform
# @Function  :数据集指纹 — 回答"这份报告对应的到底是哪份数据"
# apps/platform/src/od_platform/data_validation/fingerprint.py
"""数据集指纹 — 企业审计的第一问: 报告对应哪份数据?

设计取舍 (有意为之, 不是偷懒):
    - 指纹 = sha256(排序后的 "split/文件名:字节数" 清单)
      能抓: 增删文件 / 改名 / 文件大小变化 — 覆盖 95% 的"数据悄悄变了"
      抓不住: 同字节数的原地内容篡改 — 这需要逐文件内容哈希,
      I/O 成本与数据集体量线性增长, 属于重型可选 (留 --deep-fingerprint 扩展位)
    - 清单只含文件名不含绝对路径 — 同一份数据换机器/换盘符指纹不变,
      这才是"数据指纹"而不是"部署指纹"

产出字段全部进 report.json 与 Markdown 报告的元数据区。
"""
from __future__ import annotations

import hashlib
from datetime import datetime, timezone
from typing import Any, Dict

from od_platform.data_validation.snapshot import DatasetSnapshot


def compute_fingerprint(snapshot: DatasetSnapshot) -> Dict[str, Any]:
    """对快照中的全部图像与标签文件计算清单指纹。

    Returns:
        {
            manifest_sha256:  清单哈希 (数据身份证)
            total_files:      参与指纹的文件数
            total_bytes:      参与指纹的总字节数
            per_split:        {split: {images, label_files, bytes}}
            generated_at:     UTC ISO 时间戳
            algorithm:        指纹算法说明 (审计可读)
        }
    """
    lines = []
    total_bytes = 0
    per_split: Dict[str, Dict[str, int]] = {}

    for split in snapshot.splits:
        split_bytes = 0
        images = snapshot.images_per_split.get(split, ())
        label_files = snapshot.label_files_per_split.get(split, ())

        for path in list(images) + list(label_files):
            try:
                size = path.stat().st_size
            except OSError:
                size = -1   # 不可读也要进清单 — "有个读不到的文件"本身是身份的一部分
            lines.append(f"{split}/{path.name}:{size}")
            if size > 0:
                split_bytes += size

        per_split[split] = {
            "images":      len(images),
            "label_files": len(label_files),
            "bytes":       split_bytes,
        }
        total_bytes += split_bytes

    lines.sort()
    digest = hashlib.sha256("\n".join(lines).encode("utf-8")).hexdigest()

    return {
        "manifest_sha256": digest,
        "total_files":     len(lines),
        "total_bytes":     total_bytes,
        "per_split":       per_split,
        "generated_at":    datetime.now(timezone.utc).isoformat(),
        "algorithm":       "sha256(sorted 'split/filename:size') — 清单级指纹, 不含逐文件内容哈希",
    }
```

⚠️ **设计点 (清单哈希 vs 内容哈希——一次诚实的取舍)**: 指纹 = sha256(排序后的 `split/文件名:字节数` 清单)。能抓**增删/改名/大小变化** (覆盖 95% 的"数据悄悄变了"); 抓不住**同字节数的原地篡改**——那需要逐文件内容哈希, I/O 与体量线性增长, 归重型可选 (`--deep-fingerprint` 留位)。**取舍本身不丢人, 把取舍写进产出物里才是工程**: `algorithm` 字段把口径原文印在报告上, 审计读到的是"清单级指纹"五个字, 不是一个被高估的承诺。另一个细节: 清单只含**文件名不含绝对路径**——同一份数据换机器换盘符, 指纹不变。这才是"数据指纹", 不是"部署指纹"。

## 13.8 接进 service: 辅助产物的降级承诺

```python

                              out_dir=resolved_run_dir if write_report else None)

```

# 阶段 14: render_markdown — 给企业流程看的报告 + 端到端终态

## 14.1 最后一公里: 日志会滚走, JSON 没人读

四个客户清点一下: CI 有退出码 ✓, 工程师有日志 ✓, 标注团队有 instances.csv ✓——还差**审批流**。质量负责人要的是一份**能转发、能归档、能在评审会上投屏、能签字**的文档; 客户验收要的是一份"看得出专业"的交付物。日志会滚走, report.json 打开是一坨花括号——这两样都进不了审批流。

解药早在阶段 7 就埋好了: **数据/展示分离意味着加一种展示 = 加一个 renderer**。`render_markdown(report) -> str`, 消费同一个 ValidationReport, report.py **零改动**。这是那次分离的第二笔分红 (第一笔是 render_to_logger 和 report.json 互不打扰)。

⚠️ **设计点 (为什么是 Markdown 不是 HTML/PDF)**: Markdown 是"可读纯文本"——git diff 友好 (两版报告能 diff!)、转 HTML/PDF/docx 都是一条 pandoc 命令、贴进 MR 评论原样渲染。**先产中间格式, 终端格式留给使用方**, HTML 美化版归 TODO D4.6。

def _table(headers: List[str], rows: List[List[Any]]) -> str:
    """Markdown 表格。空 rows 返回占位行。"""
    out = ["| " + " | ".join(str(h) for h in headers) + " |",
           "|" + "|".join([" --- "] * len(headers)) + "|"]
    if not rows:
        out.append("| " + " | ".join(["—"] * len(headers)) + " |")
    for row in rows:
        out.append("| " + " | ".join(
            "—" if v is None or v == "" else str(v) for v in row
        ) + " |")
    return "\n".join(out)


def _bar(count: int, max_count: int, width: int = 20) -> str:
    if max_count <= 0:
        return ""
    n = round(count / max_count * width)
    return "█" * max(n, 1 if count > 0 else 0)


```

## 14.4 趋势对比: 上一次的我 vs 这一次的我

"这版数据比上版好没好"是负责人最常问的问题。实现便宜得出奇——**阶段 0 的目录规矩在此第三次付息**: 每次运行一个独立目录, 那"上一次"就躺在同级目录里:

```python
def _load_previous_summary(report: ValidationReport) -> Optional[Dict[str, Any]]:
    """在 run_dir 的同级目录里找 run_id 更早的最近一次 report.json。

    任何异常静默降级 (趋势是锦上添花, 绝不让它弄死报告生成)。
    """
    try:
        if report.run_dir is None:
            return None
        runs_root = report.run_dir.parent
        candidates = []
        for d in runs_root.iterdir():
            if not d.is_dir() or d.name >= report.run_id:
                continue
            rp = d / "report.json"
            if rp.exists():
                candidates.append((d.name, rp))
        if not candidates:
            return None
        _, prev_path = max(candidates)
        return json.loads(prev_path.read_text(encoding="utf-8"))
    except Exception:
        logger.debug("读取历史报告失败, 趋势段降级", exc_info=True)
        return None

⚠️ **设计点 (趋势为什么长在 renderer 不长在数据层)**: ValidationReport 描述**本次**运行——让它知道"上一次"会污染它的边界 (本次的纯数据里混进历史)。趋势是**展示时**的需求, 就在展示层读盘解决; report.json 里永远没有 previous 字段。降级同样优雅: 找不到上次/解析失败 → 这一节显示"首次运行", 绝不影响其余 13 节。

## 14.5 签字栏与合规提示: 报告替流程留位

**签字栏**是三行空表 (验证执行人 / 数据负责人 / 放行批准人 + 日期)——工具不能替人签字, 但**可以让"该签字"这件事无法被遗忘**。打印出来或导出 PDF, 这就是质量记录。

**合规提示**自动触发: 类别表里出现 `person` / `face` / `head` 等人员相关类 → 追加一节固定话术 (人员影像数据注意 PIPL/GDPR 的采集授权与脱敏义务)。⚠️ **宁可多提醒不可漏提醒**——这一节的成本是 200 字, 漏掉一次的成本是法务事故。工具不判定合规与否 (它判不了), 只负责**把该问的问题在显眼处问出来**。

def validate_dataset(
    yaml_path:    Path,
    task_type:    Optional[str] = None,
    run_id:       Optional[str] = None,
    run_dir:      Optional[Path] = None,
    write_report: bool = True,
    options:      Optional[ValidationOptions] = None,
    operator:     Optional[str] = None,
) -> ValidationReport:
    """端到端验证: snapshot → check → 指纹 → 画像 → report → 写盘 (JSON + Markdown)。

    Args:
        yaml_path:    数据集 yaml 文件路径
        task_type:    'detect' / 'segment' / None (None → 读 yaml.task, 再不行 detect)
        run_id:       手动指定运行 ID; None 表示自动用时间戳
        run_dir:      手动指定运行目录; None 表示用 validation_run_dir(run_id)
        write_report: 是否写报告产物 (report.json / report.md / *.csv)
        options:      运行开关 (None = 全默认: 画像开 / 图像深检关)
        operator:     操作人 (None = 系统用户名) — 进报告元数据与签字栏上下文

    容错承诺 (与 run_all_checks 的聚合承诺同源):
        指纹 / 画像 / Markdown 是【辅助产物】, 任何一步抛异常只记日志降级,
        绝不阻断验证主流程 — exit_code 永远只由 check 结果决定。

    Returns:
        ValidationReport (run_dir 已填; .report_path / .markdown_path 可取产物位置)
    """
    # ---- 解析 run_id / run_dir / options ----
    resolved_run_id  = run_id  or datetime.now().strftime("%Y%m%d_%H%M%S")
    resolved_run_dir = run_dir or (validation_run_dir(resolved_run_id) if write_report else None)
    resolved_options = options or ValidationOptions()

    if write_report and resolved_run_dir is not None:
        resolved_run_dir.mkdir(parents=True, exist_ok=True)

    # ---- 跑核心流程 ----
    t0 = time.perf_counter()
    started_iso = datetime.now(timezone.utc).isoformat()

    log_device_info()   # 端到端入口打一次设备信息(复用 D1 system_utils), 让"慢"可归因
    snapshot = build_snapshot(yaml_path=yaml_path, task_type=task_type)
    ctx      = CheckContext(
        yaml_path=yaml_path, snapshot=snapshot, options=resolved_options,
    )
    results  = run_all_checks(ctx)

    # ---- 辅助产物 1: 数据指纹 (轻量, 失败降级) ----
    fingerprint = None
    try:
        fingerprint = compute_fingerprint(snapshot)
    except Exception:
        logger.exception("数据指纹计算失败 — 降级为 None, 不阻断验证")

    # ---- 辅助产物 2: 画像 + 明细 CSV (失败降级) ----
    profile = None
    if resolved_options.profile:
        try:
            profile = run_profile(
                snapshot,
                resolved_options,
                out_dir=resolved_run_dir if write_report else None,
            )
        except Exception as e:
            logger.exception("画像构建失败 — 降级为未生成, 不阻断验证")
            profile = {"generated": False, "error": f"{type(e).__name__}: {e}"}

    duration = time.perf_counter() - t0

    # ---- 包装 ValidationReport ----
    report = ValidationReport(
        run_id=resolved_run_id,
        yaml_path=yaml_path,
        snapshot=snapshot,
        results=results,
        duration_seconds=duration,
        started_at_iso=started_iso,
        run_dir=resolved_run_dir,
        fingerprint=fingerprint,
        profile=profile,
        operator=operator or _default_operator(),
        tool_version=__version__,
        environment=_collect_environment(),
    )

    # ---- 写报告产物 (JSON + Markdown) ----
    if write_report and resolved_run_dir is not None:
        report_path = resolved_run_dir / "report.json"
        report_path.write_text(
            json.dumps(report.to_dict(), indent=2, ensure_ascii=False),
            encoding="utf-8",
        )
        try:
            md_path = resolved_run_dir / "report.md"
            md_path.write_text(render_markdown(report), encoding="utf-8")
            logger.info(f"Markdown 报告已写入: {md_path}")
        except Exception:
            logger.exception("Markdown 报告渲染失败 — JSON 报告不受影响")

```

⚠️ **设计点 (operator 与 environment 是审计字段, 不是装饰)**: `--operator` 未指定时退到系统登录名, 再不行为 None——**尽力而为但不撒谎** (取不到就空着, 不编一个)。`_collect_environment` 记录主机/平台/Python 版本——"在哪台机器、哪个版本上验的"在排查"为什么我这边跑结果不一样"时是第一手线索。

# 阶段 15: 测试哲学 — 测合同, 不测实现

## 15.1 质检工具的测试为什么倒着写

普通功能的测试先测"它能做对"; **质检工具的测试先测"它不冤枉人"**。原因写在企业行为学里: 报告第一次误报 (干净数据被标 WARNING), 用户皱眉; 第二次, 用户开始"先怀疑工具再怀疑数据"; 第三次, 团队给 CI 加 `|| true`——**闸门死于狼来了, 不死于狼**。所以 D4 测试金字塔的底座是一条铁律:

> **干净数据集, 14 项检查, 必须 0 WARNING / 0 ERROR, exit_code == 0。任何一次假阳性都是 P0 bug。**

完整结构三层:

| 层 | 测什么 | 手段 |
|---|---|---|
| ① 零误报底线 | 干净集全绿 + exit 0 + 四件产物齐 | 微缩干净集夹具 |
| ② 脏集逐点 | 12 个埋点问题, **每个被且仅被对应的 check 抓到** | 微缩脏集夹具, 逐 check 断言 |
| ③ 口径单测 | `_overflow_amount` / `_iou_xywh` / `_jsd` 数学正确 | 纯函数直测, **不造一个文件** |

⚠️ **设计点 (③层为什么值得单列)**: 几何/散度算错是最隐蔽的灾难——check 流程全对, 数字悄悄错。把这些算法做成纯函数 (阶段 9/13 的伏笔), 测试就能用手算得出的简单值直接对答案 (两个半重叠框 IoU = 1/3; 相同分布 JSD = 0; 完全分离分布 JSD = 1)。**纯函数是单元测试的天然朋友, 这是把它们抽出来的第二个理由。**

## 15.2 脏集夹具: 一处埋一个雷, 一个雷只响一个 check

夹具即文档——读它等于读"D4 能抓什么"的清单。注意每个埋点旁的注释写明了**期望的债主 check**:

```python
def build_dirty_dataset(root: Path) -> Path:
    """脏数据集 — 每个问题对应一个 check 的断言, 埋点清单见各行注释。"""
    tr_img = root / "images" / "train"
    tr_lbl = root / "labels" / "train"
    va_img = root / "images" / "val"
    va_lbl = root / "labels" / "val"

    # t00: 出图框 (cx+w/2=1.15>1)            → bbox_within_image
    _img(tr_img / "t00.png"); _lbl(tr_lbl / "t00.txt", ["0 0.95 0.50 0.40 0.20"])
    # t01: 零宽框 + 极端宽高比 (0.84/0.02=42) → box_geometry (两种 kind)
    _img(tr_img / "t01.png")
    _lbl(tr_lbl / "t01.txt", ["0 0.50 0.50 0.0005 0.200", "0 0.50 0.50 0.84 0.02"])
    # t02: 完全重复行 + 同类近重复 (IoU≈0.99) → duplicate_annotations
    _img(tr_img / "t02.png")
    _lbl(tr_lbl / "t02.txt", [
        "0 0.30 0.30 0.20 0.20", "0 0.30 0.30 0.20 0.20",          # 完全重复
        "2 0.60 0.60 0.40 0.40", "2 0.602 0.60 0.40 0.40",         # 近重复
    ])
    # t03: 55 个 class0 框 (合法)              → class_balance WARNING (55x) + 密度
    _img(tr_img / "t03.png")
    _lbl(tr_lbl / "t03.txt", [
        f"0 0.{i % 9 + 1}0 0.{i // 9 % 9 + 1}0 0.050 0.050" for i in range(55)
    ])
    # t04: 无标注图 (空 txt)                   → annotation_coverage
    _img(tr_img / "t04.png"); _lbl(tr_lbl / "t04.txt", [])
    # t05: 缺标签文件                          → pair_existence
    _img(tr_img / "t05.png")
    # t06: 正常图 (class1 出现, 维持基线)
    _img(tr_img / "t06.png"); _lbl(tr_lbl / "t06.txt", ["1 0.50 0.50 0.30 0.30"])
    # dup.png + dup.jpg 同 stem                → stem_collision
    _img(tr_img / "dup.png"); _img(tr_img / "dup.jpg")
    _lbl(tr_lbl / "dup.txt", ["0 0.50 0.50 0.20 0.20"])
    # ghost1/ghost2: 孤儿标签 (无对应图)        → orphan_labels
    _lbl(tr_lbl / "ghost1.txt", ["0 0.50 0.50 0.10 0.10"])
    _lbl(tr_lbl / "ghost2.txt", ["0 0.50 0.50 0.10 0.10"])

    # val: class3 出现而 train 为 0            → class_presence WARNING
    _img(va_img / "v00.png"); _lbl(va_lbl / "v00.txt", ["3 0.50 0.50 0.30 0.30"])
    _img(va_img / "v01.png"); _lbl(va_lbl / "v01.txt", ["0 0.40 0.40 0.20 0.20"])
    # v02: 损坏图 (junk bytes)                 → image_integrity (--check-images)
    bad = va_img / "v02.jpg"
    bad.parent.mkdir(parents=True, exist_ok=True)
    bad.write_bytes(b"this is not an image at all")
    _lbl(va_lbl / "v02.txt", ["0 0.50 0.50 0.20 0.20"])

    # yaml 声明 test 但目录不存在               → split_presence (scan_warning)
    return _write_yaml(root, declare_test=True)

```

⚠️ **设计点 (埋点要避开"一雷多响")**: t01 的两行故意都用 class0——如果第二行用了 class1, class1 全集仅 1 实例, 会顺带触发 class_balance 的 50x 档, 脏集断言就开始互相纠缠。**夹具设计的功夫在隔离**: 每个断言失败时, 你要能立刻说出是哪个 check 坏了, 而不是先解三元方程。

## 15.3 合同测试与灵魂测试

**合同测试**守护对外承诺——附两条原文:

```python
    def test_image_integrity_disabled_by_default(self, tmp_path):
        yaml_path = build_clean_dataset(tmp_path / "ds")
        report = validate_dataset(yaml_path=yaml_path, write_report=False)
        r = _result(report, "image_integrity")
        assert r.severity == "INFO"
        assert r.details["reason"] == "disabled"


    def test_fingerprint_stable_and_sensitive(self, report, tmp_path):
        snap = build_snapshot(yaml_path=report.yaml_path)
        fp1 = compute_fingerprint(snap)
        fp2 = compute_fingerprint(build_snapshot(yaml_path=report.yaml_path))
        assert fp1["manifest_sha256"] == fp2["manifest_sha256"], "同数据指纹必须稳定"
        # 加一个文件 → 指纹必须变
        _img(report.yaml_path.parent / "images" / "train" / "extra.png")
        fp3 = compute_fingerprint(build_snapshot(yaml_path=report.yaml_path))
        assert fp3["manifest_sha256"] != fp1["manifest_sha256"], "数据变更指纹必须变"


# ============================================================
# 脏数据集: 每个埋点对应一个 check
# ============================================================

class TestDirtyDataset:
    @pytest.fixture()
    def report(self, tmp_path):
        yaml_path = build_dirty_dataset(tmp_path / "ds")
        return validate_dataset(
            yaml_path=yaml_path,
            run_dir=tmp_path / "run",
            options=ValidationOptions(check_images=True),
        )
```

第一条测**默认即承诺**: 不传 options, image_integrity 必须报"未开启"的 INFO 而不是偷偷跑 (默认值变更 = 行为变更 = 要走评审)。第二条是**指纹的双向灵魂测试**——只测"两次运行指纹相同"是不够的: 一个永远返回 `"abc"` 的假实现也能通过。**稳定性 (同数据同值) 和敏感性 (动一字节变值) 必须同时断言**, 缺一边, 测试就在给假实现发通行证。

## 金句集

> "质检子系统的客户从来不止算法工程师一个——CI 要退出码, 审批人要结论, 标注团队要整改清单, 审计要指纹。"

> "一个子系统的工程素养, 在它的第一次 commit 就能看出来。"

> "性能优化最值钱的一种, 是把'扫 14 遍'变成'扫 1 遍'——速度和一致性一起到手, 而后者在审计面前更贵。"

> "逐字段合法 ≠ 整体合法。盲区永远藏在字段之间的组合关系里。"

> "单向的成对检查只查了一半——企业数据事故, 一半藏在没人查的那个方向上。"

> "沉默的跳过, 是在替未来的读者做'这不重要'的决定——质检工具没有这个权力。"

> "把主观词换成可吵架的数字, 是企业报告成熟的标志。"

> "告警疲劳是质量闸门的头号死因, 分级的全部意义就是防它。闸门死于狼来了, 不死于狼。"

> "取舍本身不丢人, 把取舍写进产出物里才是工程。"

> "报告不是数据的倾倒场, 是问题的回答集——说不出'谁靠这一节回答什么问题'的节, 删。"

## 练习

**练习 1 (必做)**: 跟着文档从阶段 0 抄到阶段 16, 每阶段跑通再前进。预算 10-14 小时。

**练习 2 (动手)**: 实现 D4.1 的最小版——给 `ValidationOptions` 加 `threshold_overrides: Mapping[str, float]` 字段, 让 `class_balance` 优先读覆盖值。要求: ① frozen 约束不破; ② 不传时行为与现在逐字节一致 (合同测试证明); ③ 报告里如实打印生效阈值与来源。写完回头看: 哪些设计让这 30 行改动如此便宜?

**练习 3 (灵魂测试)**: 阶段 13 立了"check 与 profiler 口径同源"的合同, 但现在没有测试守着它。如果有人把 profiler 里的退化判定改成硬编码 `w < 0.002` (而 check 仍用 constants 的 1e-3)——**当前 27 条测试能抓到吗?** 设计一条"口径漂移哨兵"测试: 构造一个 `w` 恰好落在 1e-3 与 0.002 之间的实例, 断言 box_geometry 的计数与 instances.csv 的 flag 数**相等**。

**写 200 字的设计权衡**: 这条测试焊死的是"两边数字相等"还是"两边都引用同一常量"? 前者允许两边各自实现只要结果一致, 后者更强但测的是实现细节。哨兵该立在哪一层? 这是一门艺术, 不是科学。

# 给你的最后一段话

D4 的难, 不在写 14 个 check——后 10 个你越写越快, 最后一个 20 分钟就能出活, 那正是注册表模式在替你打工。D4 的难在三次**身份转换**:

- 阶段 1→2, 从"写检查的人"变成"**立合同的人**"——你开始为 14 个还不存在的 check 作者做决定
- 阶段 8→9, 从"实现需求的人"变成"**找盲区的人**"——全绿的报告下面, 你得知道还有什么没被看见
- 阶段 13→14, 从"给工程师写工具的人"变成"**给组织写工具的人**"——退出码、评级、签字栏、合规提示, 每一样都在伺候一个不写代码的角色

学完 D3 + D4, 同一个注册表 idiom 你已经用它解了两道题——**互斥分发**和**聚合执行**; 同一条"数据/展示分离", 你已经收了两次分红。D5 训练子系统开工时, 工具箱里躺着 24 个模式, **80% 的设计是从里面挑一个, 不是发明一个**。

工程师和程序员的差别——**程序员让代码跑起来, 工程师让代码可以解释自己为什么是这样**。D4 教完, 你写的不只是一个 validator, 而是一个**每个决策都能在评审会上自我辩护**的 validator: 为什么 14 项、为什么这个默认关、为什么报告长这样、为什么指纹只到清单级——每一问, 这份文档里都有一段"因为企业里……"。这两者的差别, 三年后会变成你工资条上的数字。
