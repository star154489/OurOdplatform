## 1.1 一种格式,一个纯函数(直接就能跑)

需求很具体:把 VOC 格式的 XML 转成 YOLO 格式的 txt。先把两种格式说清楚——它们只差在"框怎么记":

- **VOC**:每张图一个 `.xml`,框记成左上、右下四个**绝对像素**坐标 `(xmin, ymin, xmax, ymax)`。
- **YOLO**(训练要用):每张图一个 `.txt`,每行 `cls_id cx cy w h`,坐标**全部归一化到 [0,1]**(即除以图宽 / 图高)。

所以"VOC→YOLO"本质就是:逐个读 xml、把像素坐标换算成归一化坐标、写 txt。最朴素的写法——**一个函数,直接调用**。

> 📄 **`apps/platform/src/od_platform/data_pipeline/convert.py`** · 约 40 行
> **内容**:一个函数 `convert_voc_to_yolo(annotations_dir, out_labels_dir)`——读所有 `.xml`,换算成 YOLO 归一化坐标写 txt,返回"出现过的类名列表"。
> **为什么这样设计**:此刻**故意不引入任何架构**——只有一种格式,一个普通函数、用普通参数、直接调用就够。类名按"首次出现"累积、下标即 `class_id`,保证不同机器上编号一致。
> **现在为什么不做别的**:学员常问"要不要先写个基类 / 工厂?"——不要。只有一个实现时,任何分发结构都是**没东西可分发的空壳**;等第二种格式出现、调用方被迫 `if/elif`(1.2),抽象才有对象。**抽象是被第二个实现逼出来的,不是预设的。**

```python
"""data_pipeline/convert.py —— 第一版:就一个普通函数,把 VOC 转成 YOLO。

VOC 标注:每张图一个 .xml,框是绝对像素坐标 (xmin, ymin, xmax, ymax)。
YOLO 标注:每张图一个 .txt,每行 `cls_id cx cy w h`,坐标全部归一化到 [0,1]。
现在没有任何架构 —— 只有一种格式,一个函数,直接调用就够了。
"""
import xml.etree.ElementTree as ET
from pathlib import Path


def convert_voc_to_yolo(annotations_dir: Path, out_labels_dir: Path) -> list[str]:
    out_labels_dir.mkdir(parents=True, exist_ok=True)
    classes: list[str] = []                       # 类名按"首次出现"累积,下标即 class_id

    for xml_path in sorted(annotations_dir.glob("*.xml")):   # sorted:顺序确定,编号才稳定
        root = ET.parse(xml_path).getroot()
        size = root.find("size")
        W = float(size.findtext("width"))         # 这张图的宽,用来把 x 归一化
        H = float(size.findtext("height"))        # 这张图的高,用来把 y 归一化

        lines: list[str] = []
        for obj in root.findall("object"):        # 一张图可能有多个框
            name = obj.findtext("name")
            if name not in classes:
                classes.append(name)              # 第一次见到这个类,登记进列表
            cls_id = classes.index(name)          # 类名 → 数字编号(列表下标)

            b = obj.find("bndbox")
            xmin = float(b.findtext("xmin")); ymin = float(b.findtext("ymin"))
            xmax = float(b.findtext("xmax")); ymax = float(b.findtext("ymax"))
            cx = (xmin + xmax) / 2 / W             # 像素坐标 → 归一化中心点 + 宽高
            cy = (ymin + ymax) / 2 / H
            bw = (xmax - xmin) / W
            bh = (ymax - ymin) / H
            lines.append(f"{cls_id} {cx:.6f} {cy:.6f} {bw:.6f} {bh:.6f}")

        out = out_labels_dir / (xml_path.stem + ".txt")
        out.write_text("\n".join(lines), encoding="utf-8")   # 没框也写空 txt,保证图-标注配对

    return classes                                # 顺便返回"这批数据有哪些类"
```

> 📌 三件**学员一定会问**的小事,都已写进注释,老师可以主动点一句:
> - `sorted(...)`:让文件处理顺序确定,这样 `class_id` 在不同机器/不同次运行上一致(否则今天 `head=0`、明天 `head=1`,训练标签全乱)。
> - 类名"首次出现即登记":所以 `class_id` 的顺序由扫描顺序决定,不是凭空指定的。
> - **没有框也写一个空 `.txt`**:下游靠"每张图都有一个同名标注文件"来配对;缺了它,纯背景图会被当成"没标注"而漏掉。

**这里没有 `import registry`、没有 `@register`——那些东西现在根本不存在,也用不上。**

不需要真实像素:转换器只从 XML 读 width/height 和框坐标,图像文件只用来做"图-标注配对"
检查,所以建空 .jpg 即可。故意让稀有类 `反光衣` 只占 12/62,为阶段 4 的"撞墙②"埋伏笔。

from pathlib import Path
import random

ROOT = Path(__file__).parent / "data" / "raw" / "demo"
IMG = ROOT / "images"; ANN = ROOT / "annotations"


def xml_for(stem, w, h, objects):
    objs = ""
    for name, (xmin, ymin, xmax, ymax) in objects:
        objs += (f"  <object>\n    <name>{name}</name>\n"
                 f"    <bndbox><xmin>{xmin}</xmin><ymin>{ymin}</ymin>"
                 f"<xmax>{xmax}</xmax><ymax>{ymax}</ymax></bndbox>\n  </object>\n")
    return (f"<annotation>\n  <filename>{stem}.jpg</filename>\n"
            f"  <size><width>{w}</width><height>{h}</height><depth>3</depth></size>\n{objs}</annotation>\n")


def main():
    random.seed(0)
    IMG.mkdir(parents=True, exist_ok=True); ANN.mkdir(parents=True, exist_ok=True)
    for i in range(50):                                   # 50 张普通图,只含 head
        stem = f"common_{i:03d}"; objs = []
        for _ in range(random.randint(1, 3)):
            x = random.randint(0, 400); y = random.randint(0, 400)
            objs.append(("head", (x, y, x + 80, y + 80)))

            encoding="utf-8")
        (IMG / f"{stem}.jpg").write_bytes(b"")
    print(f"生成完毕: {len(list(ANN.glob('*.xml')))} 个 xml, {len(list(IMG.glob('*.jpg')))} 张图 -> {ROOT}")


if __name__ == "__main__":
    main()

## 1.2 加第二种格式 COCO:`if/elif` 自然冒出来

老板第二天加需求:"也要支持 COCO。" COCO 的标注是整个数据集**一个** `instances.json`,框记成 `[x, y, w, h]`(左上角 + 宽高,绝对像素)。**读法和 VOC 完全不同,但读完之后归一化、写 txt 的逻辑一模一样。** 我们照样写一个 `convert_coco_to_yolo(...)`,再加一个**按字符串选函数**的 `run()`——都**追加到 `convert.py`**:

> 📄 **`apps/platform/src/od_platform/data_pipeline/convert.py`(追加部分)** · 约 40 行
> **内容**:`convert_coco_to_yolo(annotations_dir, out_labels_dir)`(读那个 json,换算写 txt)、`run(fmt, ...)`(按字符串 `fmt` 用 `if/elif` 决定调哪个转换器)。
> **为什么这样设计**:两个转换器签名一致 `(annotations_dir, out_labels_dir) -> list[str]`,所以 `run()` 能用同样的方式调它们;`run()` 就是那个"按名字选函数"的中心。
> **现在为什么不做别的**:`run()` 里这段 `if/elif` 看着无害——本节的全部意义,就是让你**亲眼看到它马上会出问题**,从而理解为什么需要后面那一套。**先撞,再修。**

```python
# ── 阶段 1.2 追加到 convert.py 顶部的 import ──────────────
import json
from collections import defaultdict


def convert_coco_to_yolo(annotations_dir: Path, out_labels_dir: Path) -> list[str]:
    """COCO -> YOLO。整个数据集一个 .json,框是 [x,y,w,h](左上角+宽高,绝对像素)。"""
    out_labels_dir.mkdir(parents=True, exist_ok=True)
    json_path = sorted(annotations_dir.glob("*.json"))[0]    # COCO 习惯单个 json
    data = json.loads(json_path.read_text(encoding="utf-8"))

    cat_name = {c["id"]: c["name"] for c in data["categories"]}   # 类别 id → 名字
    img_info = {im["id"]: im for im in data["images"]}            # 图 id → 图信息
    per_image: dict[int, list] = defaultdict(list)               # 图 id → 它的所有标注
    for ann in data["annotations"]:
        per_image[ann["image_id"]].append(ann)

    classes: list[str] = []
    for img_id, im in img_info.items():
        W, H = float(im["width"]), float(im["height"])
        stem = im["file_name"].rsplit(".", 1)[0]   # 去扩展名,作为 txt 文件名
        lines: list[str] = []
        for ann in per_image.get(img_id, []):
            name = cat_name[ann["category_id"]]
            if name not in classes:
                classes.append(name)
            cls_id = classes.index(name)
            x, y, w, h = ann["bbox"]                # COCO:左上角 (x,y) + 宽高 (w,h)
            cx = (x + w / 2) / W
            cy = (y + h / 2) / H
            lines.append(f"{cls_id} {cx:.6f} {cy:.6f} {w / W:.6f} {h / H:.6f}")
        (out_labels_dir / (stem + ".txt")).write_text("\n".join(lines), encoding="utf-8")
    return classes


def run(fmt: str, annotations_dir: Path, out_labels_dir: Path) -> list[str]:
    """根据 fmt 字符串,决定调哪个转换器。最直接的办法:if/elif。"""
    if fmt == "pascal_voc":
        return convert_voc_to_yolo(annotations_dir, out_labels_dir)
    elif fmt == "coco":
        return convert_coco_to_yolo(annotations_dir, out_labels_dir)
    else:
        raise ValueError(f"不支持的格式: {fmt}")
```

## 1.3 🧱 撞墙①:同一段判断,悄悄散进了好几个文件

`run()` 里那段 `if/elif` 看着还行。但问一句:**做"按格式选函数"这件事的地方,真的只有这一处吗?**

不止。去代码里翻一圈,想想都有谁需要它:

- 命令行入口 `cli.py`:接到 `--format`,要判断该调哪个 converter。
- 流水线 `pipeline.py`:批量处理时,也得按格式决定走哪条路。
- 测试夹具 `test_fixture.py`:造假数据来测试时,要按格式生成对应的样子。

同一个"我支持哪些格式"的事实,被**抄进了好几个文件**:

```
cli.py            pipeline.py        test_fixture.py
if voc:  …        if voc:  …         if voc:  …
elif coco: …      elif coco: …       elif coco: …
elif 新: ←要加     elif 新: ←也要加     忘了加 ← 静默出 bug
```

设想你某天加了个 `labelme` 格式,改了 `cli.py` 和 `pipeline.py`,却忘了改 `test_fixture.py`——测试照样用旧逻辑跑、还全绿,你以为没事,结果某条路径根本不认 `labelme`,直到上线被用户踩中才发现。**改一处、漏一处,那条没改到的路径就悄悄"不支持"了,还不报错。** 这种毛病有个俗称叫**漂移(drift)**。

> 💡 **金句**:**"`if/elif` 的本质问题不是'丑',是'分支信息会跨文件复制'。同一份真相散在 N 个文件里,改一处忘一处,迟早打架。"**

看到这儿,第一反应很自然:把这堆重复的判断收到一个地方去。**这正是阶段 2 的起点。**

```bash
git add make_demo.py make_coco_demo.py apps/platform/src/od_platform/data_pipeline/convert.py
git commit -m "feat(data_pipeline): naive VOC/COCO converters + if/elif dispatch (hits wall #1)"
```

# 阶段 2:撞墙后重构 —— 注册表与自动注册

> 这一节不背定义,看着注册表**怎么从那段 `if/elif` 一步步长出来**:先收成中间层(治了漂移、还别扭)→ 换成一张表 → 让实现自己报到 → 撞上空表 → 开门自动叫醒。等问题攒够,注册表会自己浮现——到那时,它的每一行你都知道为什么。

## 2.1 第一次抢救:中间层(治了漂移,却还是别扭)

最直接的修法:写一个**中间函数**,把所有 `if/elif` 收进去;其它文件(`cli` / `pipeline` / 测试)都不再各自判断,统一来调它。我们 1.2 写的那个 `run()`,其实就是这个中间层的雏形。

漂移**确实治好了**——全项目只剩这一处 `if/elif`,加格式只改这一个地方,几个调用方不会再各自漏改。这是实打实的进步。

可你心里还是有点别扭。哪里别扭?**每加一种格式,你还是得回来,在这段旧代码里再补一支 `elif`。** 你只是把"改好几处"压成了"改一处"——但"加东西,就得回去动已经写好的代码"这件事本身,一点没变;而且这个函数会随格式数一直变长。

```python
# 中间层 run():漂移没了,但它自己随格式数越长越长,且每加一种都得回来改它
def run(fmt, ...):
    if fmt == "pascal_voc": ...
    elif fmt == "coco": ...
    elif fmt == "yolo": ...        # ← 每加一种
    elif fmt == "labelme": ...     # ← 这里就再长一行,还得回来动这个旧文件
    else: raise ValueError(...)
```

> 🧭 **把这个别扭说成一句话**:有没有可能,让"加一种格式"**根本不用碰任何已经写好的代码**?——加的时候只**新增**东西(比如新建一个文件),旧代码一行都不动。这个朴素愿望有个正经名字叫**开闭原则**("对扩展开放,对修改封闭"),但名字先放着,**记住这个愿望就行**。

带着这个愿望,我们换个方向想。

## 2.2 换个方向:让每种格式"自己报到"

中间层为什么非写 `if/elif` 不可?因为是**它**在操心"哪个名字对应哪个函数"——所有对应关系都压在它一个人身上,每多一种它就得多记一条。这件事能不能反过来?

打个生活里的比方,一场活动要"按名字找到人":

- **办法 A(像中间层)**:主持人自己拿个小本子,把每个人的名字和座位都记下来。来一个新人,他就得翻开本子添一行。人越多,本子越厚,全压在他一个人身上。
- **办法 B(注册表)**:门口摆一本**签到册**,每个人进来**自己**签上"我叫某某,坐第几排"。主持人要找谁,翻签到册查一下就行——他什么都不用记,来多少新人都不关他的事。

**注册表就是办法 B**:让每个 converter 自己声明"我能处理 `coco`",中间那一层不再判断、只查一张表。这张"名字 → 函数"的表,就是**注册表(Registry)**。下面我们一步步把它搭出来——而且搭的过程本身,也是一步步撞出来的。

## 2.3 第一步:先用一张表,把 `if/elif` 换掉

"按名字找函数",本质就是查字典(Python 的 `dict` 就是"键 → 值"的表)。那我们就别写 `if/elif` 了,真用一个字典存"名字 → 函数":

```python
_REGISTRY = {}                                   # 名字 → 函数 的表
_REGISTRY["pascal_voc"] = convert_voc_to_yolo    # 注意:右边没有括号!
_REGISTRY["coco"] = convert_coco_to_yolo
# 用的时候再查表 + 加括号调用:
#     _REGISTRY[fmt](annotations_dir, out_labels_dir)
```

> 💡 **关键:函数本身也能当"值"存进字典。** `_REGISTRY["coco"] = convert_coco_to_yolo` 右边**没有括号**——存进去的是"这个函数"本身,不是"调用它的结果"。要用时再 `_REGISTRY[fmt](...)` 加括号调用。**把函数当普通值传来传去,是后面一切的基础。**

调用方的 `if/elif` 没了,这是进步。但你再看一眼——往字典里塞这几行,**还是集中写在一个地方**;加一种格式,还是得回这里加一行。我们真正想要的,是让每种格式**在自己的文件里**把自己写进表(对应签到册:让每个人自己签,而不是主持人替大家签)。

顺带说一件马上要做的事:要让一张表**一视同仁**地调所有 converter,它们的**签名必须统一**。阶段 1 的转换器是 `(annotations_dir, out_labels_dir)`,以后还要加白名单、任务类型等参数。与其每加一个参数就改所有 converter 的签名,不如把这些参数打成**一个参数包** `ConvertOptions` 传进去——以后加字段不破坏任何已有签名。

## 2.4 第二步:`@register` 让实现自己登记

怎么让 `coco.py` **自己**把自己写进表?用一个小装饰器 `@register`。先别被"装饰器"三个字唬住——它在这里做的事,说穿了就一句话:**把这个函数,按这个名字,写进表。**

最该说清的一点:`@register("coco")` 和你手写 `_REGISTRY["coco"] = convert_coco` **完全是一回事**,只是把"登记"这行挪到函数头顶,顺手而已:

```python
# 你写的(顺手版)              # 完全等价于(摊开版)
@register("coco")              def convert_coco(...):
def convert_coco(...):             ...
    ...                        _REGISTRY["coco"] = convert_coco
```

> 💡 **装饰器不是魔法**,就是"把登记这行写到函数头顶"的语法糖。而且这行登记,是在**"import 到 `coco.py`"的那一刻**才执行的——**记住这句,下一步要用。**

> 📌 **顺带一提"能力声明"**:我们这版的表里,每一条记录除了"函数",还会多存一句"我支持哪些 task(detect / segment)"。这样调度层在分发**之前**就能判断"这个格式不支持你要的 task",当场报清楚,而不是进到函数里才炸一个难懂的异常。这一层是注册表比裸 `if/elif` 多出来的价值,后面 `service` 会用到。

于是每个格式,在自己的文件里 `@register` 一下,就登记好了。看起来大功告成:加新格式 = 新建一个文件、写个 `@register("新名字")`,框架一行不用改。可你运行一下试试……

## 2.5 🧱 一个意外:表居然是空的

你写好了 `coco.py`、`pascal_voc.py`,信心满满地**只 import 那张表所在的文件**,查一下:

```python
# ckpt2b_empty.py —— 复现"空表"撞墙
from od_platform.data_pipeline.convert import registry as R
print("只 import registry 后,_REGISTRY =", dict(R._REGISTRY))     # 期望:空
import od_platform.data_pipeline.convert.converters.coco           # 手动 import 一个实现
print("import coco.py 后,_REGISTRY 的键 =", sorted(R._REGISTRY))   # 期望:出现 'coco'
```

实际运行(已验证):

```text
只 import registry 后,_REGISTRY = {}
import coco.py 后,_REGISTRY 的键 = ['coco']
```

**表是空的!** 为什么?回到上一步那句话——登记是在"import 到那个文件"的一刻才执行的。而关键事实是:

> ⚠️ **Python 不会执行你没 import 的文件。** 你只 import 了 `registry`(那张空表所在的文件),从没 import 过 `coco.py`,它顶上那行 `@register` 就**从没运行过**,自然没人往表里写。**代码写在那儿 ≠ 它被执行了**——没被 import,它就只是躺在硬盘上的文字。

上面第二行 `import ...coco` 之后键里立刻冒出 `'coco'`,正是这条规律的现场证据:**import 一个实现文件,就触发它顶上的 `@register`。** 问题一下清楚了:得有人在"用表"之前,把这些实现文件都 import 一遍。那……谁来做?

## 2.6 收尾:开门时,自动把它们都"叫醒"

最笨的办法:手写 `import coco, pascal_voc` 把它们一个个 import 进来。能用,但你发现没——那样"加一个文件,还得回来补一行 import",又绕回了 2.1 那个"加东西要改旧代码"的老毛病。**我们不想退回去。**

更好的办法:让程序自己来。**开门(第一次要用表)时,把存放实现的那个文件夹挨个扫一遍、逐个 import。** 这件事就是"自动发现"。下面把整张 `registry.py` 给全——它把前面几步(表 + `@register` + 自动发现)合到了一个文件:

> 📄 **`apps/platform/src/od_platform/data_pipeline/convert/registry.py`** · 约 60 行
> **内容**:参数包 `ConvertOptions`、注册表条目 `ConverterEntry`(函数 + 能力声明)、装饰器 `register(...)`、查询 `get_converter` / `available_formats`,以及自动发现 `_lazy_init`(首次用表时扫 `converters/`)。
> **为什么这样设计**:① 参数包让所有 converter 签名统一成 `(input_dir, out_labels_dir, options)`;② `register` 是带参装饰器,登记后把函数**原样返回**;③ 自动发现用 `pkgutil` 扫目录,**加文件就被收录,连 import 列表都不用手写**——这一句是"加格式 = 加文件、框架一行不改"能**字面成立**的关键。
> **现在为什么不做别的**:为什么不在 `service.py` 启动时手写 `import` 列表?那就退回 2.1 的老毛病了(加文件还得改旧代码)。为什么扫描逻辑此刻写在这儿、不抽成公共件?因为**只有一个用例**——等阶段 4 的 split 第二张表也要同样的扫描,我们才抽进 `common/registry_utils.py`(第二次才抽象)。

```python
"""convert 框架层:一张表(格式名→条目) + 登记装饰器 @register + 参数包 ConvertOptions。

框架(本文件 + service.py)永远不动;加新格式 = 在 converters/ 加一个文件。
注:本阶段把"扫描 converters/ 目录"的逻辑【内联】在 _lazy_init 里;等阶段 4 的 split
   第二张表也要同样的扫描时,我们才把它抽进 common/registry_utils.py(第二次才抽象)。
"""
from __future__ import annotations

import logging
from dataclasses import dataclass, field
from pathlib import Path
from typing import Callable, Dict, List, Optional, Tuple

logger = logging.getLogger(__name__)


# ── 参数包:所有 converter 签名统一成 (input_dir, out_labels_dir, options) ──
@dataclass
class ConvertOptions:
    """所有格式转换共用的参数包。

    task   : 目标任务(detect / segment)。默认 detect 覆盖多数场景。
    classes: 类别白名单。
             None  = 用户没指定,由 converter 自行探测(仅 voc/coco 支持探测)。
             [...] = 用户指定的类别(同时决定 class_id 顺序)。
             ⚠️ 不要用 [] 当默认:None="没提供",[]="确实没有任何类别",语义不同。
    """
    task: str = "detect"
    classes: Optional[List[str]] = field(default=None)


# 一个 converter = 吃 (input_dir, out_labels_dir, options)、返回类名列表(顺序即 class_id)的函数。
ConverterFunc = Callable[[Path, Path, ConvertOptions], List[str]]


@dataclass(frozen=True)
class ConverterEntry:
    """注册表里一条记录:实现函数 + 它"能干哪些 task"的能力声明。"""
    func: ConverterFunc
    supported_tasks: Tuple[str, ...]

    def supports(self, task: str) -> bool:
        return task in self.supported_tasks


# 注册表本体:就是一个 {格式名: 条目} 的字典(模块级单例)
_REGISTRY: Dict[str, ConverterEntry] = {}


def register(format_name: str, *, supported_tasks: Tuple[str, ...]):
    """装饰器:把被装饰的函数(连同它支持的 task)登记进 _REGISTRY。

        @register("coco", supported_tasks=("detect", "segment"))
        def convert_coco(...): ...

    完全等价于在文件底部手写:
        _REGISTRY["coco"] = ConverterEntry(convert_coco, ("detect", "segment"))
    它在【该文件被 import 的那一刻】执行。
    """
    def decorator(func: ConverterFunc) -> ConverterFunc:
        if format_name in _REGISTRY:
            logger.warning("格式 %s 被重复注册,后者覆盖前者", format_name)
        _REGISTRY[format_name] = ConverterEntry(func=func, supported_tasks=tuple(supported_tasks))
        return func                       # 函数原样还回去,本身不变
    return decorator


def get_converter(format_name: str) -> ConverterEntry:
    """按 format 名取出条目。Raises: ValueError 未注册的格式。"""
    _lazy_init()
    if format_name not in _REGISTRY:
        raise ValueError(f"未注册的格式: {format_name!r}。已注册: {sorted(_REGISTRY)}")
    return _REGISTRY[format_name]


def available_formats() -> List[str]:
    """当前已注册的格式名(会先触发自动发现)。"""
    _lazy_init()
    return sorted(_REGISTRY)


# ── 自动发现:首次用表时,扫描 converters/ 把每个实现 import 一遍,触发它们的 @register ──
_LAZY_INITIALIZED = False


def _lazy_init() -> None:
    """扫描 converters/*.py 自动触发 @register。
    · 跳过 _ 开头的私有模块。
    · 标志位放在 import 全部成功【之后】——任何 import 失败都不污染状态,下次可重试。
    """
    global _LAZY_INITIALIZED
    if _LAZY_INITIALIZED:
        return
    import importlib, pkgutil
    from od_platform.data_pipeline.convert import converters
    for m in pkgutil.iter_modules(converters.__path__):
        if not m.name.startswith("_"):
            importlib.import_module(f"{converters.__name__}.{m.name}")
    _LAZY_INITIALIZED = True
```

再写薄薄的调度层。调用方只跟它说话,永远不写 `if/elif`:

> 📄 **`apps/platform/src/od_platform/data_pipeline/convert/service.py`** · 约 16 行
> **内容**:`convert_data_to_yolo(input_dir, output_labels_dir, annotation_format, options)`——查表 + 校验 task + 分发。
> **为什么这样设计**:把"按名字分发"收口到一个入口;它**永不增长**——加多少格式都不改。
> **现在为什么不做别的**:为什么这里要 `entry.supports(task)` 校验?因为这是 `ConverterEntry` 那句"能力声明"的兑现点——格式不支持该 task,在这儿当场报清楚,而不是进到 converter 里炸难懂的异常。

```python
"""convert 调度层 —— 几行代码接住所有格式。

它是项目其它部分(指挥家 / CLI / 测试)唯一该调用的"转换入口"。
关键性质:【永不增长】—— 无论将来支持多少种格式,本文件一行都不用改,因为它只查表分发。
"""
from __future__ import annotations

from pathlib import Path
from typing import List

from od_platform.data_pipeline.convert.registry import ConvertOptions, get_converter


def convert_data_to_yolo(
    input_dir: Path,
    output_labels_dir: Path,
    annotation_format: str,
    options: ConvertOptions,
) -> List[str]:
    """统一入口:按 annotation_format 分发到具体 converter,返回类名列表。

    Raises:
        ValueError: 格式未注册,或该 converter 不支持请求的 task(在此提前 fail-fast)。
    """
    entry = get_converter(annotation_format)
    # 能力声明的兑现点:格式不支持该 task,在这里当场报清楚,而不是进到 converter 里炸。
    if not entry.supports(options.task):
        raise ValueError(
            f"格式 {annotation_format!r} 不支持 task={options.task!r}。支持: {entry.supported_tasks}"
        )
    return entry.func(input_dir, output_labels_dir, options)
```

> 📄 **`apps/platform/src/od_platform/data_pipeline/convert/converters/pascal_voc.py`** · 约 30 行
> **内容**:`@register("pascal_voc", supported_tasks=("detect",))` + `convert_voc(input_dir, out_labels_dir, options)`——和阶段 1 同样的 VOC→YOLO 逻辑,加白名单。
> **现在为什么不做别的**:为什么白名单(删类)能走 VOC?因为 VOC 的 xml 里**有类名**,能按名字筛掉不要的类——这一点 YOLO 做不到(阶段 3 会专门讲这个差异)。

```python
"""Pascal VOC -> YOLO 转换器(detect)。

VOC:每张图一个 .xml,框是绝对像素坐标 (xmin,ymin,xmax,ymax)。
YOLO:每张图一个 .txt,每行 `cls_id cx cy w h`,坐标全部归一化到 [0,1]。
本转换器的核心就是"逐个读框 → 像素换算成归一化 → 写 txt",顺带收集类名表。

它如何接入框架:头顶一行 @register 即把自己签进格式注册表,框架代码无需任何改动。
"""
from __future__ import annotations

import logging
import xml.etree.ElementTree as ET
from pathlib import Path
from typing import List

from od_platform.common.constants import AnnotationFormat, Task
from od_platform.data_pipeline.convert.registry import ConvertOptions, register

logger = logging.getLogger(__name__)


@register(AnnotationFormat.PASCAL_VOC, supported_tasks=(Task.DETECT,))
def convert_voc(input_dir: Path, out_labels_dir: Path, options: ConvertOptions) -> List[str]:
    out_labels_dir.mkdir(parents=True, exist_ok=True)

    # classes 既是"输出的类名表",也是"白名单":
    #   options.classes 为 None → discovering=True,边读边把新类名追加进去(探测模式);
    #   options.classes 给了名单 → discovering=False,不在名单的框直接跳过(=删类)。
    classes: List[str] = list(options.classes) if options.classes else []
    discovering = options.classes is None

    for xml_path in sorted(input_dir.glob("*.xml")):   # sorted:处理顺序确定 → class_id 跨机一致
        root = ET.parse(xml_path).getroot()
        size = root.find("size")
        W = float(size.findtext("width"))               # 用图宽把 x 归一化
        H = float(size.findtext("height"))              # 用图高把 y 归一化

        lines: List[str] = []
        for obj in root.findall("object"):              # 一张图可能有多个框
            name = obj.findtext("name")
            if name not in classes:
                if discovering:
                    classes.append(name)                # 探测模式:首次见到就登记
                else:
                    continue                            # 白名单模式:名单外的框跳过(删类)
            cls_id = classes.index(name)                # 类名 → 数字编号(列表下标)

            b = obj.find("bndbox")
            xmin = float(b.findtext("xmin")); ymin = float(b.findtext("ymin"))
            xmax = float(b.findtext("xmax")); ymax = float(b.findtext("ymax"))
            # 像素的左上/右下 → YOLO 的归一化中心点 + 宽高
            cx = (xmin + xmax) / 2 / W
            cy = (ymin + ymax) / 2 / H
            lines.append(f"{cls_id} {cx:.6f} {cy:.6f} {(xmax-xmin)/W:.6f} {(ymax-ymin)/H:.6f}")

        # 没框也写一个空 .txt:下游靠"每张图都有同名标注文件"配对,缺了它纯背景图会被漏掉。
        (out_labels_dir / (xml_path.stem + ".txt")).write_text("\n".join(lines), encoding="utf-8")

    return classes
```

> 📄 **`apps/platform/src/od_platform/data_pipeline/convert/converters/coco.py`** · 约 30 行 —— `@register("coco", supported_tasks=("detect","segment"))` + `convert_coco(...)`,读法沿用阶段 1,加白名单。与 `pascal_voc.py` 签名完全一致,所以两者对调度层"长得一样"。

```python
"""COCO -> YOLO 转换器(detect / segment)。

COCO 与 VOC 的差异只在"读法":整个数据集一个 .json,框是 [x,y,w,h](左上角 + 宽高)。
读完之后,归一化、写 txt、白名单处理与 VOC 完全一样 —— 这也是为什么三种 converter 能
被同一张表一视同仁地调用:它们的输入/输出契约相同。
"""
from __future__ import annotations

import json
import logging
from collections import defaultdict
from pathlib import Path
from typing import List

from od_platform.common.constants import AnnotationFormat, Task
from od_platform.data_pipeline.convert.registry import ConvertOptions, register

logger = logging.getLogger(__name__)


@register(AnnotationFormat.COCO, supported_tasks=(Task.DETECT, Task.SEGMENT))
def convert_coco(input_dir: Path, out_labels_dir: Path, options: ConvertOptions) -> List[str]:
    out_labels_dir.mkdir(parents=True, exist_ok=True)

    # COCO 习惯整个数据集一个 json(取目录下第一个);先把它解析成几张查找表。
    data = json.loads(sorted(input_dir.glob("*.json"))[0].read_text(encoding="utf-8"))
    cat_name = {c["id"]: c["name"] for c in data["categories"]}   # 类别 id → 名字
    img_info = {im["id"]: im for im in data["images"]}            # 图 id → 图信息(含宽高/文件名)
    per_image = defaultdict(list)                                 # 图 id → 它的所有标注
    for ann in data["annotations"]:
        per_image[ann["image_id"]].append(ann)

    classes: List[str] = list(options.classes) if options.classes else []
    discovering = options.classes is None                         # 同 VOC:None=探测,[...]=白名单
    for img_id, im in img_info.items():
        W, H = float(im["width"]), float(im["height"])
        stem = im["file_name"].rsplit(".", 1)[0]                  # 去扩展名,作为输出 txt 的文件名
        lines: List[str] = []
        for ann in per_image.get(img_id, []):
            name = cat_name[ann["category_id"]]
            if name not in classes:
                if discovering:
                    classes.append(name)
                else:
                    continue
            cls_id = classes.index(name)
            x, y, w, h = ann["bbox"]                              # COCO:左上角 (x,y) + 宽高 (w,h)
            lines.append(f"{cls_id} {(x+w/2)/W:.6f} {(y+h/2)/H:.6f} {w/W:.6f} {h/H:.6f}")
        (out_labels_dir / (stem + ".txt")).write_text("\n".join(lines), encoding="utf-8")

    return classes
```

写 `@register(AnnotationFormat.PASCAL_VOC, ...)` 时,`"pascal_voc"` / `"detect"` 这些字符串,马上要在两个 converter、CLI、测试里反复出现——又是 1.3 那个漂移的味道。所以**现在**(不是开篇,是被这个具体需求逼到的此刻)给项目建一张"共同词汇表" `constants.py`,收下第一批词:

> 📄 **`apps/platform/src/od_platform/common/constants.py`(今天它的第一段)** · 约 20 行 —— 只有"标注格式名"和"任务类型"两段。划分策略、各种阈值,等后面撞到再加。**你会看着它一行行长出来。**

```python
"""项目级共享常量——所有模块的"共同词汇表"。
放这里的标准:多模块共享(>=2 处用到) + 纯定义无逻辑 + 极少改动。本文件按需增长。
"""
from __future__ import annotations
from typing import Tuple

# —— 标注格式名(阶段 2:@register 要用,且会在多个 converter/CLI/测试复用)——
class AnnotationFormat:
    PASCAL_VOC = "pascal_voc"
    COCO       = "coco"
    @classmethod
    def all(cls) -> Tuple[str, ...]:
        return (cls.PASCAL_VOC, cls.COCO)

# —— 任务类型(同上)——
class Task:
    DETECT  = "detect"
    SEGMENT = "segment"
    @classmethod
    def all(cls) -> Tuple[str, ...]:
        return (cls.DETECT, cls.SEGMENT)
```

## 回头看:加一种格式,从"改旧代码"变成"加一个文件"

把镜头拉回 1.3 那个问题:**加一种新格式,要动多少地方?** 一路走下来,这件事被彻底改写了:

| 同一件事 | 一开始(`if/elif`) | 走到注册表 |
|---|---|---|
| 加一种新格式 | 改旧代码:回去加 `elif`(可能好几处) | 只**新增一个文件** |
| "支持哪些格式"在哪 | 散在 N 段判断里 | 就一份:那张表 |
| 牵动几个文件 | 可能好几个 | 一个(新文件) |
| 分发代码会膨胀吗 | 会,越加越长 | 不会,永远那几行 |
| 谁决定"名字→函数" | 中心的 `if/elif` | 每个实现**自己声明** |

> 💡 **金句**:**"注册表把'谁支持什么'的决定权,从中心的 `if/elif` 下放给了每个实现自己(这个转向有个名字叫控制反转)。于是'扩展'从'改一段越来越长、还到处复制的代码',变成了'加一个互不打扰的小文件'。"** ——这就是 2.1 那个"加东西不碰旧代码"的愿望,终于兑现。

## ⚠️ 慢着:注册表什么时候是多余的

讲到这儿你可能想到处都用注册表。**先按住。** 它不是"更高级所以默认更好"的东西:

> **只有一种实现时,注册表纯属过度设计。** 没东西可分发,你却凭空多了表、装饰器、自动发现一堆零件,读代码的人还得绕一圈才看懂"到底调了啥"。它该上场的时刻是:你已经有 **2 个以上真实实现,而且预期还会继续加**(本课的格式 VOC/COCO/YOLO 三种、划分策略三种,D4 的检查项十几个,都满足)。**口诀:先朴素写,撞到 `if/elif` 到处复制、且确定还要继续加,才升级。**

回想我们这一路:正是撞到了第二种格式、撞到漂移、撞到中间层的别扭,被一个个问题推着,才走到注册表的。**这才是它该出现的方式——不是开局就架好,而是被需求逼出来。**

<!-- ↑↑↑ 讲师块结束 -->

第三种来源很常见:数据本身**已经是 YOLO 格式**了——别人导出的、上一批训练剩下的。它不需要"换算",只需要把散落的 `.txt` **收拢**到我们的目录里。按阶段 2 立下的规矩,加它应该**只是往 `converters/` 丢一个文件**。我们来验证这句话是不是真的。

> 📄 **`apps/platform/src/od_platform/data_pipeline/convert/converters/yolo.py`** · 约 30 行
> **内容**:`@register("yolo", supported_tasks=("detect","segment"))` + `convert_yolo(input_dir, out_labels_dir, options)`——把原始 `.txt` 搬到目标目录(优先硬链接、跨盘退回复制)。
> **为什么这样设计**:YOLO→YOLO 是"直通",不解析、不换算,只搬运;硬链接零拷贝、省磁盘。
> **现在为什么不做别的**:① 为什么这里 `options.classes` 是**必填**、不像 voc/coco 可探测?因为 YOLO 的 txt 里**只有数字 `class_id`、没有类名**——名字只能由用户给(否则 yaml 的 `names` 没法写)。② 为什么不顺手支持 `--classes` 删类?见下方的边界说明:对 yolo 删类会让 yaml 和 txt 的 id 对不上,是个隐蔽的坑,所以这里**明确拒绝**。

```python
"""YOLO -> YOLO 直通转换器(detect / segment)。

"直通"的意思:源数据已经是 YOLO 格式,不需要任何坐标换算,只需把散落的 .txt 搬到
统一的输出目录。这是"接口相同、行为却不同"的典型 —— 它和 voc/coco 签名一致,但内部
什么都不解析。

⚠️ --classes 在这里的边界(重要,务必讲清):
    yolo 的 txt 每行只有数字 class_id、不含类名。所以 options.classes 在这里是
    【必填的命名信息】(没有它就没法写 yaml 的 names),而【不是过滤器】。
    本转换器原样搬运、不改内容、不重映射 id;因此对 yolo 源数据"删类"无法走 --classes
    —— 那样只会让 yaml 的 names 与 txt 里的 id 对不上(越界 / 张冠李戴)。
    yolo 源要删类,只能在源头重写 txt。
"""
from __future__ import annotations

import logging
import os
import shutil
from pathlib import Path
from typing import List

from od_platform.common.constants import AnnotationFormat, Task
from od_platform.data_pipeline.convert.registry import ConvertOptions, register

logger = logging.getLogger(__name__)


@register(AnnotationFormat.YOLO, supported_tasks=(Task.DETECT, Task.SEGMENT))
def convert_yolo(input_dir: Path, output_labels_dir: Path, options: ConvertOptions) -> List[str]:
    if not options.classes:
        # yolo 不含类名,必须由用户显式给出类别顺序,否则 yaml 的 names 无从填起。
        raise ValueError("yolo 格式不含类别名信息,必须通过 options.classes 显式提供类别顺序。")

    txt_files = sorted(input_dir.glob("*.txt"))
    if not txt_files:
        raise FileNotFoundError(f"在 {input_dir} 下未找到任何 yolo txt")

    output_labels_dir.mkdir(parents=True, exist_ok=True)
    for txt in txt_files:
        dst = output_labels_dir / txt.name
        try:                                    # 优先硬链接(零拷贝、省磁盘);跨盘会失败
            if dst.exists():
                dst.unlink()
            os.link(txt, dst)
        except OSError:                         # 跨文件系统等情况:退回普通复制
            shutil.copy2(txt, dst)

    logger.info("YOLO 直通: %d 个 txt 就位, %d 个类别", len(txt_files), len(options.classes))
    return list(options.classes)
```

注意第一行那个 `AnnotationFormat.YOLO`——它在 `constants.py` 里也得有。这是 `constants.py` 今天**第二次生长**(阶段 2 加了 `PASCAL_VOC`/`COCO`,现在补上 `YOLO`,`all()` 也跟着多一个)。

### 但要小心:接口长得一样 ≠ 行为处处一样

三个 converter 的签名完全一致 `(input_dir, out_labels_dir, options) -> list[str]`,所以调度层"一视同仁"地调它们。但**同一个 `--classes` 参数,对三种格式的意义并不相同**:

| 格式 | `--classes` 的作用 | 能用它"删类"吗 |
|---|---|---|
| `pascal_voc` | 白名单:不在名单的框**跳过**,并按名单**重排** `class_id` | ✅ 能(xml 里有类名,可按名字筛) |
| `coco` | 同上 | ✅ 能(json 里有类名) |
| `yolo` | **仅命名**:给那些数字 id 配上名字,**不过滤、不重映射** | ❌ 不能(删了会让 yaml 的 `names` 和 txt 的 id 对不上) |

> 💡 **金句**:**"接口签名相同,不代表行为处处相同。`--classes` 走 voc/coco 是过滤器,走 yolo 只是命名信息——这种差异要写在文档和代码注释的明处,而不是让用户自己踩坑。"** YOLO 源要删类,只能在源头重写 txt。(阶段 5 的报告会据此对 yolo 给出不同的建议文案。)

## 🧭 转换这一半做完了——接下来要开发什么

停一下,抬头看看整张地图。到这里,**"转换"这半边**已经完整:三种格式(VOC / COCO / YOLO)都能进来,统一吐出 YOLO 标注,加新格式只是加文件。但回看[起点与终点]那条命令,用户要的不止"转换":

```
odp-transform --dataset X --format voc --split-strategy stratified --classes ...
└─ 已完成 ─┘                  └────────── 还没做 ──────────┘
```

一个真正可训练的数据集,转换之后还差三步,正是**下半场**要开发的:

1. **划分(split)**:把转换出来的样本切成 `train / val / test`。这件事本身又是一次"多实现"——随机、分层、多标签……所以它会**第二次撞上注册表**(也正是在这一步,我们把阶段 2 内联的扫描逻辑抽成公共件)。→ **阶段 4**
2. **落盘 + 喂模型(materialize + yaml)**:把三组样本摆进目录,并生成 ultralytics 能直接吃的 `dataset.yaml`。→ 阶段 4 内
3. **可信(report + 前置校验)**:开跑前打一份类别平衡报告、把残缺数据集挡在门外。→ **阶段 5 / 6**

最后由一个**指挥家(orchestrator)** 把"转换 → 报告 → 划分 → 落盘 → yaml"串成一条龙,再包一层 `odp-transform` CLI(**阶段 7**),配齐测试(**阶段 8**)。

整条流水线的全貌(先扫一眼,细节在各阶段长出来):

```
        ┌─ convert ─┐   ┌──────────── split ────────────┐
raw ──▶ │ 三种格式  │ ─▶│ 策略划分 → 落盘 → 写 yaml      │ ─▶ 可训练数据集 + yaml
        └───────────┘   └───────────────────────────────┘
              ▲                                  ▲
         report(平衡报告 / 覆盖率前置校验)        指挥家 orchestrator + odp-transform CLI 串起全程
```

> 📌 **再往后(本课之外,给学员一个坐标)**:这份 `dataset.yaml` 正是 **D4 验证子系统**(检查标注合法性、可视化抽检)和 **D5 训练子系统**(喂给 ultralytics 训练 / 评估)的输入。今天产出的 `odp_meta`(策略 / 种子 / 比例)就是为了半年后回查实验时,还说得清"这个数据集当初是怎么切出来的"。

好,进入下半场。
