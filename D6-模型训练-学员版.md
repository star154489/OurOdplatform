## 上一节回顾 + 本节挑战

**D5 给了我们什么。** 一个 `runtime_config` 子系统:用户写一份 `train.yaml`(或 `odp-gen-config train` 现生成),加上命令行覆盖,经过合并、验证、类型转换,变成一个 `YOLOTrainConfig` 对象;再调 `.to_ultralytics_kwargs()` 就能得到一份干净的、可以直接喂给 ultralytics 的参数字典。配置这件事,D5 收尾得干干净净。

**但 D5 只到"配置"为止。** 它产出的 config,自己不会去训练——它只是"描述了这次该怎么跑"。真把训练跑起来,你立刻会撞上一串配置子系统管不着的问题:

1. **谁来把 config 喂给 ultralytics、按什么顺序?** 是先校验数据还是先建模型?训练前要不要打一份环境快照?这套"编排"该放哪。
2. **训练前数据是脏的,谁来拦?** D4 能查出"标签里有越界的 class_id",但 D4 是个独立命令——开训这条路上,谁负责在 `model.train()` 之前调它、并且在数据不合格时**不让训练开始**?
3. **跑完一地鸡毛,谁来收拾?** ultralytics 把 `best.pt` / `last.pt` / 各种图丢在 `runs/` 里,目录名是它自己按规则生成的。你想长期留的那个权重在哪?日志文件叫什么?三个月后还认得出是哪次跑的吗?
4. **训练崩了,异常往哪去?** 一个跑了两小时的训练,中途因为某个边角问题抛异常——这个异常该让调用方(CLI、notebook、未来的 web 后端)怎么收场?

这四个问题,没有一个是 D5 该管的,也没有一个是 `model.train()` 自带的。它们正是今天要一块块逼出来的东西。

**照旧不先给答案。** 先写最直接的版本,把它往真实里推,看它在第几步撑不住;每撑不住一次,逼出一块结构。等结构攒齐,`training/` 这棵子树会自己长出来——到那时,它的每一个文件你都知道为什么在那。

# 阶段 0:立规矩

写第一行代码之前,先立两件事:**这个子系统的边界在哪**(它该做什么、不该做什么),以及**我手上已经有哪些现成的邻居**(凭什么不重造)。这两件事立准了,后面整篇都顺;立歪了,你会不知不觉把 D5/D4/D2 的活又写一遍。

## 0.1 TrainService 的边界:它不发明任何东西

训练子系统的核心会是一个 `TrainService`。在写它之前,先把它的**职责边界**钉死。一句话:**它是个编排器(orchestrator),不是发明家。**

```
                 ┌────────────────────────────────────────┐
                 │   D6: TrainService 编排器              │
                 │                                        │
                 │   它【不发明】(全部走邻居):           │
                 │     · 配置合并 / 验证   → D5 runtime_config │
                 │     · 数据校验          → D4 validate_dataset│
                 │     · 名字 / 路径解析    → D3 refs           │
                 │     · logging handler   → D2 logging_utils  │
                 │     · 训练执行          → ultralytics        │
                 │                                        │
                 │   它【只做】:                          │
                 │     · 把上面这些按正确顺序编排起来      │
                 │     · 训练后整理(改日志名 / 归档权重) │
                 │     · 写一份审计快照(audit)          │
                 │     · 兜住异常(永不让它穿到调用方)   │
                 └────────────────────────────────────────┘
```

这条边界不是口号,是**可以用 CI 跑的硬指标**。比如:service 里不该再出现 D5 的内部细节(它该调 D5 的对外 API,而不是自己 `import` D5 的合并器零件):

```bash

  apps/platform/src/od_platform/training/service.py
```


## 0.2 先盘家底:D0–D5 已经给了我们什么(动手前的必修课)

这是 D6 最重要的一节。在写训练子系统之前,把 `common/` 和前几天的产出**逐个过一遍**,问同一个问题:**训练要用到的能力,它是不是已经有了?**

| 我训练时会需要…… | 已有的邻居 | 来自 |
|---|---|---|
| 一份合并好、验证过的训练配置 | `build_train_config(yaml, cli)` → `(config, merger)` | D5 |
| 把配置变成 ultralytics 参数 | `config.to_ultralytics_kwargs()` | D5 |
| 训练前查数据合不合格 | `validate_dataset(...)` + 看 exit code | D4 |
| **把"模型名/路径"解析成实际权重文件** | **`refs.resolve_pretrained_model(name)`** | **D3** |
| **把"数据集名/路径"解析成 yaml 文件** | **`refs.resolve_yaml(name)`** | **D3** |
| 任务类型这个封闭取值 | `Task.DETECT` / `Task.SEGMENT` / `Task.all()` | D3 |
| 打一份 GPU/CUDA/环境快照(可复现) | `system_utils.log_device_info()` | D2 |
| 给某段流程计时 | `performance_utils.@time_it` | D2 |
| 拿一个挂对地方的 logger | `logging_utils.get_logger()` | D2 |
| 中文对齐打印(表格 / 字段名) | `string_utils.pad_to_width(...)` | D1 |

把这张表读两遍。它在告诉你:**训练子系统真正"缺"的东西,比你以为的少得多。** 配置、校验、路径解析、日志、计时、环境快照——全有了。D6 要新写的,只是"把它们按训练的顺序编排起来",外加几个**确实没有**的训练专属工具(跑完整理日志、归档权重、格式化指标)。

### 重点说一件容易做错的事:模型路径解析

训练时,用户在 `train.yaml` 或命令行写 `model: yolo26n.pt`——这是个**裸文件名**。你需要把它解析成实际的权重文件路径:优先去本项目的预训练目录找一份,没有再让 ultralytics 自己下载。

看到"解析模型路径"这几个字,**第一反应不该是"我来写个 `resolve_model_path`"**,而该是——

```bash
# 🐧 先翻一下 refs 有没有
grep -n "def resolve" apps/platform/src/od_platform/common/refs.py
```

你会看到 D3 早就留好了:

```python
# od_platform/common/refs.py(D3 §8.1 已建,节选)
def resolve_pretrained_model(ref: str) -> Path:
    """【训练时】用的预训练权重:裸名字 → models/pretrained/<name>.pt。(给【训练】子系统的接缝)"""
    return resolve_ref(ref, base_dir=PRETRAINED_MODELS_DIR, default_suffix=".pt")


def resolve_trained_model(ref: str) -> Path:
    """【推理/验证时】用的已训练权重:裸名字 → models/trained/<name>.pt。(给【推理】子系统的接缝)"""
    return resolve_ref(ref, base_dir=TRAINED_MODELS_DIR, default_suffix=".pt")
```

D3 当时就把这件事想透了,还留了一句金句钉死设计:

> 💡 **D3 §8.1 原话**:**"`model` 不能只有一个解析器。训练要的是【预训练】权重(`pretrained/`),推理要的是【你训好】的权重(`trained/`)——两个完全不同的场景。所以是两个语义明确的封装,各认各的目录;绝不写一个 `resolve_model` 再靠'猜'去决定该从哪找——那种'聪明'迟早半夜坑你。"**

所以训练子系统的模型解析,**直接用 `refs.resolve_pretrained_model`**。同理,数据集 yaml 的解析用 `refs.resolve_yaml`(D4 验证子系统已经在用它了——我们和 D4 走同一个解析器,不另立一套)。

⚠️ **设计点(为什么这件事值得专门强调)**:"看到一个需求就新写一个工具"是最自然、也最危险的冲动。如果 D6 自己写一个 `resolve_model_path`,马上会产生两个后果:① 和 D3 的 `refs` 重复,以后改解析规则要改两处;② 你多半会写成"一个函数 + 一个 `search_dirs` 参数,靠调用方传不同目录来区分训练/验证"——这恰恰是 D3 金句里点名要避免的"一个 resolver 靠猜"。**两个场景就是两个解析器**,这是 D3 替你做好的决定,D6 照用即可。

🤔 **思考题(30 秒)**:`refs` 里有 `resolve_pretrained_model` 和 `resolve_trained_model` 两个。训练子系统(D6)该用哪个?验证子系统(D7)又该用哪个?为什么不能合成一个?

## 0.3 立三条工程规矩

边界和家底盘清了,最后立三条贯穿全篇的规矩。它们都很短,但每一条后面都有一次真实的踩坑(后面会撞到)。

**规矩一:业务模块不碰 logging handler。** 训练子系统的业务代码(service、archive 等)**只负责"发声"**(`logger.info(...)`),绝不负责"把声音接到哪去"(`addHandler` / `setLevel` / `basicConfig`)。挂 handler 是**程序入口**(CLI)的唯一职责。为什么?阶段 3 会撞墙给你看。

```bash
# 🐧 业务模块不应出现 addHandler / setLevel / basicConfig
grep -rnE "addHandler|setLevel|basicConfig" \
  apps/platform/src/od_platform/training/
# 期望:没有输出(唯一豁免在阶段 5 讲,且不在 training/ 里)
```

**规矩二:fail-fast 用退出码,不用异常穿透。** 训练前的校验若不通过,该让流程**干净地停下并返回一个非零退出码**,而不是让一个异常一路抛到用户脸上。service 的对外方法**永不抛异常**——它把结果(成功/失败)包成一个对象返回,让调用方决定怎么收场。

**规矩三:子系统之间的依赖方向是单向的。** 训练是上游,验证/推理是下游。所以**验证/推理子系统可以依赖训练产物的【目录约定】,但代码上不该 import 训练子系统**。这条今天先立住(D7 会兑现它,届时 evaluation 不会 `from od_platform.training import ...`)。

```bash
# 🐧 (留给 D7 的契约)evaluation 不该 import training
grep -rn "from od_platform.training" \
  apps/platform/src/od_platform/evaluation/ 2>/dev/null
# 期望:D7 写完后,这里也没有输出
```

> **金句**:**"规矩不是写给道德看的,是写给 CI 看的。一条规矩如果不能变成一句 `grep`,它迟早会被违反。"**

✅ **检查点(阶段 0 收尾)**:你现在还没写任何 training/ 代码,但你应该能回答三个问题——① TrainService 该做什么、不该做什么?② 训练要解析模型路径,该用哪个现成函数?③ 训练产物归档到哪个目录?如果三个都答得上,地基就立稳了,可以开始撞墙。

## 1.1 复刻一份 legacy `yolo_train.py`

很多人第一次写 YOLO 训练脚本,长这样——argparse 收几个参数,`YOLO(...)` 一把梭:

```python
# /tmp/yolo_train_naive.py — 朴素方案,演示用,不进 git
import argparse
from ultralytics import YOLO


def main():
    p = argparse.ArgumentParser()
    p.add_argument("--model", default="yolo26n.pt")
    p.add_argument("--data", required=True)
    p.add_argument("--epochs", type=int, default=100)
    p.add_argument("--lr0", type=float, default=0.01)
    p.add_argument("--batch", type=int, default=16)
    p.add_argument("--imgsz", type=int, default=640)
    args = p.parse_args()

    model = YOLO(args.model)                 # 模型名直接交给 ultralytics
    model.train(
        data=args.data,                      # 数据集 yaml 路径直接透传
        epochs=args.epochs,
        lr0=args.lr0,
        batch=args.batch,
        imgsz=args.imgsz,
    )


if __name__ == "__main__":
    main()
```

它能跑。但它"能跑"和"能用"之间,隔着今天整篇 D6。

## 1.3 🧱 撞墙①:朴素方案的四个伤痛

把这版脚本当成真要用的东西,推它四下:

**伤痛 A:这次的 `lr0` 到底谁说了算?——配置溯源黑洞。**

```text
# 用户问:我这次跑的 lr0=0.001,是从哪来的?
```

# 阶段 2:先立对外面板——TrainResult + TrainService

撞完墙,先别急着填内脏。先把训练子系统的**对外形状**定下来:用户怎么调它、它返回什么。形状对了,内脏后面一块块填进去。

## 2.1 选型:编排器(orchestrator),不是 `model.train` 的包装器

有两种做法,差别很大:

- **包装器(wrapper)**:写个函数把 `model.train()` 包一层,加点参数转换。——这只是给 ultralytics 套层皮,治不了上面任何一个伤痛。
- **编排器(orchestrator)**:写一个 `TrainService`,它**按顺序调度** D5 配置、D4 校验、D2 日志、ultralytics 训练、训练后整理——每一段都调邻居,自己只管"顺序"和"兜底"。

我们要的是后者。理由阶段 0 已经立过:service 不发明,只编排。**包装器关心"怎么调 train",编排器关心"训练这件事的完整生命周期"。**

## 2.2 TrainResult:训练结果的载体(frozen dataclass)

service 的对外方法该返回什么?不该是"裸的 ultralytics 结果对象"(那等于把 ultralytics 的内部结构泄露给调用方),而该是一个**我们自己定义的、稳定的结果载体**。它主要是"数据",所以用 dataclass,而且 `frozen=True`(造好就不可改——结果是事实,不该被下游偷偷篡改):

```python
# od_platform/training/service.py(节选 —— 结果载体)
from __future__ import annotations
from dataclasses import dataclass
from pathlib import Path
from typing import Optional


@dataclass(frozen=True)
class TrainResult:
    """一次训练的结果事实(不可变载体)."""
    success: bool                       # 这次训练成功了吗
    save_dir: Optional[Path]            # ultralytics 实际落盘目录
    best_weight: Optional[Path]         # 归档后的 best.pt 路径(失败则 None)
    metrics: Optional["TrainMetrics"]   # 训练指标(阶段 4 定义)
    error: Optional[str] = None         # 失败时的人话原因(成功则 None)
```

🔗 **概念手册**:`TrainResult` 是个 `@dataclass(frozen=True)`——为什么数据载体默认该不可变、`frozen` 在多消费者共享时为什么重要,见**手册① dataclass 全解**。下面的 `TrainMetrics` 同理。

⚠️ **设计点(为什么返回自定义载体而不是 ultralytics 的结果)**:如果 service 直接把 ultralytics 的 `results` 对象返回出去,调用方就被绑死在 ultralytics 的内部结构上——哪天 ultralytics 改了字段名,你所有调用方都得改。`TrainResult` 是一道**隔离层**:它把"我们关心的事实"(成不成、权重在哪、指标多少)固定下来,把 ultralytics 的易变结构挡在子系统内部。这和 D5 的 `to_ultralytics_kwargs` 是同一种"边界处做翻译"的思路。

## 2.3 `TrainService.train()` 的签名:keyword-only 开关

service 的核心方法 `train()`。它的签名要解决一个问题:训练流程里有几个**可开关的环节**(训练前要不要校验数据、训练后要不要归档、要不要改日志名),用户得能精细控制。这些开关用 **keyword-only 参数**(强制写参数名):

```python
class TrainService:
    """训练编排器:调度 D5 配置 + D4 校验 + D2 日志 + ultralytics 训练 + 训练后整理."""

    def train(
        self,
        config_path: str,
        data: str,
        *,                              # ← 星号之后全是 keyword-only
        cli_overrides: Optional[dict] = None,
        pre_validate: bool = True,      # 训练前是否跑 D4 校验
        archive: bool = True,           # 训练后是否归档权重到 models/trained/
        rename_log: bool = True,        # 训练后是否把日志名对齐 save_dir
    ) -> TrainResult:
        """编排一次完整训练,永不抛异常,结果包进 TrainResult 返回."""
        ...                             # 内脏在后续阶段一块块填进来
```

为什么强制 keyword-only?对比一下就懂:

```python
# ✅ keyword-only:一眼看出在关哪个开关
service.train("train.yaml", "rsod.yaml", pre_validate=False)

# ❌ 如果允许位置参数:这种调用合法,但没法读
service.train("train.yaml", "rsod.yaml", False, True, False)
#                                         ↑ 这第一个 False 是 pre_validate?archive?谁知道
```

> **金句**:**"keyword-only 不是麻烦,是把'调用处的可读性'强制写进 API。一个 `False` 看不出含义,`pre_validate=False` 一眼就懂。"**

⚠️ **设计点(三个开关的默认值都是"完整行为")**:`pre_validate` / `archive` / `rename_log` **默认全 `True`**——也就是说,默认情况下 service 做"完整的、负责任的"训练。开关存在,是为了让特殊场景能关掉某一环(跑 smoke test 关 `pre_validate`、测试时关 `archive` 防止污染 `models/trained/`、自定义日志命名时关 `rename_log`)。**默认完整、按需精简**,而不是默认简陋、按需开启。

## 2.4 `train_yolo()`:给"不想实例化 service"的人

有些场景(notebook、一次性脚本)不想先 `service = TrainService()` 再调方法,就想一行搞定。给个便捷函数:

```python
def train_yolo(
    config_path: str,
    data: str,
    *,
    cli_overrides: Optional[dict] = None,
    pre_validate: bool = True,
    archive: bool = True,
    rename_log: bool = True,
) -> TrainResult:
    """便捷函数:内部就是 TrainService().train(...),给不想管实例化的人."""
    return TrainService().train(
        config_path, data,
        cli_overrides=cli_overrides,
        pre_validate=pre_validate,
        archive=archive,
        rename_log=rename_log,
    )
```

两条路,各有适用:

- **实例化路**(`TrainService().train(...)`):适合长进程 / 服务化 / 需要复用同一个 service 实例。
- **便捷函数路**(`train_yolo(...)`):适合 notebook / 一次性脚本 / CLI 入口。

⚠️ **设计点(便捷函数不是第二套实现)**:`train_yolo` 内部**就是** `TrainService().train(...)`——它不重复任何逻辑,只是省掉实例化那一步。两条路最终走的是同一段代码,所以行为绝不会分叉。这是"提供便利"和"制造重复"之间的正确做法:**便利是入口的不同,不是实现的不同。**

# 阶段 3:handler 该装哪里?——logging 的硬纪律

开始填内脏。第一块最不起眼、却最容易出错:训练真跑起来,日志该怎么挂。这里藏着一堵很多人都撞过的墙。

## 3.1 先回顾 D2 的 `get_logger`

D2 建过一个 `get_logger`,它把 handler(控制台 + 文件)挂在一个**有名字的 root logger** 上——名字就是 `"od_platform"`:

```python
# D2 logging_utils.py(简化回顾,真版还有彩色输出 / 路径约定 / 幂等保护)
import logging

def get_logger(log_file=None, level=logging.INFO):
    logger = logging.getLogger("od_platform")     # ← 关键:挂在 "od_platform" 这个 named root
    logger.setLevel(level)
    if not logger.handlers:                        # 幂等:已挂过就不重复挂
        # ... 加控制台 handler、(可选)文件 handler ...
        pass
    return logger
```

记住这个名字 `"od_platform"`——下面的墙就栽在它上面。

## 3.2 🧱 撞墙②:`logging.getLogger()` 不带参数 ≠ `"od_platform"`

朴素的想法是:业务模块里要打日志,直接 `logging.getLogger()` 就行了吧?**不行。** 看反模式:

```python
# ❌ 反模式 —— D6 CLI 入口这么写
import logging

def main():
    logging.basicConfig(level=logging.INFO)        # 挂到【无名 root】
    logger = logging.getLogger()                   # 拿的也是【无名 root】
    logger.info("开始训练")
    # ... 调 service ...
```

```python
# ❌ 与此同时,service.py 里这么写
import logging
logger = logging.getLogger("od_platform")          # 拿的是【od_platform named root】
# service 打的日志,挂在 od_platform 上;CLI 挂的 handler 在【无名 root】上
# 结果:两者各挂各的,日志要么重复输出、要么 service 的日志没进文件
```

**问题的根**:`logging.getLogger()` 不带参数,拿到的是 Python 的**无名 root logger**;`logging.getLogger("od_platform")` 拿到的是**名为 od_platform 的 logger**。这是**两个不同的 logger 对象**。如果 CLI 把 handler 挂在无名 root,而业务模块在 od_platform 上发声,两者就对不上——日志可能重复(无名 root 和 od_platform 的传播链都挂了 handler)、也可能 service 的日志根本没进你指定的文件。

> **金句**:**"`getLogger()` 和 `getLogger('od_platform')` 看着像,其实是两个 logger。日志这种'看着没报错但行为不对'的坑,最费时间——因为它不崩,只是悄悄不对。"**

## 3.3 正解:入口挂 handler,业务只发声

纪律(阶段 0 立过的规矩一)在这里兑现:**只有程序入口(CLI)负责挂 handler,而且必须挂在和业务模块一致的 named root 上。**

```python
# ✅ 正解 —— CLI 入口 cli/train_model.py
from od_platform.common.logging_utils import get_logger

def main():
    logger = get_logger(log_file=...)              # ← 调 D2 的 get_logger,挂在 "od_platform"
    # 不要自己 basicConfig、不要自己 getLogger()、不要自己 addHandler
    ...
```

```python
# ✅ 正解 —— 业务模块 training/service.py
import logging
logger = logging.getLogger(__name__)               # ← __name__ = "od_platform.training.service"
# 它是 "od_platform" 的子 logger,日志会【传播】到 od_platform 上的 handler
# 业务模块【只发声】:logger.info / logger.warning,绝不 addHandler / setLevel
```

为什么 `getLogger(__name__)` 是对的?因为 `__name__` 在 `od_platform/training/service.py` 里是 `"od_platform.training.service"`——它是 `"od_platform"` 的**子 logger**。Python 的 logging 会把子 logger 的日志**向上传播**到父 logger 的 handler。所以业务模块用 `__name__` 拿 logger、只管发声,声音自动流到 CLI 在 `"od_platform"` 上挂的 handler。**入口挂一次,全子系统的日志都归位。**

## 3.4 业务模块统一只发声(可 CI 自检)

这条纪律可以用阶段 0 立的那条 grep 守住:

```bash
# 🐧 业务模块不该出现 addHandler / setLevel / basicConfig
grep -rnE "addHandler|setLevel|basicConfig" \
  apps/platform/src/od_platform/training/
# 期望:没有输出。这些操作只该出现在 cli/ 入口(且 cli 是间接通过 get_logger 调的)
```

```powershell
# 🪟 Windows (PowerShell)
Select-String -Path apps\platform\src\od_platform\training\*.py `
  -Pattern "addHandler|setLevel|basicConfig"
# 期望:没有输出
```

⚠️ **设计点(为什么 service 连 FileHandler 都不持有)**:你可能想"那让 service 自己持有一个 FileHandler,训练时写文件不就好了?"——不行。一旦 service 自己挂 handler,就回到了撞墙②:谁实例化两次 service,日志就重复一次;测试时 mock service 还得处理它的 handler。**handler 的生命周期只属于程序入口**,业务对象自始至终只是"发声者"。这条边界清晰了,日志才不会在多次调用 / 测试 / 服务化时出乱子。

# 阶段 4:接内脏——模型/数据集解析复用 refs,只补真正缺的工具

## 4.1 模型路径:用 `refs.resolve_pretrained_model`,不自写

训练时用户写 `model: yolo26n.pt`(裸名字)。我们要的行为是:**优先用本项目 `models/pretrained/` 里的那一份;本地没有,就让 ultralytics 自己下载。** 解析"裸名字 → `models/pretrained/<name>.pt`"这一步,refs 已经给了:

```python
# 直接用 D3 §8.1 留好的接缝,不新写任何解析函数
from od_platform.common.refs import resolve_pretrained_model

p = resolve_pretrained_model("yolo26n.pt")        # → <项目>/models/pretrained/yolo26n.pt
```

注意 refs 的解析是**纯语法的**——它只回答"这个名字按约定该落在哪",**故意不检查文件存不存在**(D3 明确拒绝用 `.exists()` 嗅探)。那"本地有就用、没有就让 ultralytics 下载"这层判断放哪?——放在**训练侧**,因为它是个**训练专属的策略**(验证子系统就**不要**这个兜底:验证的权重找不到,就该 fail-fast,见 D7)。所以 service 里加一小段策略,而不是去改 refs:

```python
# training/service.py 内部,解析模型(纯解析走 refs,下载兜底是训练自己的策略)
from od_platform.common.refs import resolve_pretrained_model

def _resolve_model_arg(model: str) -> str:
    """训练用模型实参:本地预训练权重存在则用它,否则交回裸名让 ultralytics 自己下."""
    p = resolve_pretrained_model(model)            # refs:纯语法解析(不查存在)
    if p.exists():
        logger.info(f"使用本地预训练权重:{p}")
        return str(p)
    logger.info(f"本地无 {p.name},交给 ultralytics 自动下载:{model}")
    return model                                   # 裸名 → ultralytics 去 hub 下
```

> **金句**:**"refs 回答'这名字该落在哪'(纯解析);'本地有没有、要不要下载'是训练自己的策略。解析归 refs、策略归子系统——各管一层,谁都不越界。"**

⚠️ **设计点(为什么不把 `.exists()` 兜底塞进 refs)**:很自然会想"干脆让 `resolve_pretrained_model` 自己查存在、自己兜底,多省事"。但那会犯两个错:① **破坏 D3 的纯语法原则**——refs 全篇坚持"不嗅探",一旦它开始 `.exists()`,`my.model` 这种带点名字就可能被误判,规则也不再"一句话说得清";② **把训练专属策略塞进共享层**——验证子系统不要这个下载兜底,如果 refs 自带它,验证就得想办法绕过。**纯解析放共享的 refs,带策略的判断放各自子系统**,这正是 D5 讲过的"共享 vs 专属"判据在路径解析上的复用。

## 4.2 数据集 yaml:用 `refs.resolve_yaml`,和 D4 走同一个

用户写 `data: rsod.yaml`(裸名字),要解析成 `configs/datasets/rsod.yaml`。这一步——**D4 验证子系统已经在用 `refs.resolve_yaml` 了**。我们和它走同一个解析器,绝不另写一套:

```python
from od_platform.common.refs import resolve_yaml

data_yaml = resolve_yaml("rsod.yaml")             # → <项目>/configs/datasets/rsod.yaml
# D4 的 validate_dataset 内部用的也是这个 resolve_yaml —— 同一个解析,零漂移
```

⚠️ **设计点(为什么"和 D4 同一个解析器"很重要)**:如果 D6 自己写一个 `resolve_dataset_path`,那"把 `rsod.yaml` 解析成实际路径"这件事就有了**两套实现**——D4 一套、D6 一套。哪天解析规则要改(比如支持子目录),你得改两处,漏一处就出现"D4 校验的是 A 文件、D6 训练的是 B 文件"的诡异错位。**同一个概念,只能有一个解析器**——这是 SSoT 在"路径解析"上的体现。所以 `common/dataset_path.py` 这种文件,这一版**不会有**。

## 4.3 现在补真正缺的:`common/result.py`——训练指标载体

复用完邻居,该看"什么是真没有、得自己建"的了。第一个:把 ultralytics 训练完返回的那一堆指标,收成一个**干净、稳定的载体**。ultralytics 的结果对象字段多且易变,我们只取关心的几个,装进一个 frozen dataclass:

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : result.py
# @Project   : ODPlatform
# @Function  : 训练/验证指标载体(frozen dataclass)+ 指标日志输出(数据/日志分离)
"""训练指标的稳定载体 + 格式化输出。

刻意把'数据'(TrainMetrics)和'日志输出'(log_train_metrics)分开:
载体只装事实,要不要打、怎么打,是另一件事。
"""
from __future__ import annotations

import logging
from dataclasses import dataclass
from typing import Optional

from od_platform.common.constants import Task
from od_platform.common.string_utils import pad_to_width


@dataclass(frozen=True)
class TrainMetrics:
    """一次训练/验证的核心指标(不可变)."""
    task: str                              # detect / segment(用 Task 的值)
    fitness: float                         # ultralytics 的综合分
    map50: float                           # mAP@0.5
    map50_95: float                        # mAP@0.5:0.95
    precision: float
    recall: float


# ── 数据驱动:每个任务要打哪些指标,写成一张表,加新任务只加一行 ──
_TASK_METRIC_FIELDS: dict[str, tuple[str, ...]] = {
    Task.DETECT.value:  ("map50", "map50_95", "precision", "recall", "fitness"),
    Task.SEGMENT.value: ("map50", "map50_95", "precision", "recall", "fitness"),
    # 将来加 Task.OBB,这里加一行即可,log 函数一行不用改
}


def log_train_metrics(
    metrics: TrainMetrics,
    *,
    logger: logging.Logger | None = None,
    key_width: int = 14,
) -> None:
    """按任务把该打的指标逐行打印(数据/日志分离:本函数只读 metrics,不改它)."""
    log = logger or logging.getLogger(__name__)
    fields = _TASK_METRIC_FIELDS.get(metrics.task, ("map50", "map50_95"))
    log.info(f"训练指标(task={metrics.task}):")
    for f in fields:
        log.info(f"  {pad_to_width(f, key_width)}: {getattr(metrics, f):.4f}")
```

🔗 **概念手册**:`TrainMetrics` 是 frozen dataclass(手册①);`_TASK_METRIC_FIELDS` 用 `Task.DETECT.value` 当键、"加任务只加一行"的数据驱动写法,正是**手册⑥ 枚举**里讲的"枚举做 SSoT、对外用 `.value`"。

⚠️ **设计点(数据 / 日志分离)**:`TrainMetrics` 只装事实,`log_train_metrics` 只负责打印——两者分开,而不是给 dataclass 加个 `.print()` 方法。为什么?因为"有哪些指标"和"要不要打、打给谁看(控制台?文件?)"是两件正交的事。分开后,你能在不打日志的情况下拿指标做别的(比如写进 audit、比较两次训练)。**载体管数据,函数管呈现。**

## 4.4 `common/config_log.py`——把 D5 的溯源换个视角打印

D5 的 `merger` 已经能产配置溯源报告了(`get_source_report` 按来源分组)。但训练日志里,用户更想要**按字段一行**看"这个值最终怎么来的"。这不是重做 D5 的活——是**在 D5 的数据之上换个视角呈现**。两个函数都**消费 D5 的 `merger.get_metadata()`**,自己不碰合并逻辑:

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : config_log.py
# @Project   : ODPlatform
# @Function  : 按字段维度打印配置参数 / 来源链(消费 D5 merger,纯视图层)
"""配置参数日志:跟 D5 get_source_report(按来源分组)互补——本模块按字段一行展示。

它【不重做】合并:所有来源信息都来自 D5 的 merger.get_metadata(field)。
"""
from __future__ import annotations

import logging
from typing import Any

from od_platform.common.string_utils import pad_to_width


def log_effective_config(
    config: Any, merger: Any, *,
    logger: logging.Logger | None = None,
    key_width: int = 20, section_width: int = 60,
) -> None:
    """每个字段当前生效值 + 来源,一行一个."""
    log = logger or logging.getLogger(__name__)
    log.info("=" * section_width)
    log.info(pad_to_width("配置参数信息", section_width, align="center"))   # ← CJK 安全居中
    log.info("-" * section_width)
    for name in config.__class__.model_fields:
        value = getattr(config, name, None)
        meta = _safe_get_metadata(merger, name)
        src = meta.source_label if meta is not None else "未知"
        log.info(f"{pad_to_width(name, key_width)}: {value}  (来源: {src})")


def log_override_chains(
    config: Any, merger: Any, *,
    logger: logging.Logger | None = None,
    key_width: int = 20, section_width: int = 60,
) -> None:
    """每个字段完整来源链(DEFAULT → YAML → CLI 顺序,贴合阅读习惯)."""
    log = logger or logging.getLogger(__name__)
    log.info("-" * section_width)
    log.info(pad_to_width("配置覆盖情况", section_width, align="center"))   # ← 同样用 pad_to_width
    log.info("-" * section_width)
    for name in config.__class__.model_fields:
        meta = _safe_get_metadata(merger, name)
        if meta is None:
            log.info(f"{pad_to_width(name, key_width)}: {getattr(config, name, None)}")
            continue
        chain = list(reversed(meta.chain()))            # D5 链是 newest-first,reverse 成 oldest-first
        chain_str = " <- ".join(f"{m.value}({m.source_label})" for m in chain)
        log.info(f"{pad_to_width(name, key_width)}: {chain_str}")


def _safe_get_metadata(merger: Any, field_name: str) -> Any:
    """get_metadata 的防御封装:测试传 mock 时不崩."""
    if not hasattr(merger, "get_metadata"):
        return None
    try:
        return merger.get_metadata(field_name)
    except Exception:
        return None
```

⚠️ **设计点(标题也用 `pad_to_width`,不用 `.center()`)**:注意两个标题用的是 `pad_to_width(..., align="center")`,**不是** `"配置参数信息".center(60)`。原因 D1 撞墙⑤ 讲过:Python 的 `str.center()` 按"字符个数"算宽度,而一个中文字在终端占两格,所以中文标题用 `.center()` 会**居中歪掉**。`pad_to_width` 是 D1 专门为此造的 CJK 感知工具——字段对齐用它,标题也得用它,**别一半用对、一半用错**。

⚠️ **设计点(这是"纯视图层",不是第二个配置系统)**:`config_log` 里没有任何"合并""覆盖判断"的逻辑——它要的所有信息都来自 `merger.get_metadata()`。它和 D5 的关系是:**D5 算出真相并存进 merger,config_log 只是换个排版把真相打出来。** 所以它和 D5 永远不会"对不上"(同一个数据源)。这也是为什么它能安心放在 `common/`:它是个无状态的格式化工具,D6/D7 都能用(D7 验证时也想按字段看配置)。

# 阶段 5:日志名跟训练目录对齐——log_rename + 撞墙③

伤痛 D 还剩一半没治:日志文件叫什么、跟哪次训练对得上。这里有最后一堵墙。

## 5.1 🧱 撞墙③:ultralytics 的 `save_dir` 训练前不知道

你想让日志文件名带上"这是哪次训练"的信息——比如跟 ultralytics 的输出目录同名。问题是:

```text
# 你想在【训练开始前】就把日志命名为 "train3-yolo26n.log"
# 但 "train3" 这个名字是 ultralytics 在【train() 内部】才决定的:
#   第一次跑   → runs/detect/train/
#   第二次跑   → runs/detect/train2/
#   第三次跑   → runs/detect/train3/   ← 它自己 +N,你训练前无从得知
# 于是:训练前你不知道该把日志叫什么,训练后日志又已经写了一半。
```

这是个"鸡生蛋"的顺序问题:**日志要在训练前就开始记(不然训练过程没日志),但它该叫的名字要等训练后才知道。**

## 5.2 方案:先训练、后改名

解法很直接——**让日志先用一个临时名开始记(训练过程不丢),训练拿到 `save_dir` 后,再把日志文件改名对齐。** 这需要一个工具:训练后,找到当前的文件日志,把它的文件改名为 `<save_dir 名>-<模型名>.log`。

## 5.3 `common/log_rename.py`:唯一被允许碰 handler 的工具

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : log_rename.py
# @Project   : ODPlatform
# @Function  : 训练后把日志文件改名对齐 ultralytics 的 save_dir
"""rename_log_to_save_dir():训练后把当前 FileHandler 的文件改名,与训练产物对齐。

【合法豁免声明】本模块是项目里唯一被允许操作 logging handler 的业务工具——因为它的
职责【就是】重新指向 FileHandler。它放在 common/(不在 training/),所以 training/ 的
'业务模块不碰 handler' 自检依然干净。
"""
from __future__ import annotations

import logging
from pathlib import Path
from typing import Optional


def rename_log_to_save_dir(
    save_dir: Path,
    model_stem: str,
    *,
    logger_name: str = "od_platform",
    logger: logging.Logger | None = None,
) -> Optional[Path]:
    """把 named root 上的 FileHandler 改名为 <save_dir 名>-<model_stem>.log."""
    log = logger or logging.getLogger(__name__)
    root = logging.getLogger(logger_name)            # ← named root!不是 logging.getLogger()
    for h in list(root.handlers):
        if isinstance(h, logging.FileHandler):
            old = Path(h.baseFilename)
            new = old.parent / f"{Path(save_dir).name}-{model_stem}.log"
            fmt = h.formatter
            h.flush(); h.close(); root.removeHandler(h)   # 关旧 handler
            if old.exists() and old != new:
                old.rename(new)                            # 物理改名(过程日志不丢)
            new_handler = logging.FileHandler(new, encoding="utf-8")
            new_handler.setFormatter(fmt)
            root.addHandler(new_handler)                   # 挂新 handler
            log.info(f"日志已对齐训练目录:{new.name}")
            return new
    return None
```

⚠️ **设计点(为什么找 handler 要在 named root 上找)**:注意 `logging.getLogger(logger_name)` 里那个 `logger_name="od_platform"`——它去**有名字的 root** 上找 FileHandler。如果写成 `logging.getLogger()`(无名 root),就找不到 CLI 挂的那个 handler(CLI 是通过 D2 `get_logger` 挂在 `"od_platform"` 上的)。这正是撞墙②的回声:**handler 在哪个 logger 上挂的,就得去哪个 logger 上找**,无名 root 和 named root 是两回事。

⚠️ **设计点("合法豁免"是什么意思)**:阶段 0 立的规矩一说"业务模块不碰 handler"。`log_rename` 偏偏要碰——这不是违反规矩,是**规矩明确放行的唯一例外**:它的整个职责就是"重新指向 FileHandler",不碰 handler 它什么都做不了。关键是它**放在 `common/` 而不是 `training/`**,所以"`training/` 不碰 handler"的 CI 自检照样全绿。一个例外,声明清楚、隔离干净,比"假装没有例外"诚实得多。

## 6.1 `training/archive.py`:归档到 `models/trained/`

ultralytics 把 `best.pt` / `last.pt` 丢在 `runs/.../weights/` 里。但 `runs/` 是**全部痕迹**(只增不减),你想**长期留**的那个权重,该归到 D0 定的"精选成品仓库"——`TRAINED_MODELS_DIR`(`models/trained/`):

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : archive.py
# @Project   : ODPlatform
# @Function  : 把训练产出的 best/last.pt 归档到 models/trained/(精选成品仓库)
"""archive_checkpoints():从 runs 的 save_dir 把 best/last.pt 复制到 TRAINED_MODELS_DIR。"""
from __future__ import annotations

import logging
import shutil
from pathlib import Path
from typing import Optional

from od_platform.common.paths import TRAINED_MODELS_DIR

logger = logging.getLogger(__name__)


def archive_checkpoints(save_dir: Path, *, logger_: logging.Logger | None = None) -> Optional[Path]:
    """把 best.pt(及 last.pt)复制到 models/trained/,用 run 目录名做标识防覆盖.

    返回归档后的 best.pt 路径;若 best.pt 不存在则返回 None.
    """
    log = logger_ or logger
    weights = Path(save_dir) / "weights"
    best = weights / "best.pt"
    if not best.exists():
        log.warning(f"未找到 best.pt:{best},跳过归档")
        return None

    TRAINED_MODELS_DIR.mkdir(parents=True, exist_ok=True)
    stem = Path(save_dir).name                       # 如 "train3",做归档标识
    dst_best = TRAINED_MODELS_DIR / f"{stem}-best.pt"
    shutil.copy2(best, dst_best)
    log.info(f"已归档 best.pt → {dst_best}")

    last = weights / "last.pt"
    if last.exists():
        shutil.copy2(last, TRAINED_MODELS_DIR / f"{stem}-last.pt")
    return dst_best
```

⚠️ **设计点(归档到 `TRAINED_MODELS_DIR`,而不是另造一个目录)**:训练产物的去处,D0 早就定好了——`runs/` 放每次训练的全部痕迹,`models/trained/` 放你挑出来长期用的那几个。归档这个动作("从 runs 里把成品挑出来留好")**正好就是 `models/trained/` 的语义**。所以它归到 `TRAINED_MODELS_DIR`,不需要也不该再发明一个 `CHECKPOINTS_DIR`——那会让"训练成品到底在哪"出现两个答案。**目录的语义 D0/D1 定死,D6 照用。** 而且好处立刻兑现:D7 验证时,用 `refs.resolve_trained_model` 就能从 `models/trained/` 直接找到这些归档——上下游靠同一个目录约定对齐。

## 6.2 给训练计时:`@time_it`(兑现 performance_utils)

D2 建过 `@time_it` 计时装饰器。训练是整个流程里最耗时的一步,正该给它计时。把真正调 `model.train()` 的那一小段单独抽成方法、套上 `@time_it`:

```python
from od_platform.common.performance_utils import time_it

class TrainService:
    @staticmethod
    @time_it                                    # ← 训练耗时自动打印(performance_utils 在 D6 兑现)
    def _run_training(model_arg: str, config, data_yaml) -> object:
        from ultralytics import YOLO
        model = YOLO(model_arg)
        return model.train(data=str(data_yaml), **config.to_ultralytics_kwargs())
```

⚠️ **设计点(`@staticmethod` 和 `@time_it` 的叠放顺序)**:`@staticmethod` 在**外**、`@time_it` 在**内**(贴着函数)。这样 `time_it` 先把函数包成"带计时的函数",再由 `staticmethod` 包成静态方法,计时正常生效。反过来叠会出问题。这也顺带印证了**手册②装饰器**讲的"装饰器叠放是从下往上套"。

## 6.3 完整 `training/service.py`:把所有零件编排起来

现在把前面所有东西拼成完整的 service——你会看到,它**几乎全是在调邻居**,自己只管顺序和兜底:

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : service.py
# @Project   : ODPlatform
# @Function  : TrainService —— 编排 D5 配置 + D4 校验 + D3 解析 + D2 日志/计时 + ultralytics 训练 + 训练后整理
"""训练编排器。它不发明任何东西,只把邻居们按正确顺序串起来,并兜住一切异常。"""
from __future__ import annotations

import json
import logging
from dataclasses import dataclass
from datetime import datetime, timezone
from pathlib import Path
from typing import Optional

from od_platform.common.refs import resolve_pretrained_model, resolve_yaml      # D3 接缝
from od_platform.common.system_utils import log_device_info                     # D2 环境快照
from od_platform.common.performance_utils import time_it                        # D2 计时
from od_platform.common.config_log import log_effective_config, log_override_chains
from od_platform.common.result import TrainMetrics, log_train_metrics
from od_platform.common.log_rename import rename_log_to_save_dir
from od_platform.runtime_config import build_train_config                       # D5 配置
from od_platform.data_validation import validate_dataset                        # D4 校验
from od_platform.training.archive import archive_checkpoints

logger = logging.getLogger(__name__)


@dataclass(frozen=True)
class TrainResult:
    """一次训练的结果事实(不可变载体)."""
    success: bool
    save_dir: Optional[Path]
    best_weight: Optional[Path]
    metrics: Optional[TrainMetrics]
    error: Optional[str] = None


class TrainService:
    """训练编排器:不发明,只编排;永不抛异常."""

    def train(
        self, config_path: str, data: str, *,
        cli_overrides: Optional[dict] = None,
        pre_validate: bool = True, archive: bool = True, rename_log: bool = True,
    ) -> TrainResult:
        try:
            return self._train(
                config_path, data, cli_overrides=cli_overrides,
                pre_validate=pre_validate, archive=archive, rename_log=rename_log,
            )
        except Exception as e:                          # 永不抛:兜住一切,包成失败结果
            logger.error(f"训练失败:{e}", exc_info=True)
            return TrainResult(False, None, None, None, error=str(e))

    # —— 真正的编排在这里(被 train() 包在 try 里)——
    def _train(self, config_path, data, *, cli_overrides, pre_validate, archive, rename_log) -> TrainResult:
        # 1) 配置:合并 + 验证 + 转换(走 D5,service 不重做)
        config, merger = build_train_config(config_path, cli_overrides or {})
        log_effective_config(config, merger, logger=logger)     # 按字段打印生效值+来源
        log_override_chains(config, merger, logger=logger)      # 打印完整来源链

        # 2) 数据集 yaml:解析(走 D3 refs,和 D4 同一个解析器)
        data_yaml = resolve_yaml(data)

        # 3) 训练前校验(走 D4):不过则干净停下,不开训
        if pre_validate:
            exit_code = validate_dataset(str(data_yaml))
            if exit_code != 0:
                return TrainResult(False, None, None, None,
                                   error=f"数据校验未通过(exit={exit_code}),训练未开始")

        # 4) 环境快照(走 D2):GPU/CUDA/版本,可复现
        log_device_info(logger=logger)

        # 5) 模型实参(走 D3 refs + 训练侧 local-or-download 策略)
        model_arg = self._resolve_model_arg(config.model)

        # 6) 真训练(@time_it 计时)
        started = datetime.now(timezone.utc)
        results = self._run_training(model_arg, config, data_yaml)
        ended = datetime.now(timezone.utc)

        # 7) 提取产物:save_dir + 指标
        save_dir = Path(results.save_dir)
        metrics = self._extract_metrics(results, config.task)
        log_train_metrics(metrics, logger=logger)

        # 8) 训练后整理:对齐日志名 + 归档权重
        if rename_log:
            rename_log_to_save_dir(save_dir, Path(model_arg).stem)
        best_weight = archive_checkpoints(save_dir) if archive else None

        # 9) 审计快照
        self._write_audit(save_dir, config, metrics, started, ended)

        return TrainResult(True, save_dir, best_weight, metrics)

    @staticmethod
    def _resolve_model_arg(model: str) -> str:
        p = resolve_pretrained_model(model)             # 纯解析(走 refs)
        if p.exists():
            logger.info(f"使用本地预训练权重:{p}")
            return str(p)
        logger.info(f"本地无 {p.name},交给 ultralytics 自动下载:{model}")
        return model                                    # 下载兜底是训练侧策略

    @staticmethod
    @time_it
    def _run_training(model_arg, config, data_yaml):
        from ultralytics import YOLO
        model = YOLO(model_arg)
        return model.train(data=str(data_yaml), **config.to_ultralytics_kwargs())

    @staticmethod
    def _extract_metrics(results, task: str) -> TrainMetrics:
        rd = getattr(results, "results_dict", {}) or {}
        box = getattr(results, "box", None)
        g = lambda a, k, d=0.0: float(getattr(box, a, None) or rd.get(k, d)) if box else float(rd.get(k, d))
        return TrainMetrics(
            task=task,
            fitness=float(getattr(results, "fitness", 0.0) or rd.get("fitness", 0.0)),
            map50=g("map50", "metrics/mAP50(B)"),
            map50_95=g("map", "metrics/mAP50-95(B)"),
            precision=g("mp", "metrics/precision(B)"),
            recall=g("mr", "metrics/recall(B)"),
        )

    @staticmethod
    def _write_audit(save_dir, config, metrics, started, ended) -> None:
        audit = {
            "kind": "train",
            "started_utc": started.isoformat(),
            "ended_utc": ended.isoformat(),
            "elapsed_sec": round((ended - started).total_seconds(), 2),
            "config": config.to_audit_snapshot(),       # D5:脱敏快照(敏感字段已 mask)
            "metrics": {
                "fitness": metrics.fitness, "map50": metrics.map50,
                "map50_95": metrics.map50_95, "precision": metrics.precision, "recall": metrics.recall,
            },
            "save_dir": str(save_dir),
        }
        out = Path(save_dir) / "odp_audit.json"
        out.write_text(json.dumps(audit, ensure_ascii=False, indent=2), encoding="utf-8")
        logger.info(f"审计快照已写入:{out}")


def train_yolo(
    config_path: str, data: str, *,
    cli_overrides: Optional[dict] = None,
    pre_validate: bool = True, archive: bool = True, rename_log: bool = True,
) -> TrainResult:
    """便捷函数:内部就是 TrainService().train(...),不重复任何逻辑."""
    return TrainService().train(
        config_path, data, cli_overrides=cli_overrides,
        pre_validate=pre_validate, archive=archive, rename_log=rename_log,
    )
```

停下来数一眼这段 `_train` 的每一步注释:**配置→D5、yaml 解析→D3、校验→D4、环境→D2、模型解析→D3、训练→ultralytics、计时→D2**。service 自己写的逻辑,只有"按这个顺序调""失败包成 TrainResult""写 audit"——这就是编排器的样子。

🔗 **概念手册**:`build_train_config` 返回的 config 是 Pydantic 模型、`to_ultralytics_kwargs` / `to_audit_snapshot` 都是它的方法——配置怎么被消费,见**手册④ Pydantic v2**。

# 阶段 7:CLI 入口——odp-train

service 能跑了,给它配个命令行入口。**CLI 只做三件事:解析参数、挂日志、调 service。** 任何业务逻辑都不在这里。

## 7.1 `cli/train_model.py`

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : train_model.py
# @Project   : ODPlatform
# @Function  : odp-train entry-point —— YOLO 模型训练命令行入口
"""CLI 极薄:解析参数 → 挂日志(唯一 handler 职责)→ 调 service → 按结果返回退出码。"""
from __future__ import annotations

import argparse
import sys

from od_platform.common.logging_utils import get_logger
from od_platform.common.paths import LOGGING_DIR
from od_platform.training import train_yolo


def main() -> int:
    parser = argparse.ArgumentParser(prog="odp-train", description="训练 YOLO 模型")
    parser.add_argument("--config", required=True, help="运行配置 yaml(D5 train.yaml)")
    parser.add_argument("--data", required=True, help="数据集名 或 yaml 路径")
    parser.add_argument("--no-validate", action="store_true", help="跳过训练前数据校验")
    parser.add_argument("--no-archive", action="store_true", help="不归档权重到 models/trained/")
    parser.add_argument("--epochs", type=int, default=None, help="覆盖 epochs(其余覆盖走 yaml/CLI)")
    parser.add_argument("--batch", type=int, default=None, help="覆盖 batch")
    args = parser.parse_args()

    # CLI 唯一的 handler 职责:调 D2 get_logger 挂在 named root(撞墙②的正解)
    LOGGING_DIR.mkdir(parents=True, exist_ok=True)
    get_logger(log_file=LOGGING_DIR / "train.log")

    cli_overrides = {
        k: v for k, v in (("epochs", args.epochs), ("batch", args.batch)) if v is not None
    }

    result = train_yolo(
        args.config, args.data,
        cli_overrides=cli_overrides or None,
        pre_validate=not args.no_validate,
        archive=not args.no_archive,
    )

    # 规矩二:fail-fast 用退出码,不抛异常。成功 0,失败非 0。
    return 0 if result.success else 1


if __name__ == "__main__":
    sys.exit(main())
```

⚠️ **设计点(CLI 为什么这么薄)**:整个 `main()` 没有一行训练逻辑——它解析参数、挂日志、把活全交给 `train_yolo`,最后把 `result.success` 翻译成退出码。**业务逻辑全在 service,CLI 只是"把命令行翻译成一次 service 调用"。** 这样同一套训练能力,CLI 能用、notebook 能用、未来 web 后端也能用(各自写各自的薄入口),而核心逻辑只有一份。

# 阶段 9:ADR-006——训练子系统架构决策记录

把今天的关键决策记成一份 ADR,留给未来的你和接手的人。

```markdown
# ADR-006:training 子系统 —— TrainService 编排器与"复用而非重造"

## 1. 背景与问题
D5 产出了"怎么跑"的配置,但配置自己不会训练。把训练真跑起来,需要把
D5 配置、D4 校验、D3 路径解析、D2 日志、ultralytics 训练、训练后整理串成
一条流水线。问题是:这条流水线该自己写多少、复用多少?

## 2. 决策
2.1 用【编排器】而非【包装器】:TrainService 只负责调用顺序、异常兜底、审计,
    不重做任何邻居已有的能力。service.train() 永不抛异常,结果包进 frozen
    TrainResult 返回。
2.2 路径解析【全部复用 D3 refs】:训练模型走 refs.resolve_pretrained_model,
    数据集 yaml 走 refs.resolve_yaml(与 D4 同一个)。不自写 model_path /
    dataset_path —— refs 在 D3 §8.1 就为此留好了接缝。
2.3 训练侧的"本地有就用、没有让 ultralytics 下载"作为【训练专属策略】留在
    service,不塞进 refs(保持 refs 纯语法、不嗅探;验证侧不要此兜底)。
2.4 训练产物归档到 TRAINED_MODELS_DIR(models/trained/),不发明
    CHECKPOINTS_DIR —— 沿用 D0 决定 5:runs/ 是全部痕迹,models/trained/ 是
    精选成品。
2.5 logging:CLI 入口挂 handler(get_logger,named root),业务模块只发声;
    log_rename 是唯一被允许碰 handler 的工具(合法豁免,隔离在 common/)。
2.6 用 performance_utils @time_it 给训练计时(兑现 D2 工具在 D6 的归宿)。

## 3. 不选择的方案
3.1 包装器模式:只给 model.train 套层皮 —— 治不了配置溯源/校验/整理任何伤痛。
3.2 自写 resolve_model_path(model, *, search_dirs=...):用一个函数 + 参数干
    训练/验证两个场景 —— 正是 D3 金句点名要避免的"一个 resolver 靠猜",且与
    refs 重复。已弃。
3.3 单造 CHECKPOINTS_DIR 目录:与 D0/D1 的 TRAINED_MODELS_DIR 语义重叠,会让
    "训练成品在哪"出现两个答案。已并入 TRAINED_MODELS_DIR。
3.4 service 自持 FileHandler:任何二次实例化都会重复日志,违反"handler 生命周期
    只属于入口"。已弃。

## 4. 后果
- 正面:新增代码极少(仅 service / archive / 3 个 common 工具 / CLI);训练、验证、
  推理共享同一套路径解析(refs)和目录约定(models/trained/),上下游零漂移。
- 代价:service 依赖较多邻居,需靠边界 CI(grep service.py 无 D5 内部零件)守住。
- D7 受益:evaluation 用 refs.resolve_trained_model 即可从 models/trained/ 找到
  D6 归档的权重;且不需 import training。
```

## 一次撞墙,回看

| 撞的墙 | 朴素方案的痛 | 逼出的结构 |
|---|---|---|
| 🧱 撞墙① | 配置溯源黑洞 / 脏数据直接开跑 / 日志乱挂 / 跑完一地鸡毛 | 整个 TrainService 编排器(后续每阶段各治一痛) |
| 🧱 撞墙② | `getLogger()` 拿的是无名 root,和业务的 named root 对不上 | CLI 挂 handler、业务只发声;named root 统一 |
| 🧱 撞墙③ | ultralytics `save_dir` 训练前未知,日志没法提前命名 | log_rename:先训后改名对齐(合法豁免碰 handler) |

每一块结构,都是被一个具体的痛逼出来的——这正是"撞墙法"要给你的:你不只是会用 TrainService,你知道它每一部分**为什么**存在。

## 复用清单:这一篇真正的主角

| 能力 | 复用了谁 | 来自 |
|---|---|---|

> **金句**:**"D6 教的不是'怎么调 model.train'。它教的是:一个成熟的子系统,绝大部分能力来自'认得邻居并正确复用',而不是'什么都自己造'。你写的代码越少、复用得越准,系统越稳。"**

## D6 给 D7 的接口承诺

D7(评估子系统)接手时,能直接享受 D6 立好的这些约定:

- **找权重**:D6 归档到 `models/trained/`,D7 用 `refs.resolve_trained_model` 就能找到——上下游靠同一个目录约定对齐,不用商量。
- **复用工具**:`result.py`(`TrainMetrics`,D7 会起别名 `ValMetrics`)/ `config_log.py` / `plot_style.py` / `log_rename.py` 全在 `common/`,D7 直接 import,**不从 `training/` 拿**。
- **边界契约**:`evaluation/` 代码里不会出现 `from od_platform.training import ...`(单向依赖,见规矩三)。D7 会用一条 CI grep 守住它。

## 收尾

你现在手里的 `training/` 子系统,能做朴素脚本做不到的事:**配置全程可溯源、脏数据开训前被拦、训练有计时、产物归档进精选仓库、全程有审计、异常永不甩给调用方。** 更重要的是——它新增的代码极少,绝大部分能力来自正确复用 D1–D5 的邻居。

下一天(D7),我们做**评估子系统**。你会看到 D6 立的每一个约定如何兑现成 D7 的"免费午餐":refs 的另一个解析器、common 里的那几个工具、`models/trained/` 的目录约定——D7 几乎不用造新东西,只需把它们按"评估"的顺序再编排一次。

> **金句**:**"好架构的回报,藏在'下一个子系统几乎免费'里。D6 把地基复用对了,D7 才能站上去几乎不费力——这,就是'复用'两个字真正的分量。"**
