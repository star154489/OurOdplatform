# 阶段 0: 立规矩(paths.py 增量)

D5 第一件事,和 D3/D4 一样:**立项目基线**——但不写一行业务代码。

D5 要立的规矩,核心只有一条:**运行配置文件目录,从 `configs/` 里拆出来,放 `configs/runtime/`。**

## 0.1 为什么要把 `configs/runtime/` 单拆出来?

D3 立过一个 `configs/datasets/`,装的是"描述数据的 YAML"——`safety_helmet.yaml`、`rsod.yaml`,每份写的是 `path / train / val / nc / names`,由 `odp-transform` 产出,被 ultralytics 直接消费。

D5 要立的"运行配置",装的是"描述跑法的 YAML"——`train.yaml`、`val.yaml`,每份写的是 `epochs / lr0 / batch / optimizer`,由 `odp-gen-config` 产出,被 D6 的 TrainService 加载后转成 ultralytics 参数。

两件事语义完全不同:

| | `configs/datasets/`(D3) | `configs/runtime/`(D5) |
|---|---|---|
| 描述什么 | 【数据】长什么样 | 【跑法】是什么 |
| 谁产出 | `odp-transform` | `odp-gen-config` 或手写 |
| 谁消费 | ultralytics 直接吃 | TrainService 加载 → 转 ultralytics 参数 |
| 改频率 | 数据集做完基本不动 | 调参时一直在改 |
| 进 git 吗 | ✅ 进(数据契约的一部分) | ✅ 进(实验可复现的一部分) |

⚠️ **设计点(物理分目录不是"为了好看")**:当用户问"我把 `mosaic: 0.5` 写错了,改 yaml 还是改哪儿?",一句"改 `configs/runtime/train.yaml`"就解决——**目录名本身就是路径上的文档**。要是两类 yaml 混在一个目录,你在里头 grep `safety_helmet`(数据集名)和 `safety_helmet`(有人拿它当实验名)就分不清谁是谁。

> **金句**:**"目录名是路径上最便宜也最显眼的文档——能用目录回答的问题,别留给 README。"**

## 0.2 paths.py 增量(本节只新增 1 个常量 + 1 个函数)

回到 D1 立的铁律:**任何路径只从 `paths.py` 来。** 所以 `configs/runtime/` 这条路径,也得在 `paths.py` 里有个正经的名字。

打开 `apps/platform/src/od_platform/common/paths.py`,末尾追加:

```python
# ============================================================
# D5 增量: 运行配置目录 (跟 D3 立的 DATASET_CONFIGS_DIR 语义完全不同)
#   DATASET_CONFIGS_DIR: 描述【数据】的 yaml (odp-transform 产出)
#   RUNTIME_CONFIGS_DIR : 描述【跑法】的 yaml (runtime_config 子系统产出)
# 两类 yaml 物理分目录, 语义不会互相污染.
# ============================================================
RUNTIME_CONFIGS_DIR: Path = CONFIGS_DIR / "runtime"


def runtime_config_path(name: str) -> Path:
    """返回某个运行配置文件的路径: <CONFIGS_DIR>/runtime/<name>.yaml

    Args:
        name: 配置名 (如 "train" / "val" / "infer"), 不带 .yaml 后缀

    Returns:
        Path 对象 (尚未创建, 调用方自己负责 mkdir)

    用法:
        train_yaml = runtime_config_path("train")
        # → <APP_DIR>/configs/runtime/train.yaml
    """
    return RUNTIME_CONFIGS_DIR / f"{name}.yaml"
```

再把 `RUNTIME_CONFIGS_DIR` 加进 `get_dirs_to_initialize()` 的返回列表,让 `odp-init` 顺手把这个空目录建好:

```python
def get_dirs_to_initialize() -> List[Path]:
    return [
        # ... D1/D2/D3/D4 已有目录 ...
        DATASET_CONFIGS_DIR,    # D3
        RUNTIME_CONFIGS_DIR,    # ★ D5 新增
    ]
```

⚠️ **设计点("启动建"还是"按需建", 跟 D4 的判断不一样)**:D4 的 `runs/data_validation/` 是**按需建**——它不存在本身就是信息("还没跑过验证")。但 D5 的 `configs/runtime/` 是**启动建**——它是"项目结构的骨架",和 `configs/datasets/` 一样,用户哪天会用,先建好不冤枉,空着也没坏处,还能让用户一眼看到"配置该放这儿"。两条规则记住:**运行结果目录 → 按需建;配置 / 数据骨架目录 → 启动建。**

⚠️ **设计点(不用动 `PROTECTED_DIRS`)**:D2 把 `CONFIGS_DIR` 放进了 `odp-reset` 的 `PROTECTED_DIRS`(配置是手写资产,reset 不能删)。`RUNTIME_CONFIGS_DIR` 是它的子目录,`is_protected()` 检查祖先链时自动覆盖——**不用重新登记**。这正是 D2 那条"保护按祖先链继承"设计在今天白捡的便宜。

## 0.3 task 取值复用 D3 的 `Task`(别另起炉灶)

D5 的训练 / 验证配置都有一个 `task` 字段,合法取值是 `detect` / `segment`。这个词 D3 早就立过了:

```python
from od_platform.common.constants import Task
# Task.DETECT  -> "detect"
# Task.SEGMENT -> "segment"
# Task.all()   -> 合法取值集合, 给 validator 用
```

⚠️ **陷阱预警(跟 D4 阶段 0 同款警告再来一次)**:不要在 `runtime_config/` 内部立第二份 task 字符串集。反模式:

```python
# ❌ 反模式 —— 在 runtime_config/train.py 里又立一份
class TrainTask:
    DETECT  = "detect"
    SEGMENT = "segment"
```

这一立,立刻冒出"`Task.DETECT == TrainTask.DETECT` 吗?"的问号——SSoT 当场就破。**以后凡是遇到"我需要任务类型"的需求,第一反应永远是 `from od_platform.common.constants import Task`**,而不是手敲一个字符串。

> **金句**:**"SSoT 不是写在 README 里的承诺, 是写在 import 里的事实——能复用一份真相的, 永远别抄第二份。"**

## 0.4 立两条贯穿全篇的公约

阶段 0 顺手把**两条**会贯穿整个子系统的工程纪律立起来。立得早,后面每写一行业务代码都自动合规,不必回头加固。

**公约一:路径只从 `paths.py` 来。** `runtime_config/` 内任何路径,都走 `paths.py` 的常量或函数,禁止以下写法:

```python
# ❌ 反例 1: 绝对路径字符串(跨平台炸雷)
temp_dir = Path(r"C:\Users\xxx\Desktop\yolo") / "configs"

# ❌ 反例 2: 相对路径硬编码
yaml_path = "configs/runtime/train.yaml"

# ❌ 反例 3: 业务模块里自己拼 APP_DIR
yaml_dir = APP_DIR / "configs" / "runtime"
```

```python
# ✅ 正例: 一律走 paths.py
from od_platform.common.paths import RUNTIME_CONFIGS_DIR, runtime_config_path

yaml_path = runtime_config_path("train")    # → <APP_DIR>/configs/runtime/train.yaml
yaml_dir  = RUNTIME_CONFIGS_DIR
```

这条规矩落在 D2 立的 ADR-001(路径 SSoT)上。跨平台、mock 测试、CI 跑多操作系统——这些痛点全因为这一条自动消解。

**公约二:每个 `runtime_config/` 文件首行 `from __future__ import annotations`,类型注解用 PEP 585/604 风格。**

```python
from __future__ import annotations    # ← 每个文件第一行

# 之后注解一律现代写法:
def merge(self, config_class: type[T], *,
          sources: list[tuple[str, dict]] | None = None) -> T:
    ...
```

⚠️ **设计点(`from __future__ import annotations` 为什么必须有)**:没这行,`list[int]` 这种内置泛型注解要 Python 3.10+ 才能在运行时写;加上它,所有注解被"字符串化",任何 3.7+ 都能写,而 mypy / Pydantic 仍能正常解析。**代价 0,收益是注解能往未来看齐。** 配套 ruff 开 `UP`(pyupgrade)规则,谁写了老式 `List[int]` 直接 CI 拦下。

> 至于"用户警告该走哪个渠道""命令入口怎么注册"——这些**不在阶段 0 预立**。它们各自要等后面一个具体需求(阶段 5 第一次发警告、阶段 6 第一次想要一条干净命令)撞上来时,才登场。**没撞到的规矩,先不立。**

## 1.1 任务设定:复刻你的 legacy `yolo_train.py`

大概率你电脑上已经有一份这种训练脚本(讲义资料包里的 `yolo_train.py` 就是)。它长这样——一个大 `argparse`,把 ultralytics 的几十个参数一个个 `add_argument`:

```python
# 朴素第一版(几乎 1:1 复刻 legacy yolo_train.py —— 反面教材!)
import argparse
import yaml
from ultralytics import YOLO


def parse_args():
    p = argparse.ArgumentParser()

    # ---- 核心 ----
    p.add_argument("--model", default="yolo26n.pt")
    p.add_argument("--data",  default="data.yaml")
    p.add_argument("--epochs", default=100, type=int)
    p.add_argument("--batch",  default=16,  type=int)
    p.add_argument("--imgsz",  default=640, type=int)
    p.add_argument("--device", default=None)

    # ---- 优化器 ----
    p.add_argument("--lr0", default=0.01, type=float)
    p.add_argument("--lrf", default=0.01, type=float)
    p.add_argument("--momentum", default=0.937, type=float)
    p.add_argument("--weight_decay", default=0.0005, type=float)
    p.add_argument("--optimizer", default="auto")
    p.add_argument("--cos_lr", action="store_true")

    # ---- 数据增强(颜色 / 几何 / 拼接) ----
    p.add_argument("--hsv_h", default=0.015, type=float)
    p.add_argument("--hsv_s", default=0.7,   type=float)
    p.add_argument("--hsv_v", default=0.4,   type=float)
    p.add_argument("--degrees",   default=0.0, type=float)
    p.add_argument("--translate", default=0.1, type=float)
    p.add_argument("--scale",     default=0.5, type=float)
    p.add_argument("--fliplr",    default=0.5, type=float)
    p.add_argument("--mosaic",    default=1.0, type=float)
    p.add_argument("--mixup",     default=0.0, type=float)
    p.add_argument("--close_mosaic", default=10, type=int)

    # ---- 损失权重 ----
    p.add_argument("--box", default=7.5, type=float)
    p.add_argument("--cls", default=0.5, type=float)
    p.add_argument("--dfl", default=1.5, type=float)

    # ---- 输出 ----
    p.add_argument("--project", default=None)
    p.add_argument("--name",    default=None)
    p.add_argument("--exist-ok", action="store_true")

    # ---- 可选 YAML 配置 ----
    p.add_argument("--config", default=None, help="可选 YAML, CLI 优先级最高")

    # ... 还有 20+ 个参数没列 ...
    return p.parse_args()


def main():
    args = parse_args()

    # 有 --config 就拿 YAML 当默认值
    if args.config:
        with open(args.config) as f:
            yaml_cfg = yaml.safe_load(f) or {}
    else:
        yaml_cfg = {}

    # 合并: CLI 覆盖 YAML 覆盖 default
    merged = {**yaml_cfg, **{k: v for k, v in vars(args).items() if v is not None}}
    merged.pop("config", None)

    # 跑
    model = YOLO(merged.pop("model"))
    model.train(**merged)


if __name__ == "__main__":
    main()
```

## 1.3 🧱 撞墙①:朴素方案的四个伤痛

跑完上面这版,我们诚实地数一下它的毛病。每数出一个,**先别急着想"该用什么修",先把"它到底缺了什么"说清楚**——缺口说准了,解药自然就有形状了。

### 伤痛①:一个字段,三处定义

假设你想加一个新参数 `freeze`(冻结前 N 层)。你得改三个地方:

1. argparse 里加一行:`p.add_argument("--freeze", default=None, type=int)`
2. YAML 模板里加一行:`freeze: null`
3. 翻 ultralytics 文档确认:它接受的参数是不是真叫 `freeze`?

```
┌─────────────────────────────────────────────┐
│  argparse:  --freeze  default=None  type=int │  ← 真相 A
│  YAML:      freeze: null                     │  ← 真相 B
│  ultralytics 接受的参数名里有没有它?         │  ← 真相 C
└─────────────────────────────────────────────┘
                  ↓
       三个真相, 任何一个漂移 = bug
```

**三处改动 + 一次查文档。** 漏一处就跑不通——还不一定立刻发现,可能跑到训练中途,ultralytics 报个 `unknown argument 'freez'`,你才知道是拼错了。

这是 D2 立"路径 SSoT"、D3 立"词汇 SSoT"之后,我们撞到的**第三种 SSoT 失守**:一个字段的"名字 / 类型 / 默认值 / 帮助文档",在这版里有**三份真相**散在三个文件里。

> **缺口①**:我们需要让"一个字段的全部信息"只有**一份**真相——定义一次,argparse、YAML、文档全从它派生。

### 伤痛②:没有验证,错的输入静默通过

朴素方案完全信任用户。看几个真实的坑:

```bash
# 非 32 倍数的 imgsz
python /tmp/yolo_train_naive.py --imgsz 649
# → ultralytics 自动补到 672, 但不告诉你. 你以为跑的 649, 实际 672.

# 零批
python /tmp/yolo_train_naive.py --batch 0
# → 某处 "ZeroDivisionError", 栈里完全看不出是 batch=0 引起的.

# 拼错 device
python /tmp/yolo_train_naive.py --device cpuu
# → PyTorch 报 "expected device, got cpuu", 用户花 5 分钟才反应过来是自己手滑.
```

这些都是**"在入口就该拒"的输入,不是"跑到一半才崩"的故障**。验证理应发生在"数据进配置对象的那一刻",并且报错要精确指出:**哪个字段、哪个值、为什么不合法**——而不是延迟到 ultralytics 内部、或者 PyTorch 栈里,让你猜。

> **缺口②**:我们需要在"输入变成配置"的入口处做验证,而且校验规则最好就**贴在字段定义旁边**,错了能精确定位。

### 伤痛③:跑完,不知道 `lr0=0.001` 是谁给的

```bash
python /tmp/yolo_train_naive.py --config train.yaml --lr0 0.001 --epochs 200
# 日志: lr0=0.001, epochs=200, batch=16, optimizer=auto, ...
```

跑完看日志,一片**"它就是这样"**——没人告诉你:`lr0=0.001` 和 `epochs=200` 是你刚敲的 CLI 给的,`batch=16` 是 YAML 给的,`optimizer=auto` 是默认值。

三个月后你回看那行日志,想复现这次跑法——"我当时 CLI 敲了啥?YAML 改过哪几个字段?"——**猜不出来**。更糟的是,实习生跟你说"师兄这个 `lr0` 不对劲,我改成 0.005 了",你打开 yaml 一看确实是 0.005,但**到底是他改了 yaml,还是他在 CLI 里加了 `--lr0 0.005`?** 你俩谁也说不清。

朴素那行 `merged = {**yaml_cfg, **cli}` 把一切压成了一个扁平 dict:**信息只剩"最后赢的那个值",谁覆盖了谁、原值是啥、来源是谁,全丢了。**

> **缺口③**:配置不该只是"参数表",得是"决策记录"——每个最终生效的值,都该带一条"我是谁、从哪来、覆盖了谁"。

### 伤痛④:YAML 模板手维护,"配置即文档"失效

你给学员一份 `train.yaml`:

```yaml
mosaic: 1.0
mixup: 0.0
hsv_h: 0.015
```

学员第一反应:"`mosaic: 1.0` 是什么?范围 0–1 还是 0–10?调到多少合适?"——**没注释,等于没文档。** 于是你只能手写注释:

```yaml
mosaic: 1.0      # Mosaic 增强概率(0-1, 1.0 = 总是开启)
mixup: 0.0       # Mixup 增强概率(0-1, 0 = 禁用)
hsv_h: 0.015     # HSV 色调扰动强度(0-1)
# ... 手写 60 行下来 ...
```

然后过两周加了个新参数,忘改这份模板——**注释和代码漂移**。又是经典的"两份真相"。

> **缺口④**:字段的元数据(范围、含义、示例)就该跟字段定义在**同一处**,YAML 模板由代码从它自动产出,而不是手写、手维护。

## 1.4 四个缺口摞在一起,答案才浮出来

停一下,把四个缺口并排放:

| 伤痛 | 缺口 |
|---|---|
| ① 字段三处定义 | 字段的全部信息只有一份真相 |
| ② 无验证 | 在入口验证 + 校验贴着字段、报错能定位 |

Python 里正好有个库,就是为"装不可信输入、带验证、带元数据"这个场景生的——**Pydantic**。下一阶段,它登场,我们就从这四个缺口里,把第一块结构 `BaseConfig` 长出来。

那缺口 ③(溯源)呢?它要的不是"装字段",是"记录合并过程"——这是另一块结构,得等配置真的开始"从外部多个源进来、按优先级打架"时才撞得到。所以**现在先不碰它**,留到它自己浮出来的那一阶段。我们不一次性把所有解法摆出来——**那正是这门课要避免的事**。

⚠️ **设计点(为什么不一次给全)**:你可能已经隐约猜到后面会有"合并器""生成器"。**忍住别现在写。** 一个结构只有在"不写它就会真出事"的那一刻动手,你才会记住它为什么存在;提前给,它就退化成"看图填空"。这也是为什么阶段 1 只摊问题、只解决前两个缺口里那块最先撞到的——其余的,等被撞到。

> **金句**:**"朴素方案不是错, 是它把伤痛延迟到了运行时才爆。工程化的本质, 是把痛苦提前到设计阶段——但一次只提前一步, 别把整本答案先抄给自己。"**

# 阶段 2: BaseConfig —— Pydantic 入场,只装"共享字段 + 验证"

## 2.1 四个缺口里,先动手解决前两个

回到阶段 1 结尾那张表。四个缺口里,**① 字段唯一真相** 和 **② 入口验证** 要的其实是同一种东西:一个能在"定义字段"这一处,同时把**类型、默认值、校验规则**装下的载体;而且这个载体面对的是 YAML / CLI 这种**不可信输入**,得能验证、能友好报错、能从 dict 反序列化成对象。

这几样能力,用 Python 标准库的 `dataclass` 行不行?能,但每一样都得自己写:`__post_init__` 里手写校验、手写类型检查、手写从 dict 还原。而有个库,**这些全是它生来就有的**——它就是为"装不可信输入、带验证"这个场景做的:**Pydantic**(v2)。

所以这一阶段我们不"选型辩论",直接动手:用 Pydantic 写出共享配置 `BaseConfig`,把缺口 ① ② 一次填上。至于缺口 ③(溯源)和 ④(模板),要等配置真的"从多个源进来打架""要生成给人看的模板"时才撞得到——**现在不碰**。

⚠️ **依赖小坑(Pydantic v2)**:本讲义所有代码基于 **Pydantic v2**(`BaseModel` / `field_validator` / `model_validator` 的 API 跟 v1 差别很大)。在 `apps/platform/pyproject.toml` 的 `dependencies` 里加上,然后 `pip install -e ./apps/platform` 让它生效:

```toml
dependencies = [
    # ... D1-D4 已有 ...
    "pydantic>=2.5,<3",     # ★ D5 新增
    "pyyaml>=6.0",          # ★ D5 新增(阶段 4 loaders 用)
]
```

⚠️ **设计点(不要为了"和前几天一致"强行 dataclass)**:D2/D3/D4 装的 `DatasetSnapshot` 之类用的是 `@dataclass(frozen=True)`——那些是**程序内部产出**,自己生自己用,类型基本不会错,写校验是过度工程。D5 装的是**用户手写的输入**,必须验证。**SSoT 是"一种问题一个解法",不是"全项目所有数据类长一个样"**——用错才丢一致性,用对了不是。

## 2.2 BaseConfig 装哪些字段?先把判据立死

不要打开编辑器就开始码。先想清楚:哪些字段属于"共享层"?

**判据(两条都满足才进 BaseConfig)**:① 训练 / 验证 / 推理**都用**;② 语义在三种模式下**基本一致**。差一条都不进,丢给子类。

照这个判据筛 ultralytics 的参数,共享的就 15 个,分 5 组:

```
BaseConfig(共享层)
├── 核心参数: model, data
├── 输入配置: batch, imgsz, workers, cache, rect
├── 设备配置: device, amp
├── 输出配置: project, name, exist_ok, save
└── 基础设置: verbose, seed, deterministic
```

**少而精**——再多就该往子类挪了。

⚠️ **设计点(反例帮你校准判据)**:`conf`(置信度阈值)——验证 / 推理用,训练**不用**,违反判据 ①,不进 BaseConfig。`task`(detect/segment)——三模式都有,但训练拿它"决定加载哪个模板"、验证拿它"只组织日志"(`.val()` 从权重自动推断),消费方式不同,**故意不放共享层**(下一阶段拆 task 时还会再碰它)。`batch`——三模式都用,训练时是梯度批、验证时是吞吐批,语义足够近,放 BaseConfig,由子类的字段说明各自补语境。**"看起来都叫一个名,但消费方式不同"的字段,不该放共享层。**

> **金句**:**"一个字段该不该进共享层,问两句:三种模式都用吗?语义一样吗?差一句,它就是子类的字段,不是基类的。"**

## 2.3 验证:Pydantic 的两层武器

缺口 ② 要的是"在入口验证 + 报错能定位"。Pydantic 给两层武器:

- **层 1 —— `Field(ge=, le=, gt=)`**:数值范围、字符串长度这类简单约束,直接写在字段定义里。
- **层 2 —— `@field_validator` / `@model_validator`**:复杂规则、跨字段逻辑。

挑三个真有讲究的看。

### 例 1:imgsz 不是 32 倍数 —— 该警告,不该 raise(顺带:警告走哪个渠道)

```python
@field_validator("imgsz")
@classmethod
def _validate_imgsz(cls, v: int) -> int:
    if v % 32 != 0:
        warnings.warn(
            f"imgsz={v} 不是 32 的倍数. 虽然能跑, 但建议用 32 的倍数"
            f"(如 640, 672, 704)以获得最佳性能.",
            UserWarning,
        )
    return v
```

⚠️ **设计点(warn 还是 raise 的判据)**:ultralytics 会自动把 `imgsz=649` 补到 672——所以它**能跑**,只是用户跑的实际不是 649。这种"能跑但跟预期不一致"该**警告**:raise 会让"我就想试 649"的人骂街,warn 既提醒又不挡路。判据:**"我拒绝继续、你必须改" → raise;"我提示你、你自己定" → warn。**

到这儿,我们第一次真发了一个警告——正好把阶段 0 **故意没预立**的那条规矩补上:

⚠️ **设计点(警告走 `warnings.warn`,不走 `logger`)**:这条提示是给**用户**看的"你这个输入有点怪",不是给运维看的"系统干了件什么事"。两类提示必须分渠道:**用户输入相关 → `warnings.warn(..., UserWarning)`**(用户能用 `-W` 控制、测试能 `catch_warnings` 接);**系统行为(编码 fallback、自动备份)→ `logger.warning`**(进 ELK)。混用之后,CI 想用 `python -W error::UserWarning` 把"配置质量警告"升成错误来强制审查,结果连"UTF-8 解码失败"这种系统降级也跟着升级——`-W` 这件武器就废了。**这条规矩不在阶段 0 预立,是因为没警告时立它是空的;现在撞到第一个警告,它才有意义。**

### 例 2:batch 既能是 int 也能是 float,还得防住 bool

`batch` 可以是 `int`(固定批 / `-1` 自动)或 `float`(显存利用率,如 `0.7`)。但有个隐蔽的坑:

```python
@field_validator("batch", mode="before")     # ★ 注意 mode="before"
@classmethod
def _validate_batch(cls, v: Any) -> Union[int, float]:
    if isinstance(v, bool):                   # bool 是 int 子类, 单独防
        raise TypeError("batch 不能是 bool 类型")
    if isinstance(v, int):
        if v == -1:
            return v                          # 自动批次
        if v <= 0:
            raise ValueError("batch 为 int 时必须为 -1(自动) 或 >= 1")
        return v
    v_float = float(v)                        # float: 显存利用率
    if not (0.0 < v_float <= 1.0):
        raise ValueError(
            f"batch 为 float 时必须在 (0, 1] 内, 表示显存利用率(如 0.7 = 70%), "
            f"当前值: {v_float}"
        )
    return v_float
```

⚠️ **设计点(为什么单独防 bool)**:Python 里 `isinstance(True, int)` 是 `True`(bool 是 int 子类)。不防的话,用户在 YAML 里写 `batch: true`(解析成 Python `True`),走进 `isinstance(v, int)` 分支,`True == -1` 假、`True <= 0` 假,**静默当成 1 用**——这到底是"显存 100%"还是"批大小 1"?都不该是,直接 raise 把错糊用户脸上。

⚠️ **设计点(为什么是 `mode="before"`)**:`field_validator` 默认 `mode="after"`,在 Pydantic 完成类型转换**之后**才跑。但 `batch` 的类型是 `Union[int, float]`,Pydantic 很可能先把 `True` 归一成 `1`,等校验器拿到手 `isinstance(v, bool)` 已经是 `False`——上面那道 bool 守卫**形同虚设**。改成 `mode="before"`,校验器拿到的是**用户写进 YAML 的原始值**,`True` 还是 `True`,守卫才拦得住。**判据:校验逻辑依赖"原始输入长什么形态"时用 `before`;只在"已规范化的值"上做范围/跨字段检查时用 `after`。**

### 例 3:跨字段验证 + 第三道闸门 `extra="forbid"`

跨字段校验用 `@model_validator(mode="after")`(此时各字段已规范化,安心比较):

```python
@model_validator(mode="after")
def _cross_field_validation(self) -> "BaseConfig":
    # workers 远大于 batch 是资源浪费(警告不阻止)
    if isinstance(self.batch, int) and self.batch > 0:
        if self.workers > self.batch * 2:
            warnings.warn(
                f"workers={self.workers} 远大于 batch={self.batch}, 可能浪费资源. "
                f"建议 workers <= batch * 2", UserWarning,
            )
    return self
```

⚠️ **设计点(守卫表达式永远先想边界)**:为什么 `batch=-1` / float 时跳过这个检查?`batch=-1` 时 `batch*2 = -2`,`workers > -2` 永远 True,会误报;`batch=0.7`(float)时 `self.batch > 0` 守卫挡掉——float 模式下 workers 跟显存利用率比没意义。**跨字段校验最常踩的坑就是没想边界。**

字段级、跨字段管的都是"值对不对"。还有一类错它们都管不到——**字段名本身拼错了**。用户把 `epochs` 敲成 `epchs: 300`,Pydantic v2 **默认 `extra="ignore"`** 会把这个多出来的键**静默丢弃**,`epochs` 仍是默认 100。用户以为跑了 300 轮,实际 100——这是阶段 1 伤痛② 的翻版,而且更隐蔽:值没错,名字错了,连校验器都没机会介入。

一行模型级配置堵死:

```python
from pydantic import BaseModel, ConfigDict

class BaseConfig(BaseModel):
    model_config = ConfigDict(extra="forbid")   # 遇到未定义字段直接 raise
    ...
```

⚠️ **设计点(为什么 runtime_config 适合 `forbid`)**:`forbid` 不是哪都该开。它适合**字段集封闭**的场景——YOLO 的训练/验证参数就是 ultralytics 定死的那批,不存在"用户要往配置里塞自定义键"的合理需求。**字段集封闭 + 输入不可信 = `forbid` 的主场**,两条 runtime_config 都占。反过来,本来就允许动态扩展字段的模型开 `forbid` 会误伤。子类继承这条配置,不必重写。

> **金句**:**"`extra=\"ignore\"` 对'程序内部数据'是宽容,对'用户手写配置'是纵容——封闭字段集就该 `forbid`,把拼写错误挡在配置对象门外。"**

## 2.4 配置类的存在意义:`to_ultralytics_kwargs`

我们费劲做了个验证过的配置对象——它**为什么存在**?为了最终能一行喂给训练:`model.train(**config.to_ultralytics_kwargs())`。所以 BaseConfig 至少得提供这一个能力。

朴素写法 `model.train(**config.model_dump())` 不行,两个问题:① Pydantic 默认会带上 `None` 字段,ultralytics 看到 `device=None` 行为可能不一致,要**显式过滤**;② 有些字段是 D5 框架自己用的(比如 `verbose` 是我们的日志开关),**不该传给 ultralytics**。

死写法是直接 pop:

```python
def to_ultralytics_kwargs(self) -> Dict[str, Any]:
    d = self.model_dump(exclude_none=True)
    d.pop("verbose", None)
    return d
```

但马上有麻烦:下一阶段 `YOLOValConfig` 还要多 pop 一个 `task`(`.val()` 从权重推断,多传反而冲突)。死写法下,子类就得**把整个方法体复制一遍**只为多 pop 一个字段——子类越多重复越多。

更好的做法:把"哪些字段是框架级、不传 ultralytics"抽成一个 **`ClassVar` 集合**,方法只读集合;子类要扩展,**只改一行变量**,不碰方法体:

```python
from typing import ClassVar

class BaseConfig(BaseModel):
    FRAMEWORK_ONLY_FIELDS: ClassVar[set[str]] = {"verbose"}

    def to_ultralytics_kwargs(self) -> Dict[str, Any]:
        d = self.model_dump(exclude_none=True)
        for f in self.FRAMEWORK_ONLY_FIELDS:
            d.pop(f, None)
        return d
```

⚠️ **设计点(`ClassVar` 不是配置字段)**:`FRAMEWORK_ONLY_FIELDS` 标 `ClassVar` 后,Pydantic 不会把它当字段——不进 `model_fields`、不被 `model_dump()` 导出、也不受 `extra="forbid"` 约束(`forbid` 只管"传进来的数据",不管类属性)。它就是个普通类级常量。这个"子类改一行集合就能扩展"的设计,下一阶段拆 task 时会立刻兑现——你会看到 val 怎么用一行 `| {"task"}` 干净地比父类多 pop 一个字段。

> **金句**:**"子类扩展点设计成'改一行数据'还是'重写一个方法',直接决定这个基类好不好继承——能用数据表达的差异,别用代码重复表达。"**

> 注意:到此为止,BaseConfig 只有**字段 + 校验 + `to_ultralytics_kwargs`**。给人看的注释/示例/提示(`yaml_comment`/`examples`/`tips`)、敏感字段 mask、实验快照——**一个都还没加**。因为它们现在**没有消费者**:没有生成器要反射注释,没有报告会泄露敏感值,没人要复现实验。等它们各自的需求被撞到(阶段 5、阶段 6),我们再回来给 BaseConfig 添。**现在添,就是为不存在的需求写代码。**

## 2.5 BaseConfig 完整代码(瘦身版)

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : base.py
# @Author    : 雨霓同学 (ODPlatform team)
# @Project   : ODPlatform
# @Function  : runtime_config 子系统基础模型——BaseConfig(共享字段 + 验证)
"""运行配置子系统基础模块.

只装 BaseConfig: 所有模式(训练 / 验证 / 推理)共享的字段 + 验证.
专属字段去 train.py / val.py / infer.py.

★ 本阶段刻意只装"字段 + 校验 + to_ultralytics_kwargs":
  - 给人看的字段元数据(注释/示例/提示)→ 阶段 6 生成器需要时才加
  - 敏感字段 mask / 实验快照        → 阶段 5 报告/复现需要时才加
  没有消费者的能力, 现在不写.

★ legacy 的 ProjectConfig 已删除——paths.py 才是项目路径的唯一真相(D2 SSoT).
"""
from __future__ import annotations

import warnings
from typing import Any, ClassVar, Dict, Optional, Union

from pydantic import BaseModel, ConfigDict, Field, field_validator, model_validator


class BaseConfig(BaseModel):
    """所有模式共享的基础配置(字段 + 验证)."""

    # 拒绝 YAML / CLI 里拼错的字段名(详见 2.3 例 3)
    model_config = ConfigDict(extra="forbid")

    # 框架级字段: 只在 D5 内部用, to_ultralytics_kwargs() 会剔除, 不传给 ultralytics.
    # 子类(val/infer)在此基础上并入更多字段, 只改这一行.
    FRAMEWORK_ONLY_FIELDS: ClassVar[set[str]] = {"verbose"}

    # ---- 核心参数 ----
    model: Optional[str] = Field(default=None, description="模型文件路径(.pt 或 .yaml)")
    data:  Optional[str] = Field(default=None, description="数据集 yaml 路径")

    # ---- 输入配置 ----
    batch:   Union[int, float] = Field(default=16, description="批次大小(int 固定 / -1 自动 / 0-1 显存利用率)")
    imgsz:   int  = Field(default=640, gt=0, description="输入图像尺寸(建议 32 的倍数)")
    workers: int  = Field(default=8,   ge=0, description="数据加载进程数")
    cache:   bool = Field(default=False, description="是否缓存图像到内存/磁盘加速")
    rect:    bool = Field(default=False, description="矩形训练/推理(减少 padding)")

    # ---- 设备配置 ----
    device: Optional[str] = Field(default=None, description="运行设备(如 '0' / '0,1' / 'cpu'; None=自动)")
    amp:    bool = Field(default=True, description="自动混合精度(省显存)")

    # ---- 输出配置 ----
    project:  Optional[str] = Field(default=None, description="ultralytics 原生 project(输出根)")
    name:     Optional[str] = Field(default=None, description="ultralytics 原生 name(输出子目录)")
    exist_ok: bool = Field(default=False, description="允许复用已存在的输出目录")
    save:     bool = Field(default=True,  description="是否保存结果")

    # ---- 基础设置 ----
    verbose:       bool = Field(default=True, description="D5 框架日志开关(不传 ultralytics)")
    seed:          int  = Field(default=0, ge=0, description="随机种子")
    deterministic: bool = Field(default=True, description="确定性算法(牺牲速度换可复现)")

    # ============ 验证器 ============
    @field_validator("imgsz")
    @classmethod
    def _validate_imgsz(cls, v: int) -> int:
        if v % 32 != 0:
            warnings.warn(
                f"imgsz={v} 不是 32 的倍数. 虽然能跑, 但建议用 32 的倍数"
                f"(如 640, 672, 704)以获得最佳性能.",
                UserWarning,
            )
        return v

    @field_validator("batch", mode="before")
    @classmethod
    def _validate_batch(cls, v: Any) -> Union[int, float]:
        if isinstance(v, bool):
            raise TypeError("batch 不能是 bool 类型")
        if isinstance(v, int):
            if v == -1:
                return v
            if v <= 0:
                raise ValueError("batch 为 int 时必须为 -1(自动) 或 >= 1(固定值)")
            return v
        v_float = float(v)
        if not (0.0 < v_float <= 1.0):
            raise ValueError(
                f"batch 为 float 时必须在 (0, 1] 内, 表示显存利用率(如 0.7=70%), "
                f"当前值: {v_float}"
            )
        return v_float

    @model_validator(mode="after")
    def _cross_field_validation(self) -> "BaseConfig":
        if isinstance(self.batch, int) and self.batch > 0:
            if self.workers > self.batch * 2:
                warnings.warn(
                    f"workers={self.workers} 远大于 batch={self.batch}, 可能浪费资源. "
                    f"建议 workers <= batch * 2",
                    UserWarning,
                )
        return self

    # ============ 对外能力(目前只有一个: 喂给 ultralytics) ============
    def to_ultralytics_kwargs(self) -> Dict[str, Any]:
        """转成 ultralytics 可消费的 kwargs(过滤 None + 剔除框架级字段)."""
        d = self.model_dump(exclude_none=True)
        for f in self.FRAMEWORK_ONLY_FIELDS:
            d.pop(f, None)
        return d


if __name__ == "__main__":
    # 自测: 默认值能实例化, 非法值被拦
    cfg = BaseConfig(imgsz=640, batch=16)
    print("默认配置 OK:", cfg.to_ultralytics_kwargs())

    try:
        BaseConfig(batch=0)
    except Exception as e:
        print("batch=0 被拦 ✅:", type(e).__name__)

    try:
        BaseConfig(epchs=300)            # 拼错字段名
    except Exception as e:
        print("拼错字段名被拦 ✅:", type(e).__name__)
```

# 阶段 3: train / val 子类(撞墙②)+ infer 复用红利

BaseConfig 装好了共享字段。现在往里塞各模式专属的:训练的 60 个调参字段、验证的几个特有字段。但刚一动手写验证配置,就撞上 D5 的第二堵墙。

## 3.1 🧱 撞墙②:task 既当语义,又当目录名

legacy 的 `val_model.py` 里有这么个字段:

```python
# legacy: 一个 task 字段两件事都干, 看着省事其实病了
task: str = Field(default="detect_val", description="任务类型/输出目录名")
```

跑得起来吗?**能跑**。ultralytics 不在乎你传啥字符串当 `task`,验证时它从 `.pt` 权重反推真正的 task。那有什么问题?

> 🤔 **思考题(撞墙前先想 30 秒)**:① 你想机器化地问"这次跑的到底是 detect 还是 segment",怎么用这个字段回答?② 你想给这次跑取个名字叫 `helmet_v1`,又怎么用这个字段?

**问题①:task 不再是 SSoT。** D3 立的 `Task = {DETECT, SEGMENT}` 是项目里"任务类型"的唯一真相。而 legacy 这个字段允许 `"detect_val"` / `"segment_test"` / `"pose_val"` / `"safety_baseline_v2"`——**任何字符串**。一旦放任,想问"这次是 detect 吗"就只能字符串匹配("以 `detect` 开头算?那 `detection_v2` 算吗?`det_v1` 呢?")。更糟:D3 的 `data_pipeline` 一路在用 `Task.DETECT` 做条件分发,字段值要是 `"detect_val"`,`if task == Task.DETECT` 直接 False——逻辑链断了,用户什么都看不到。

**问题②:自定义实验名又被锁死。** 反过来,用户想给这次跑取名 `"helmet_v1"`,用 legacy 的设计就得写 `task = "helmet_v1"`——**等于把"它是个 detect 任务"也丢了**。最后用户只能折中成 `task = "detect_helmet_v1"`,字段值变成"语义 + 实验名拼字符串",以后想从里头解析回结构化信息就是字符串切片大赛。

**根本错误**:一个字段被强行承担了两个**正交**职责——

- **职责 A(任务语义)**:这是哪种 CV 任务?取值是封闭集合 `{detect, segment}`,**必须严格**。
- **职责 B(实验标识)**:这次跑叫啥名?取值是开放字符串,**必须自由**。

合二为一,两边都做不好。**拆。**

## 3.2 解法:task(SSoT)+ experiment_name(实验标识)

```python
# 职责 A: 任务语义(SSoT, 封闭取值)—— 复用 D3 的 Task
task: str = Field(default=Task.DETECT, description="任务类型(SSoT, 只能是 detect/segment)")

# 职责 B: 实验标识(自由字符串, 只用作目录组织)
experiment_name: Optional[str] = Field(default=None, description="实验名(目录组织, 不影响语义; None=自动)")

@field_validator("task")
@classmethod
def _validate_task(cls, v: str) -> str:
    if v not in Task.all():
        raise ValueError(
            f"task 必须是 {Task.all()} 之一, 当前值: {v!r}. "
            f"若想给本次实验取名, 请使用 experiment_name 字段."     # ★ 报错里直接指路
        )
    return v
```

两个字段,两个职责,互不污染:

```
用户写:      task: detect
            experiment_name: helmet_v1_lr001

D6 service: task 走任务分发(SSoT) + 把 experiment_name 翻成 ultralytics 的 name
输出目录:    runs/detect/helmet_v1_lr001/
```

⚠️ **设计点(task 的 validator 是硬墙,不是软提示)**:这里用 `raise` 不用 `warnings.warn`——task 填错(比如 `"Detect"` 大写)会让整条任务分发逻辑失灵,**必须拒**。而且错误信息**直接告诉用户"想取名走 experiment_name"**——配置报错不是"扔一行 ValueError 就完事",是"在最贴近错误的地方告诉他下一步怎么走"。用户调一次报错就懂,不用查文档。

⚠️ **设计点(BaseConfig 已有 `name` 字段,再加 `experiment_name` 不重复吗)**:不重复。`name` 是 **ultralytics 的原生参数**(对应 `model.train(name=...)`);`experiment_name` 是 D5 给用户的**对外语言**——更明确、不易跟 model name / dataset name 混。底层翻译(experiment_name → ultralytics name)由 D6 service 层做,配置层不直接打通。

> **金句**:**"一个字段同时扛两个职责,看着省一行代码,其实埋了所有后续问题的雷。两个职责,两个字段——这是最便宜的设计原则。"**

## 3.3 兑现阶段 2 的伏笔:task 在 train / val 的传递差异

`task` 字段 train、val 都有,但**传给 ultralytics 的方式不同**:train 要传(`model.train(task="detect")` 决定加载哪个模板),val **不传**(`.val()` 从权重自动推断,多传反而冲突)。

还记得阶段 2 我们把 `to_ultralytics_kwargs()` 设计成"基类读 `FRAMEWORK_ONLY_FIELDS` 集合、子类只改集合"吗?**这就是为它准备的**——子类各自声明"我有哪些字段不传 ultralytics",一行搞定:

```python
class YOLOTrainConfig(BaseConfig):
    # train: experiment_name 不传(由 service 翻成 name)
    FRAMEWORK_ONLY_FIELDS: ClassVar[set[str]] = (
        BaseConfig.FRAMEWORK_ONLY_FIELDS | {"experiment_name"}
    )

class YOLOValConfig(BaseConfig):
    # val: experiment_name 不传 + task 也不传(从权重推断)
    FRAMEWORK_ONLY_FIELDS: ClassVar[set[str]] = (
        BaseConfig.FRAMEWORK_ONLY_FIELDS | {"experiment_name", "task"}
    )
```

⚠️ **设计点(子类怎么"扩展" ClassVar)**:重新赋值要**再标一次 `ClassVar`**(否则 Pydantic 可能误判为字段);写成 `| {"task"}` 集合运算,还能让人一眼看清"我比父类多了什么",而不是"完全覆盖父类"。这就是阶段 2 说的"改一行数据 > 重写一个方法"——现在你看到它兑现了。

⚠️ **设计点(`FRAMEWORK_ONLY_FIELDS` 不是"禁止用户配置")**:用户在 YAML 写 `experiment_name: helmet_v1` 完全**合法**——它会进 Pydantic 实例、会被 service 读取。只是 `to_ultralytics_kwargs()` 把它过滤掉、不原样转发给 `model.train()`。"配置里能写" vs "原样转发给 ultralytics"是两件正交的事。

## 3.4 YOLOTrainConfig:52 个字段(仍然只写类型 + 默认 + 校验)

```python
    mask_ratio:   int  = Field(default=4, ge=1, description="分割: 掩码下采样比")

    # ---- 数据增强补全(几何 / 颜色)----
    shear:       float = Field(default=0.0, description="错切角度范围")
    perspective: float = Field(default=0.0, ge=0, le=0.001, description="透视变换强度")
    flipud:      float = Field(default=0.0, ge=0, le=1, description="垂直翻转概率")
    bgr:         float = Field(default=0.0, ge=0, le=1, description="BGR 通道翻转概率")
    # 其余增强(copy_paste_mode / auto_augment / erasing / crop_fraction 等)取值见官方 augmentation 表, 同模式补即可

    @field_validator("task")
    @classmethod
    def _validate_task(cls, v: str) -> str:
        if v not in Task.all():
            raise ValueError(
                f"task 必须是 {Task.all()} 之一, 当前值: {v!r}. "
                f"若想给本次实验取名, 请使用 experiment_name 字段."
            )
        return v
```

⚠️ **设计点(`ge=1` 这种约束一个都不能省)**:没有 `epochs` 的 `ge=1`,用户写 `epochs: 0` 不会立即报错,ultralytics 内部行为不可知。**能在配置层拦的,永远在配置层拦**——这是缺口② 的精神,贯穿每一个字段。

> ⚠️ **YOLO26 须知(你用的是最新版,这几处和老版本不同,值得留意)**
>
> 写讲义时核对了 ultralytics 官方 `cfg/default.yaml`(YOLO26)。相比 YOLO11 era,有几个**新东西**会落到配置字段上:
> - **新优化器 `MuSGD`**:YOLO26 的混合优化器(SGD + Muon 正交化更新),官方预训练 recipe 默认就用它,长训/大数据集推荐。`optimizer` 的合法取值因此多了 `MuSGD`(本讲义已加进字段)。
> - **`compile`(torch.compile 图编译)**:YOLO26 新增,`True`/`'default'`/`'reduce-overhead'`/`'max-autotune-no-cudagraphs'`,PyTorch 2.x 加速。已加进 train 字段。
> - **端到端 / 免 NMS(end2end)**:YOLO26 是 end-to-end 模型,**默认不走传统 NMS**。这让推理侧 `iou` / `agnostic_nms` 的语义变了——`agnostic_nms` 只去重"同框多标签",不再按 IoU 阈值抑制;想用回传统 NMS 管线(从而让 `iou` 生效),要显式 `end2end=False`。推理配置里因此值得带一个 `end2end` 字段(下一节 infer 已补)。
> - **新任务 `semantic`**:ultralytics 现在支持 `detect / segment / semantic / classify / pose / obb` 六类。本项目的 `Task`(D3 立)**故意只收 `detect` / `segment`**——这是项目范围的取舍,不是漏。若哪天要扩,改的是 D3 的 `Task` 枚举(SSoT 一处),runtime_config 不用动。
> - **模型名**:示例权重从 `yolo11n.pt` 换成 `yolo26n.pt`(本讲义已更新)。
>
> 这些只影响"字段取值/默认",不影响 D5 的任何结构设计——撞墙、merger、generator 一行都不用改。**这正是把"配置"做成独立子系统的好处:框架升级,改的是字段表,不是架构。**

⚠️ **设计点(`conf` 默认 0.001,跟推理的 0.25 差两个数量级)**:验证要算 mAP,得**抓住所有预测**再排序,所以阈值压到 0.001;推理是给人看结果,要**过滤低置信度**,默认 0.25。**同一个 `conf` 名字、默认值差 250 倍,正说明它不该进共享层**——这是 2.2 判据的反向印证。

## 3.6 复用红利:YOLOInferConfig

到这儿,设计原则已经立全了。加推理配置 `YOLOInferConfig` 时,你会发现**几乎没有新问题要撞**——撞墙的痛前面都付过了。它仅有两个要点:

**要点一:`source` 是推理的必填输入,但配置层不 raise。** train/val 的"数据"是 `data`(带标注的数据集 yaml),infer 的"数据"是 `source`——一张图 / 一段视频 / 一个目录 / 一个 RTSP 流,**无标注的原始输入**,语义完全不同,不放共享层:

```python
class YOLOInferConfig(BaseConfig):
    FRAMEWORK_ONLY_FIELDS: ClassVar[set[str]] = (

    end2end: Optional[bool] = Field(default=None, description="覆盖端到端模式; False=走传统 NMS, 让 iou 生效(YOLO26/v10)")
    classes: Optional[list[int]] = Field(default=None, description="只输出指定类别 id(None=全部)")
    embed:   Optional[list[int]] = Field(default=None, description="从指定层抽取特征向量(下游聚类/检索)")
    # 可视化输出(保存/绘制)----
    show:       bool = Field(default=False, description="弹窗显示标注结果")
    save_crop:  bool = Field(default=False, description="保存每个检测的裁剪图")
    save_frames: bool = Field(default=False, description="视频: 逐帧另存为图")
    show_labels: bool = Field(default=True,  description="绘制类别标签")

        if self.save_conf and not self.save_txt:
```

⚠️ **设计点(`source=None` 时配置层不 raise)**:看着像"必填字段配置层该拦",其实不是配置层的事——yaml 模板生成时 `source` 必须能是 `null`(用户拿到模板后填)。真正的"必填检查"是 D8 InferService 调 `model.predict()` 前做的。**配置层只管字段值的合法性,不管字段有没有给**——这条边界跟 val 的 task 同源。

**要点二:推理的"配置奇怪"几乎都能继续跑**(只是某些字段不生效),所以跨字段校验**全 warn 不 raise**(如 `save_conf` 需配套 `save_txt`)。

⚠️ **设计点(infer 的 `FRAMEWORK_ONLY_FIELDS` 跟 val 完全一样)**:都是 `| {"experiment_name", "task"}`(`.predict()` 也从权重推断 task)。**两者对 ultralytics 的接口形态同款**——这种"加新模式却发现它跟旧模式高度同构"的体验,正是前面把设计原则立对了的回报。

# 阶段 4: loaders —— 配置从外部进来(撞墙③)

配置类有了,但它现在只能在代码里 `YOLOTrainConfig(epochs=200)` 手动构造。真实场景里,配置是从**外部**来的:用户写的 YAML 文件、命令行敲的参数。这一阶段写两个 Loader,把外部数据装进 dict。

先把职责划清:**Loader 只负责"把外部数据读成 dict",不验证、不合并。** 字段值验证是 Pydantic 的事(阶段 2-3 已立),多源合并是 merger 的事(阶段 5)。一个模块一件事。

## 4.1 YAMLLoader 要处理的四件事

```
1. 路径解析: 文件名 / 相对路径 / 绝对路径都接受
2. 编码:     默认 UTF-8, 失败 fallback 系统默认
3. 解析:     YAML 语法错 → fail-fast(带原始错误)
4. 不存在:   ★ 撞墙③ —— 这一条最容易写错
```

前三条都直白。第四条,先按最自然的思路写,撞墙。

## 4.2 🧱 撞墙③:YAML 文件不存在,该怎么办?

用户第一次跑,还没生成 yaml,`configs/runtime/train.yaml` 根本不存在:

```python
cfg = loader.load("train.yaml")    # 文件不存在的话呢?
```

三种可能的处理:

| 方案 | 行为 | 毛病 |
|---|---|---|
| A. 自动生成默认模板继续跑 | 检测不存在 → 偷偷生成一份 → 用默认值跑 | **静默**——用户不知道 yaml 是刚生成的 |
| B. 返回空 dict | 当空文件处理,默认值跑 | 用户**以为自己改的 yaml 在生效**,实际从没读到——最危险 |
| C. raise + 修复指引 | 报错告诉用户跑哪条命令生成 | 第一次稍麻烦,但不会误导 |

**legacy 选了方案 A**(`service.py` 里检测不存在就自动 `generator.generate(...)` 继续)。看着贴心,其实埋雷:

**场景 1(CI 流水线)**:CI 镜像没把 `train.yaml` 打进去,你跑 `odp-train --epochs 200`。legacy 检测到文件没有 → 静默生成默认 yaml → 用默认值开跑。CI 跑 **5 小时**,结果一看 `epochs=100`(默认),不是你要的 200——而"你以为有 yaml、实际是刚生成的默认 yaml"这件事 **CI 不会告诉你**,排查半天。

**场景 2(改名拼错)**:你想改 `train.yaml`,手滑存成了 `train_.yaml`。你跑 `odp-train --config train.yaml`(原名)。legacy 找 `train.yaml` 不存在 → 立马生成一份新的默认的。你纳闷"为啥我 `batch=32` 不生效?"——因为你跑的是刚生成的 `batch=16` 那份,你精心调过的 `train_.yaml` 根本没被读。

两个场景共同点:**"文件不存在"这个事实被静默处理掉了**——用户以为有,实际没有。这就是阶段 1 伤痛②(静默通过)、D4 撞墙③(残缺数据白跑)在文件层面的翻版。

### D5 的方案:fail-fast + 修复指引

```python
if not filepath.exists():
    raise FileNotFoundError(
        f"YAML 配置文件不存在: {filepath}\n\n"
        f"请先生成默认配置模板:\n"
        f"  odp-gen-config {filepath.stem}\n\n"
        f"生成后编辑该文件再重新运行."
    )
```

三件事一起做:① **raise**——文件不存在是异常,必须停;② **报准确路径**——`filepath` 是绝对路径,一眼知道去哪找;③ **给修复指引**——告诉用户**下一条命令跑啥**,而不是"你自己想办法"。

⚠️ **设计点(fail-fast 报错必须带修复指引,否则就是耍流氓)**:一条好的 fail-fast 报错 = "出了啥事" + "为啥不行" + "怎么修"。三个都有,用户 5 秒理解;缺一个,用户就得查文档。**报错就是你给下一个用户写的最后一份文档。**

⚠️ **设计点(那 legacy 的"自动生成模板"功能就废了?)**:不废,只是**分开做**。`odp-gen-config train`(阶段 6)显式让用户生成模板,但**不再在 loader 里偷偷生成**。关键是**显式 vs 隐式**:显式 = 用户主动跑了 generator → 看到模板路径 → 心里有数;隐式 = 用户啥都没做,yaml 突然出现 → 不知道 → 跑出来不对。**自动化是好的,隐式自动化是糟糕的。**

> **金句**:**"`return {}` 和 `raise` 看着只差 5 个字符,实际是 'CI 跑 5 小时然后骂街' 和 '5 秒看到报错改命令' 的差别。fail-fast 不是脾气,是体谅。"**

## 4.3 YAMLLoader 完整代码

`apps/platform/src/od_platform/runtime_config/loaders.py`:

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : loaders.py
# @Author    : 雨霓同学 (ODPlatform team)
# @Project   : ODPlatform
# @Function  : runtime_config 子系统——配置加载器(只读成 dict, 不验证不合并)
"""配置加载器: 把外部数据装进 dict.

- YAMLLoader: 加载 YAML 文件, 不存在 → fail-fast + 修复指引
- CLILoader:  加载命令行参数, 过滤控制字段

★ 职责边界: Loader 只把外部数据装进 dict.
   - 字段值验证 → Pydantic(阶段 2-3)
   - 多源合并   → ConfigMerger(阶段 5)
"""
from __future__ import annotations

import logging
from argparse import Namespace
from pathlib import Path
from typing import Any, Dict, List, Mapping, Optional, Union

import yaml

logger = logging.getLogger(__name__)


def _drop_none(d: Mapping[str, Any]) -> Dict[str, Any]:
    """过滤 None, 保留 False / 0 / '' 等显式值.

    YAML 的 null 等同"没写", 合并时不该参与覆盖, 让下层默认值有机会跑.
    但 False / 0 / '' 是用户显式填的"假", 必须保留 ——
    不要写成 {k: v for k, v in d.items() if v}(经典 falsy 误伤).
    """
    return {k: v for k, v in d.items() if v is not None}


class YAMLLoader:
    """加载 YAML 配置文件 → dict."""

    def __init__(self, config_dir: Optional[Union[str, Path]] = None):
        self.config_dir = Path(config_dir) if config_dir else None

    def load(self, filename: Union[str, Path]) -> Dict[str, Any]:
        # 1. 解析路径(文件名→config_dir 下找; 绝对/相对原样)
        filepath = self._resolve_path(filename)

        # 2. 不存在 → fail-fast + 修复指引(★ 撞墙③)
        if not filepath.exists():
            raise FileNotFoundError(
                f"YAML 配置文件不存在: {filepath}\n\n"
                f"请先生成默认配置模板:\n"
                f"  odp-gen-config {filepath.stem}\n\n"
                f"生成后编辑该文件再重新运行."
            )

        # 3. 读文件(默认 UTF-8, 失败 fallback —— 系统行为, 走 logger)
        try:
            content = filepath.read_text(encoding="utf-8")
        except UnicodeDecodeError:
            logger.warning(f"UTF-8 解码失败, 尝试系统默认编码: {filepath}")
            content = filepath.read_text()

        # 4. 空文件 → {}(合法, 等同"全用默认值")
        if not content.strip():
            logger.debug(f"YAML 文件为空: {filepath}")
            return {}

        # 5. 解析 —— 失败 fail-fast, 保留 exception chain
        try:
            data = yaml.safe_load(content)
        except yaml.YAMLError as e:
            raise ValueError(
                f"YAML 格式错误: {filepath}\n原始错误: {e}\n"
                f"提示: 检查缩进、引号匹配、冒号后是否有空格."
            ) from e

        # 6. 顶层结构检查
        if data is None:
            return {}
        if not isinstance(data, dict):
            raise ValueError(
                f"YAML 顶层必须是字典, 当前是 {type(data).__name__}: {filepath}"
            )
        return data

    def _resolve_path(self, filename: Union[str, Path]) -> Path:
        p = Path(filename)
        if p.is_absolute():
            return p
        if self.config_dir is not None:
            return self.config_dir / p
        return p
```

⚠️ **设计点(YAML 错误保留 `from e` 异常链)**:`raise ... from e` 让 traceback 同时显示"我们的友好提示"和"PyYAML 的原始错误"——用户既看到该怎么查,工程师也能 grep 到底层细节。**包装异常时别把原始异常吞了。**

⚠️ **设计点(空文件返回 `{}` 是合法的,不存在才 raise)**:"文件存在但空"= 用户有意"全用默认值",合法;"文件不存在"= 用户以为有但没有,危险。**两件事必须区别对待**——这正是阶段 1 对 None 语义一致性的延续。

## 4.4 CLILoader:命令行参数 → dict

```python
class CLILoader:
    """从 argparse Namespace 提取配置参数 → dict.

    职责: 把 argparse 结果装进 dict + 滤掉控制字段(如 --config 本身).
    """

    CONTROL_FIELDS = {"config", "func", "command"}     # argparse 控制字段, 不是配置

    def load(self, args: Namespace, *, exclude: Optional[set[str]] = None) -> Dict[str, Any]:
        exclude = (exclude or set()) | self.CONTROL_FIELDS
        raw = {k: v for k, v in vars(args).items() if k not in exclude}
        return _drop_none(raw)     # None = "用户没在 CLI 给这个参数", 不覆盖
```

⚠️ **设计点(CLI 的 None 也要 `_drop_none`)**:argparse 里没传的参数默认是 `None`。如果不滤,CLI dict 里会带一堆 `lr0=None`,合并时把 YAML 的真实值覆盖成 None——灾难。`_drop_none` 在 YAML 和 CLI 两边都用,**整套系统对 None 的语义保持一致:None = 没给,不覆盖。**

# 阶段 5: ConfigMerger —— 三源合并 + 链表式溯源(缺口③ 在此填上)

阶段 4 两个 Loader 各自把 YAML、CLI 装进 dict。但两个 dict 孤立着,还差最后一步——**按优先级合并,喂给 Pydantic**。再加上阶段 1 一直欠着的**缺口③(无溯源)**,这一阶段解决两件事:① 优先级合并(CLI > YAML > DEFAULT);② 每个生效值能追溯"我是谁、从哪来、覆盖了谁"。

## 5.1 朴素合并:一行 dict 解包够吗?

```python
defaults = {"epochs": 100, "batch": 16, "lr0": 0.01}
yaml_cfg = {"epochs": 200,             "lr0": 0.001}
cli_cfg  = {"epochs": 50}

merged = {**defaults, **yaml_cfg, **cli_cfg}
# → {"epochs": 50, "batch": 16, "lr0": 0.001}
```

一行解包,后写覆盖先写,**数值上正确**。能用吗?

> 🤔 **思考题**:这个 `merged` 能解决缺口③ 吗?三个月后你看日志 `lr0=0.001`,它能告诉你这是谁给的吗?
>
> **答**:不能。`merged` 是个扁平 dict,信息只剩"最后赢的那个值"——谁覆盖了谁、原值是啥、来源是谁,全丢。朴素合并把**信息熵全压缩了**。

所以要单独写一个 ConfigMerger,而不是 5 行 dict 解包。**合并的过程要留痕迹。**

## 5.2 配置溯源:平铺标记 vs 链表式

要追溯来源,最直觉是给每个字段配一个 `source` 标记:

```python
sources = {"epochs": "CLI", "batch": "DEFAULT", "lr0": "YAML"}
```

够吗?看个场景。用户跑 `odp-train --epochs 50`,问:"为啥我 yaml 里写了 `epochs: 200` 没生效?" 平铺方案能答 `epochs` 来源是 CLI(值 50),但答不出:**"在 CLI 覆盖之前,它是 YAML 给的 200 还是默认 100?"** 这段历史被吃了。更糟:实习生说"我把 yaml 改成 200 了,为啥跑出来 50?"你要回答"你 CLI 里加了 `--epochs 50` 覆盖了它,不是 yaml 没生效"——平铺方案给不出**覆盖链**,你只能反复让他自己看 CLI 敲了啥。

**链表式溯源**解决它——每个字段记完整的覆盖链:

```
epochs 的覆盖链:
    50(CLI) ← 200(YAML) ← 100(DEFAULT)
      ↑          ↑             ↑
   当前赢的    被覆盖了       最初默认
```

回答"为啥 yaml 没生效"一行:"你 yaml 给了 200,被 CLI 的 50 覆盖了。"

⚠️ **设计点(为啥是链表不是 list)**:看着"链表"比"list"复杂,但语义更准——每次覆盖是一个"事件",事件指向"上一个事件"。这就是 **git commit 链** 的思路:你不会写 `commits = [c1, c2, c3]`,你写 `c3.parent = c2`。链表的"指向上一个"跟"覆盖了谁"在语义上完全对齐。

```python
from dataclasses import dataclass
from datetime import datetime
from typing import Any, List, Optional, Union

@dataclass
class ConfigMetadata:
    key:             str
    value:           Any
    source:          Union[ConfigSource, str]              # 枚举或自定义字符串
    timestamp:       datetime
    overridden_from: Optional["ConfigMetadata"] = None     # ★ 指向被覆盖的上一个

    @property
    def source_label(self) -> str:
        return self.source.value if isinstance(self.source, ConfigSource) else self.source

    def chain(self) -> List["ConfigMetadata"]:
        result, cur = [self], self.overridden_from
        while cur:
            result.append(cur); cur = cur.overridden_from
        return result

    def chain_str(self) -> str:
        return " ← ".join(f"{m.value}({m.source_label})" for m in self.chain())
```

⚠️ **设计点(`←` 还是 `→`)**:选 `←`。**当前值在左**——读 `50 ← 200 ← 100` 从左到右就是"现在 50、之前 200、最初 100",符合"从结果倒推原因"的阅读习惯。用 `→` 得先看到尾巴才知道当前值,反了。

## 5.3 ConfigSource:优先级是 list 顺序,不是数字

```python
from enum import Enum

class ConfigSource(str, Enum):
    """配置来源(内置 3 档)."""
    DEFAULT = "DEFAULT"
    YAML    = "YAML"
    CLI     = "CLI"
```

⚠️ **设计点(没有 `DEFAULT=1, YAML=2` 这种编号)**:故意的——优先级**由 sources list 的顺序决定**,不由数字。调用时 `sources=[(YAML, ...), (CLI, ...)]`,**list 里谁后写谁优先级高**。想插新档源只需在 list 里加一项,merger 一行不动。数字优先级反而逼人去查"ENV=4 是高还是低于 CLI=3"这种人造问题。

⚠️ **设计点(枚举只装内置 3 档,自定义源走字符串)**:部署后会想加 ENV / etcd / k8s configmap。每加一档改枚举?违反开闭原则。让枚举装核心 3 档,**字符串装扩展**:`sources=[(YAML, ...), ("ENV", env_dict), (CLI, ...)]`。跟 Python `Path` 接受 `str | PathLike` 是同款取舍。

## 5.4 ConfigMerger:sources list 接口 + 链表怎么长出来

```python
class ConfigMerger:
    """三源合并 + 链表式溯源(sources list 接口, 支持任意源)."""

    def __init__(self, track_sources: bool = True):
        self.track_sources = track_sources
        self._metadata: Dict[str, ConfigMetadata] = {}
        self._overridden_keys: List[str] = []
        self._last_config_class = None                  # 供报告 mask 用(5.6)

    def merge(self, config_class, *, sources=None):
        merged = self._do_merge(config_class, sources)
        try:
            return config_class(**merged)               # ★ forbid + 验证在这道门
        except ValidationError as e:
            if self.track_sources:
                self._enhance_validation_error(e)       # 把溯源塞进错误(5.4.1)
            raise

    def _do_merge(self, config_class, sources):
        self._metadata.clear(); self._overridden_keys.clear()
        self._last_config_class = config_class
        # DEFAULT 源永远自动注入到最前(反射自 Pydantic, 走 SSoT)
        all_sources = [(ConfigSource.DEFAULT, self._extract_defaults(config_class))]
        all_sources.extend(sources or [])
        merged: Dict[str, Any] = {}
        for source, cfg in all_sources:
            self._apply_source(merged, dict(cfg or {}), source)
        return merged

    def _apply_source(self, merged, config, source):
        for key, value in config.items():
            if value is None:                           # None = 没给, 不覆盖
                continue
            if key in merged and self.track_sources:
                self._overridden_keys.append(key)
            merged[key] = value
            if self.track_sources:                      # ★ 链表精髓
                prev = self._metadata.get(key)
                self._metadata[key] = ConfigMetadata(
                    key=key, value=value, source=source,
                    timestamp=datetime.now(), overridden_from=prev,
                )

    @staticmethod
    def _extract_defaults(config_class) -> Dict[str, Any]:
        """从 Pydantic model_fields 反射默认值(不单独维护一份)."""
        return {n: f.default for n, f in config_class.model_fields.items()
                if f.default is not None}
```

⚠️ **设计点(默认值从 Pydantic 反射,不另存一份)**:`Field(default=...)` 已经是默认值的唯一真相(SSoT 第三层)。再维护一份默认 dict,改字段时永远忘同步,SSoT 立破。**能反射的永远反射,别复制。** 同理 DEFAULT 源由 merger 自动注入,调用方只关心 YAML/CLI 这种"外部源",不手动传 default。

⚠️ **设计点(签名为啥不是 `merge(cls, *, yaml_config, cli_config)`)**:把两个源名写死在签名里看着教学清楚,但加第三档源就得改签名、碰所有调用方,违反开闭。sources list 一次堵死:数量无上限,顺序由调用方定。这跟 D3/D4 的开闭原则一脉相承。

```python
from od_platform.common.string_utils import pad_to_width    # ★ 老朋友(D1 撞墙⑤)

def _center(text: str, width: int = 70) -> str:
    return pad_to_width(text, width, align="center")          # CJK 感知的居中

def get_source_report(self) -> str:
    if not self.track_sources:
        return "配置溯源未启用"
    lines = ["=" * 70, _center("配置来源报告"), "=" * 70]

    by_source: Dict[str, List[str]] = {}
    for key, meta in self._metadata.items():
        by_source.setdefault(meta.source_label, []).append(key)

    # 内置 source 在前(CLI/YAML/DEFAULT 序), 自定义 source 在后字母序
    builtin = [s.value for s in (ConfigSource.CLI, ConfigSource.YAML, ConfigSource.DEFAULT)]
    ordered = [s for s in builtin if s in by_source]
    custom  = sorted(s for s in by_source if s not in builtin)

    for label in ordered + custom:
        keys = sorted(by_source[label])
        lines.append(f"\n{label} ({len(keys)} 项)")
        lines.append("-" * 70)
        for key in keys:
            value = self._display_value(key, self._metadata[key].value)   # mask, 见 5.6
            lines.append(f"  {key} = {value}")
    return "\n".join(lines)
```

现在标题正了:

```
======================================================================

  epochs = 200
```

⚠️ **设计点(按来源分组,不按字段名)**:这张表是给**调参视角**看的——用户最关心"我这次实际改了啥(CLI/YAML),哪些是默认"。按来源分组让"改过的字段"集中在上半部分,一眼看完。按字段名字母序排,改过的跟没改的混在一起,找不出重点。

> **金句**:**"D1 那天为对齐中文撞的墙、写的 `string_utils`,在这里一行复用就把报告排正了。基础设施的价值不在'当天用一次',在'后面每次撞到同类问题都不用重写'——这就是'撞一次、用很多次'。"**

## 5.6 报告会泄密 —— 这才逼出 `sensitive` 和 mask

上面报告里 `data = ***` 那个 `***` 是哪来的?看一个真实场景:用户的 `data` 字段填的是 `s3://my-bucket/dataset.yaml?access_key=AKIA...`,`project` 填的是公司内部代号。**这张报告会进日志**——日志一进 ELK、一被转发,**密钥就泄了**。

阶段 2 我们**故意没给** BaseConfig 加任何"敏感"概念,因为那时没有"报告会泄密"这个需求。**现在撞到了,才回头给它添**——这就是"能力随需求生长"。

需要三样东西,一处生一处:

**① 给敏感字段打个标(BaseConfig 字段的 `json_schema_extra` 加 `sensitive`)**:

```python
# base.py 里, 给少数真敏感的字段补一个 sensitive 标记(其余字段不动):
data: Optional[str] = Field(
    default=None, description="数据集 yaml 路径",
    json_schema_extra={"sensitive": True},      # ★ 现在才加, 因为现在才需要
)
project: Optional[str] = Field(
    default=None, description="ultralytics 原生 project",
    json_schema_extra={"sensitive": True},
)
```

⚠️ **设计点(`sensitive` 默认就是"没有")**:75+ 字段里真敏感的就 `data` / `project` 这种少数,**绝大多数字段一个字都不用动**。这就是为什么阶段 2 不预埋它——99% 的字段根本不需要这个槽位,提前给每个字段塞 `sensitive: False` 是纯噪声。

**② BaseConfig 加 mask 能力(配套上面的标记)**:

```python
# base.py 增量
SENSITIVE_MASK: ClassVar[str] = "***"

def mask_sensitive_dump(self) -> Dict[str, Any]:
    """model_dump() 但 sensitive 字段值替换成 *** (进日志前调)."""
    dump = self.model_dump()
    for name, field in self.__class__.model_fields.items():
        extra = field.json_schema_extra or {}
        if isinstance(extra, dict) and extra.get("sensitive") and dump.get(name) is not None:
            dump[name] = self.SENSITIVE_MASK
    return dump

@classmethod
def sensitive_field_names(cls) -> set[str]:
    """标了 sensitive=True 的字段名(供 merger 报告自动 mask 用)."""
    return {n for n, f in cls.model_fields.items()
            if isinstance(f.json_schema_extra, dict) and f.json_schema_extra.get("sensitive")}
```

**③ merger 报告里自动 mask(打通上面两样)**:

```python
def _is_sensitive(self, key: str) -> bool:
    cls = self._last_config_class
    return bool(cls and hasattr(cls, "sensitive_field_names")
                and key in cls.sensitive_field_names())

def _display_value(self, key: str, value: Any) -> Any:
    if value is None:
        return value
    if self._is_sensitive(key):
        return getattr(self._last_config_class, "SENSITIVE_MASK", "***")
    return value
```

⚠️ **设计点(None 不 mask)**:`data: None` 是合法默认,不该 mask 成 `***`——那会让用户以为"路径有但被藏了"。**None 是真实信息,替换反而误导。**

⚠️ **设计点(`__repr__` 不 mask,是明确的不做)**:`print(config)` 走 Pydantic 默认实现,会露原值。改 `__repr__` 会让 debug / IDE inspect 看不到真实值,排查更难。**mask 是显式调用 `mask_sensitive_dump()` 的事,不强加给所有打印**——边界划清。

⚠️ **设计点(`_display_value` 一处实现,三处共用)**:来源报告、覆盖报告、审计日志都走它。**mask 一处实现,加新报告时不会忘。**

## 5.7 dry-run 预览:`preview()`(填缺口⑥)

`merge()` 一调就 Pydantic 验证,用户没法"先看一眼合并结果再决定改不改"。加个预览入口:

```python
def preview(self, config_class, *, sources=None) -> Dict[str, Any]:
    """Dry-run: 合并 + 维护溯源, 但不实例化、不验证."""
    return self._do_merge(config_class, sources)
```

⚠️ **设计点(为啥单独叫 `preview()`,不给 `merge()` 加 `validate=False`)**:`merge()` 返回 `T`(验证后的实例),`preview()` 返回 `Dict`(未验证)——**返回类型不同,方法名不同,mypy 能精确推断**。一个方法名能讲清的事,别靠开关参数。

> **金句**:**"dry-run 不是'半成品调用',是'决定之前看一眼'。配置系统给用户 5 秒思考的余地,比训练 5 小时之后再后悔便宜得多。"**

## 5.8 两个审计产物:复现 + 运维

复现和运维要的是两样不同的东西,各给一个:

```python
# merger: 给运维的结构化审计日志(每个字段的来源 + 覆盖链, JSON, 进 ELK)
def to_audit_log(self) -> Dict[str, Any]:
    return {
        "fields": {
            k: {"value": self._display_value(k, m.value),       # 同样自动 mask
                "source": m.source_label,
                "chain": [f"{x.value}({x.source_label})" for x in m.chain()]}
            for k, m in self._metadata.items()
        },
        "overridden": list(dict.fromkeys(self._overridden_keys)),
    }
```

```python
# base.py 增量: 给复现的配置快照(值 + 类标识 + 时间戳)
def to_audit_snapshot(self) -> Dict[str, Any]:
    """完整快照, 用于实验复现. 跟 to_ultralytics_kwargs / model_dump 区别:
       它带类标识 + 时间戳, 三个月后能从 dict 反序列化回正确的类."""
    return {
        "config_class":  self.__class__.__name__,
        "frozen_at":     datetime.now().isoformat(timespec="seconds"),
        "values":        self.model_dump(),
    }

@classmethod
def from_audit_snapshot(cls, snapshot: Dict[str, Any]) -> "BaseConfig":
    got = snapshot.get("config_class")
    if got != cls.__name__:
        raise ValueError(f"快照来自 {got!r}, 不能恢复到 {cls.__name__!r}. 请用 {got}.from_audit_snapshot().")
    return cls(**snapshot["values"])       # ★ 复现时也跑一遍验证
```

⚠️ **设计点(`from_audit_snapshot` 走完整验证)**:用 `cls(**values)` 而不是 `__new__` 直接装值——**复现时也跑一遍验证**。半年后字段定义变了(比如 `mosaic` 范围改了),老快照会立刻报错。这比静默接受安全:想复现就得正面面对配置定义的演进。

⚠️ **设计点(两个审计产物各管一摊)**:`to_audit_snapshot`(配置类的方法)存"最终字段值",目的是**跑出同样结果**,跟"当初谁配的"无关;`to_audit_log`(merger 的方法)存"来源 + 覆盖链",目的是**审计"为什么是这个值"**。两件事别混。

# 阶段 6: ConfigGenerator —— 反射式 YAML 模板(缺口④ + 元数据槽位在此出生)

四个缺口,还差最后一个——**缺口④:YAML 模板手维护、注释和代码漂移**。这一阶段写 `ConfigGenerator`,从字段**反射**出一份会自己解释自己的 YAML 模板。但一动手,就发现字段里**没有东西可反射**。

## 6.1 缺口④:从字段反射模板 —— 可字段里没有注释

阶段 1 我们恨手写注释会漂移,结论是"字段元数据该跟字段定义在同一处,模板由代码自动产出"。现在动手:遍历字段,把每个字段的"含义、范围、示例"写成 YAML 注释。

可问题来了——阶段 2、3 我们给字段写的 `Field(default=, description=)`,`description` 是 **给程序员看的**(API 文档、IDE 提示),写得短而克制(`"批次大小"`)。而 YAML 模板里要给**最终用户**看的注释,得说清选项、给出示例、附上调参建议——**两种受众,`description` 一个字段装不下**。

换句话说:**生成器需要的元数据,字段里现在没有。** 这才是该添它们的时刻——不是阶段 2 凭空预埋,是阶段 6 被生成器的需求逼出来。

## 6.2 元数据槽位在此出生

给字段补一组**给用户看的**元数据,装在 `Field` 的 `json_schema_extra` 里。挑几个代表性字段示范(其余字段照此模式补全):

```python
# train.py: 给 epochs 补上"给用户看"的元数据(其余 51 个字段同模式)
epochs: int = Field(
    default=100, ge=1,
    description="训练总轮数",                              # 给程序员: 短
    json_schema_extra={
        "group":    "训练控制",                            # ← 槽位1: 模板按它分章节
        "examples": [50, 100, 200, 300],                  # ← 槽位2: 模板写 "# 示例: ..."
        "tips": [                                          # ← 槽位3: 模板写多行 "# 提示:"
            "小数据集: 50-100 轮",
            "中等数据集: 100-200 轮",
            "大数据集 / 工业检测: 200-300 轮",
        ],
        "yaml_comment": "训练总轮数(epoch)",               # ← 槽位4: 字段上方那行注释(给用户)
    },
)

# base.py: batch 同样补全(注意 yaml_comment 比 description 更细)
batch: Union[int, float] = Field(
    default=16,
    description="批次大小",                                # 给程序员
    json_schema_extra={
        "group":    "输入配置",
        "examples": [8, 16, 32, -1, 0.7],
        "tips": ["int: 固定批次(如 16)", "-1: 自动批次(约 60% 显存)",
                 "0.0-1.0: 显存利用率(0.7 = 70%)"],
        "yaml_comment": "批次大小(int=固定, -1=自动, 0-1=显存利用率)",   # 给用户, 含选项
    },
)
```

⚠️ **设计点(为什么 `description` 和 `yaml_comment` 是两个字段)**:`description` 给程序员(IDE/API 文档),稍长无妨;`yaml_comment` 给最终用户,一行内必须读懂。**两种受众,两个字段。** 没写 `yaml_comment` 时反射会 fallback 到 `description`(见下 `get_field_metadata`),多层 fallback 让"漏写"不挂掉、只是注释稍弱。

⚠️ **设计点(`tips` 不要写废话)**:反例 `tips: ["训练多少轮"]`——description 已经说了,重复。正例:给具体数据集规模的推荐值——**用户调参时真正想知道、但翻 ultralytics 官方文档查不到**的东西。元数据的价值在"补文档查不到的",不在"复述字段名"。

现在给 BaseConfig 加两个**反射方法**(也是被生成器逼出来的,阶段 2 没有它们):

```python
# base.py 增量: 供 generator 反射用(阶段 6 才加, 阶段 2 没消费者)
def get_field_groups(self) -> Dict[str, List[str]]:
    """{group_name: [field_names]} —— 供 generator 按组写章节."""
    groups: Dict[str, List[str]] = {}
    for name, field in self.__class__.model_fields.items():
        extra = field.json_schema_extra or {}
        group = extra.get("group", "其他") if isinstance(extra, dict) else "其他"
        groups.setdefault(group, []).append(name)
    return groups

def get_field_metadata(self, field_name: str) -> Dict[str, Any]:
    """拿单字段全部元数据(供 generator 写注释), 多层 fallback."""
    field = self.__class__.model_fields[field_name]
    meta = {"description": field.description, "default": field.default,
            "examples": [], "tips": [],
            "yaml_comment": field.description, "group": "其他"}   # yaml_comment fallback description
    if isinstance(field.json_schema_extra, dict):
        meta.update(field.json_schema_extra)
    return meta
```

⚠️ **设计点(Generator 只读、不改 Pydantic 模型)**:`get_field_groups` / `get_field_metadata` 放在 **BaseConfig** 作为通用方法,任何子类都自动支持模板生成,Generator 不需要认识它们的具体字段——它只 `config.get_field_groups()` 反射。**反射端不耦合具体子类,这就是为什么这两个方法属于配置类而不是 Generator。**

## 6.3 反射循环 + YAML 值格式化

Generator 的核心就 3 步:遍历组 → 每个字段写注释三连 → 写字段值。

```python
for group_name, field_names in config.get_field_groups().items():
    lines.append(f"#{'-'*78}\n# {group_name}\n#{'-'*78}")
    for fname in field_names:
        meta = config.get_field_metadata(fname)
        lines.append(f"# {meta['yaml_comment']}")
        if meta["examples"]:
            lines.append(f"# 示例: {', '.join(self._format_value(e) for e in meta['examples'])}")
        if meta["tips"]:
            lines.append("# 提示:")
            lines.extend(f"#   - {t}" for t in meta["tips"])
        lines.append(f"{fname}: {self._format_value(getattr(config, fname))}")
        lines.append("")
```

字段值要按 YAML 语法格式化,不同类型不同写法:

```python
def _format_value(self, value: Any) -> str:
    if value is None:
        return "null"
    if isinstance(value, bool):                       # ★ 必须在 int 之前!
        return "true" if value else "false"
    if isinstance(value, str):
        if any(c in value for c in [":", "#", "[", "]", "{", "}"]):
            return f'"{value}"'                        # 含特殊字符加引号
        return value
    if isinstance(value, (list, tuple)):
        return "[]" if not value else f"[{', '.join(str(v) for v in value)}]"
    return str(value)
```

⚠️ **设计点(bool 必须在 int 之前判)**:又是 `isinstance(True, int) == True`——若先判 int,`True` 会被格式化成 `"True"`(Python 大写,不是 YAML 的小写 `true`)。跟阶段 2 的 `_validate_batch`、merger 没踩但同源的坑一样:**bool 永远先判。**

生成的模板长这样(片段):

```yaml
#==============================================================================
# YOLO 训练配置
# 自动生成时间: 2026-06-29 14:32:15
#==============================================================================

#------------------------------------------------------------------------------
# 训练控制
#------------------------------------------------------------------------------
# 训练总轮数(epoch)
# 示例: 50, 100, 200, 300
# 提示:
#   - 小数据集: 50-100 轮
#   - 中等数据集: 100-200 轮
#   - 大数据集 / 工业检测: 200-300 轮
epochs: 100
```

一份**会自己解释自己**的模板——缺口④ 填上。

## 6.4 文件安全:默认不覆盖 + 覆盖时备份

用户跑两次 `odp-gen-config train`,第二次别把他调好的 yaml 冲了。两道闸门:

```python
import shutil
from datetime import datetime

def generate(self, config_class, output_path, *, overwrite=False, backup=True, title=None) -> bool:
    output_path = Path(output_path)
    if output_path.exists() and not overwrite:                 # 第一道: 默认不覆盖
        logger.info(f"配置文件已存在, 跳过生成: {output_path}")
        return False
    if output_path.exists() and backup:                        # 第二道: 覆盖前备份
        stamp = datetime.now().strftime("%Y%m%d_%H%M%S")
        bak = output_path.with_name(f"{output_path.name}.bak.{stamp}")
        shutil.copy2(output_path, bak)
        logger.warning(f"覆盖前已备份原配置: {bak}")             # 系统行为, 走 logger
    output_path.parent.mkdir(parents=True, exist_ok=True)
    output_path.write_text(self._generate_yaml(config_class, title), encoding="utf-8")
    logger.info(f"配置文件已生成: {output_path}")
    return True
```

⚠️ **设计点(默认 `overwrite=False` + 默认 `backup=True`)**:两个"默认安全 + 显式才破坏"——跟 D2 `odp-reset` 同款纪律,危险动作走显式入口。备份用 `logger.warning`(系统行为,不是用户输入有问题),又一次落实阶段 2 立的警告渠道分离。返回 `True/False` 而非抛异常:**"文件已存在不该生成"是合法常态,不是错误**,用返回值表达,让批量脚本据此决策。

> **金句**:**"任何会改用户磁盘的操作都该有两道闸门:一道挡掉无心之错(默认不覆盖),一道留住有心之失(覆盖前备份)。"**

## 6.5 name → 配置类:一张共享表,不是两份 if/elif

我们要让用户跑 `odp-gen-config train`,生成器内部得把 `"train"` 映射到 `YOLOTrainConfig`。最朴素是 if/elif:

```python
if name == "train":   cls = YOLOTrainConfig
elif name == "val":   cls = YOLOValConfig
elif name == "infer": cls = YOLOInferConfig
```

> 🤔 **思考题**:这段映射,真的只有生成器一处会用吗?

不止。阶段 7 的 `build_train_config` / `build_val_config` 还要同一份"名字 → 配置类"映射。**同一份真相要在两处用——这正是 D3 撞墙① 的漂移苗头**(同一份映射散在多个文件,改一处忘一处)。当年我们为"多格式"学的注册表思想,现在原样适用:**用一张共享的表分发,加一个模式 = 加一行表项,分发逻辑一行不改。**

```python
# runtime_config/registry.py: 唯一的"名字 → 配置类"真相
from od_platform.runtime_config.train import YOLOTrainConfig
from od_platform.runtime_config.val   import YOLOValConfig
from od_platform.runtime_config.infer import YOLOInferConfig

CONFIG_REGISTRY: dict[str, tuple[type, str]] = {
    "train": (YOLOTrainConfig, "YOLO 训练配置"),
    "val":   (YOLOValConfig,   "YOLO 验证配置"),
    "infer": (YOLOInferConfig, "YOLO 推理配置"),   # 加新模式 = 加一行
}
```

⚠️ **设计点(为什么不上 `registry_utils.import_submodules` 自动扫描)**:D3 / D4 的注册表用 `common/registry_utils` 自动扫描 `converters/` / `strategies/`,因为那两个实现集**会无限增长、第三方会加**,"加文件就自动注册、main 一行不改"才值。但 runtime_config 这里就 **3 个固定的核心模式**(train/val/infer),不是开放插件点——**一张显式 dict 就够,上自动扫描反而是过度设计**。这正是 D3 那条"什么时候不要上注册表"的判断:注册表(尤其自动发现)不是默认选项,是"实现集会持续增长"时才上的。**我们复用的是注册表的思想(一张表分发、加一项不改逻辑),不是机械照搬它最重的那套机制。** 等哪天真要支持第三方注册自定义配置类,再把这张 dict 升级成 `import_submodules` 自动扫描——接口不变,只是填表方式从"手写"变"自动发现"。

⚠️ **设计点(这张表同时喂生成器和 build_*)**:生成器 CLI 读它(`CONFIG_REGISTRY[name]`),阶段 7 的 build_* 也读它——**一处定义,两处消费,漂移被一次堵死**。这就是把"同一份真相散在多个文件"这个 D3 撞墙① 的病,在 D5 提前掐灭。

## 6.6 entry-point 在此出生:`odp-gen-config`

现在有了值得包成命令的主流程,阶段 0 故意没立的 entry-point 该登场了(对应 D1 阶段 11"python -m 太丑 → 包成干净命令")。generator 自带 `main()`,两个入口共享它:

```python
def main():
    import argparse
    parser = argparse.ArgumentParser(prog="odp-gen-config", description="生成 YOLO 运行配置 YAML 模板")
    parser.add_argument("name", choices=list(CONFIG_REGISTRY), help="配置名 (train/val/infer)")
    parser.add_argument("-o", "--output", type=Path, default=None, help="输出路径")
    parser.add_argument("--overwrite", action="store_true", help="覆盖已有文件(默认不覆盖)")
    parser.add_argument("--no-backup", action="store_true", help="覆盖时不备份(默认备份)")
    args = parser.parse_args()

    config_class, title = CONFIG_REGISTRY[args.name]              # ★ 读共享表
    output_path = args.output or runtime_config_path(args.name)  # ★ 走 paths SSoT
    ok = ConfigGenerator().generate(config_class, output_path,
                                    overwrite=args.overwrite, backup=not args.no_backup, title=title)
    print(f"✓ 已生成: {output_path}" if ok else
          f"- 文件已存在, 未覆盖. 如需重新生成加 --overwrite(会自动备份).\n  路径: {output_path}")

if __name__ == "__main__":
    main()
```

⚠️ **设计点(`choices=list(CONFIG_REGISTRY)`)**:argparse 的 `choices` 直接取自那张共享表——用户敲 `odp-gen-config foo` 立刻被 argparse 拒,**连合法模式列表都不用单独维护**,还是那张表说了算。能用 argparse 自带的,不自己造。

⚠️ **设计点(entry-point 直接绑 `generator:main`,不另立 `cli/generate_config.py`)**:generator 既是反射工具又是 CLI 主流程,自带 `main()` 即可,不必再套一层薄包装。`python -m od_platform.runtime_config.generator train` 走 `__main__` 块也调同一个 `main()`,**装包前用模块入口、装包后用命令,两个入口一份逻辑**。

# 阶段 7: 对外公共 API —— __init__ + build_* 便捷函数

零件齐了:配置类(2-3)、Loader(4)、Merger(5)、Generator(6)。但 D6 的 TrainService 要拿一份配置,难道每次都自己拼 `YAMLLoader().load()` + `CLILoader().load()` + `ConfigMerger().merge()`?那等于把管线细节漏给每个调用方。这一阶段给子系统一个**干净门面**:一行 `build_train_config(...)` 拿到配置。

## 7.1 build_train_config:把管线藏进一个函数

```python
def build_train_config(
    yaml_path: Optional[Union[str, Path]] = "train.yaml",
    cli_args:  Optional[Namespace] = None,
    *,
    dry_run: bool = False,
) -> Tuple[Optional[YOLOTrainConfig], ConfigMerger]:
    """构建训练配置: YAML + CLI → 合并 → 验证.

    Returns:
        (config, merger). config 可能为 None(dry_run 时); merger 始终返回,
        调用方可 merger.get_source_report() / to_audit_log() 拿溯源.
    """
    merger = ConfigMerger()
    sources: List[Tuple[ConfigSource, dict]] = []

    # 1. YAML 源(文件不存在 → YAMLLoader 在阶段 4 已 fail-fast + 指引)
    if yaml_path is not None:
        yaml_dict = YAMLLoader(config_dir=RUNTIME_CONFIGS_DIR).load(yaml_path)
        sources.append((ConfigSource.YAML, yaml_dict))

    # 2. CLI 源(没传就跳过)
    if cli_args is not None:
        cli_dict = CLILoader().load(cli_args)
        sources.append((ConfigSource.CLI, cli_dict))

    # 3. dry_run: 只合并不验证, 让用户先看报告
    if dry_run:
        merger.preview(YOLOTrainConfig, sources=sources)
        return None, merger

    # 4. 正式: 合并 + 验证(DEFAULT 由 merger 自动注入)
    config = merger.merge(YOLOTrainConfig, sources=sources)
    return config, merger
```

⚠️ **设计点(为什么返回 `(config, merger)` 而不只返回 config)**:配置值用户当然要,但"这些值从哪来"(来源报告 / 审计日志)也常要——尤其 D6 要把溯源写进训练日志。把 merger 一并返回,**调用方想要溯源随时 `merger.get_source_report()`,不想要就忽略**。只返回 config 会让溯源能力被门面挡死。

⚠️ **设计点(`yaml_path` 默认 `"train.yaml"`,但可显式传 None)**:大多数场景就读 `configs/runtime/train.yaml`,给个好默认省事;但 D6 想"纯 CLI、不读文件"时,传 `yaml_path=None` 即可——**好默认 + 可关闭**,跟阶段 6 的 `overwrite` 同款体感。

⚠️ **设计点(`dry_run=True` 返回 `(None, merger)`)**:dry_run 没验证,给不出一个保证合法的 config 实例,所以 config 位返回 None——**类型签名诚实地告诉调用方"这条路拿不到 config,只拿得到 merger 报告"**。用户拿 merger 看完报告,再 `dry_run=False` 真跑。

## 7.2 三个独立函数,不是 `build_config(mode=...)`

`build_val_config` / `build_infer_config` 与上面同构,只把配置类换成 `YOLOValConfig` / `YOLOInferConfig`、默认 yaml 换成 `"val.yaml"` / `"infer.yaml"`,并读那张共享表 `CONFIG_REGISTRY` 保持一致。

> 🤔 **思考题**:为什么不写一个 `build_config(mode, ...)` 用 `mode` 参数分发,而要三个几乎一样的函数?

⚠️ **设计点(返回类型精确 > 省几行)**:`build_config(mode=...)` 的返回类型只能是 `YOLOTrainConfig | YOLOValConfig | YOLOInferConfig` 联合——调用方 `cfg = build_config("train")` 后,mypy 不知道 `cfg` 具体是哪个,访问 train 专属字段就报警。三个独立函数 `build_train_config()` 返回精确的 `YOLOTrainConfig`,IDE 补全、类型检查全准。**三个模式是三种返回类型,用三个函数表达,比用一个 `mode` 开关把类型搅浑更值。** 这跟阶段 5 "`preview` vs `merge` 用两个方法名而非一个开关"同款判断。

```markdown
| 字段 | 值 |

## 1. 背景

D5 之前, legacy 把"训练/验证的配置"分散在三处: legacy/config/(多个 Pydantic 模型)、
legacy/training/(TrainService 直接实例化 + YAML 工具)、legacy/evaluation/(同模式 + 大量
重复)。字段共 60+, 训练特有约 52、验证特有约 11、共享约 15。五类问题:

1. **字段散乱、无共享层**: model/data/batch/imgsz 这种"训练和验证都用"的字段在两处各定义
   一遍, 改一个就要找两处, 已出现过 train 的 batch validator 跟 val 不一致的 bug。
2. **元数据缺失、YAML 模板手维护**: 用户拿到 yaml 不知道 epochs 默认多少、lr0 合理范围、
   task 能填啥。文档跟代码漂移, 改 yaml 全靠猜 + 跑起来报错。
3. **task 字段过载**: task="detect_val" 同时承担"任务语义(SSoT)+ 目录名 + 实验标识",
   三件事挤一个字段, 任何一个都做不好(详见 lecture 阶段 3 撞墙②)。
4. **配置来源不可追溯**: epochs=200 跑出来不收敛, 用户问"这 200 是 yaml 还是 CLI 给的",
   legacy 只记最终值, 答不出。
5. **YAML 不存在时静默生成**: legacy service.py 检测 yaml 不存在就自动生成默认继续跑,
   CI 场景 5 小时训练用了默认值而非预期值, 完全没信号(详见 lecture 阶段 4 撞墙③)。

---

## 2. 决策

把配置重构为独立的 runtime_config/ 包, 按职责拆分模块:

    runtime_config/
    ├── __init__.py    — 对外公共 API(__all__ 契约)
    ├── base.py        — BaseConfig(15 共享字段 + 验证 + 反射/mask/快照方法)
    ├── train.py       — YOLOTrainConfig(52 训练特有字段)
    ├── val.py         — YOLOValConfig(11 验证特有字段)
    ├── infer.py       — YOLOInferConfig(推理字段, source 必填)
    ├── loaders.py     — YAMLLoader + CLILoader(外部数据 → dict, fail-fast)
    ├── merger.py      — ConfigMerger(sources list 合并 + 链表式溯源 + mask + 审计)
    ├── generator.py   — ConfigGenerator(反射 → 自解释 YAML 模板, 默认不覆盖 + 备份)
    ├── registry.py    — CONFIG_REGISTRY(name→配置类, 生成器 + build_* 共享)
    └── api.py         — build_train/val/infer_config(管线门面)

### 2.1 核心设计选择

**A. Pydantic v2 + 字段级元数据(随需求生长)**
每字段 Field(default, description) + 按需补 json_schema_extra{group, examples, tips,
yaml_comment, sensitive}。关键: 元数据**不在定义之初一次性塞满**——给用户看的注释/示例/
提示等到 generator 要反射时才加, sensitive 等到报告会泄密时才加。字段定义与文档表达同源,
不漂移; 又不为"将来某下游"预埋无消费者的槽位。

**B. ClassVar 集合扩展, 不 override 方法**
FRAMEWORK_ONLY_FIELDS: ClassVar[set[str]] 控制"哪些字段不传 ultralytics";
SENSITIVE_MASK: ClassVar[str] 控制 mask 字符串。子类靠扩展集合/重新赋值类常量定制行为,
不 override 方法。加新框架字段只动一行集合。数据驱动 > 方法驱动。

**C. task / experiment_name 拆分**
两个字段表达两个正交职责: task 封闭取值 {detect,segment}(SSoT, 严格 validator, 错误信息
指引"取名用 experiment_name"); experiment_name 自由字符串(实验标识, 进 FRAMEWORK_ONLY_FIELDS,
由 service 翻成 ultralytics 的 name)。跨字段校验直接用 self.task != Task.SEGMENT 严格相等,
不用 legacy 的字符串模糊匹配。

**D. sources list 接口 + 链表式溯源**
ConfigMerger.merge(cls, *, sources=[(source_id, dict), ...]): 源数量无上限, 顺序=优先级
(从低到高), source_id 接受 ConfigSource 枚举或任意字符串(扩展自定义源 ENV/etcd 不必改 enum)。
DEFAULT 源由 merger 自动注入(反射自 Pydantic, 不另存)。ConfigMetadata.overridden_from 指针
保留完整覆盖历史, chain_str 输出 "50(CLI) ← 200(YAML) ← 100(DEFAULT)"。

**E. 加载/验证/合并/生成 四职责分离**
Loader 只读成 dict; Pydantic 只验证; Merger 只合并 + 溯源; Generator 只产模板。一个模块一件事。

**F. fail-fast + 修复指引(不静默)**
yaml 不存在 → raise FileNotFoundError + 准确路径 + "odp-gen-config <name>" 指引, 不再隐式生成。
显式自动化(odp-gen-config)保留, 隐式自动化(loader 偷偷生成)废除。

**G. 注册表思想(克制版)**
CONFIG_REGISTRY 一张共享 dict 分发 name→配置类, 生成器 CLI 与 build_* 同读, 掐灭"同一映射散
多处"的漂移。但**不上 registry_utils 的自动扫描**——3 个固定核心模式不是开放插件点, 显式 dict
足够; 待第三方需注册自定义配置类时再升级为自动发现(接口不变)。

---

## 3. 后果

**正面**: 字段单一真相、模板自解释、配置可溯源、非法输入入口即拒、加新模式近乎免费、
对 D6 暴露一行 build_* 门面。
**负面**: 引入 Pydantic v2 依赖; 字段多时元数据需逐字段补(可接受, 因与定义同源不漂移);
链表式溯源有轻微内存/构造开销(配置规模下可忽略)。
**中性**: __repr__ 明确不 mask(显式 mask_sensitive_dump 才脱敏); 同秒两次覆盖的备份命名
极端冲突未处理(概率近零, 接受边界)。

## 4. 备选方案(为什么没选)

- **Hydra**: 功能强(组合配置 / 多 run / 命令行覆盖), 但引入较重的目录约定 + 学习成本,
  且其魔法式 CLI 覆盖与本项目"溯源可查、报错指路"的诉求不完全吻合; 对一个三模式的训练配置
  属于杀鸡用牛刀。
- **OmegaConf(单用)**: 擅长分层合并, 但不带字段级验证 + 元数据, 还要再叠一层校验; Pydantic
  v2 一层把"验证 + 元数据 + 反序列化"全包了, 更内聚。
- **dataclass + 手写校验**: 校验 / 类型转换 / 从 dict 反序列化全得手写, 正是 legacy 的痛点来源;
  Pydantic 这些生来就有。
- **保留 legacy 扁平 dict 合并**: 数值上能合并, 但溯源信息全丢(背景问题 4), 不可接受。
```

# 全景回顾:现在,这棵树才长出来

到这里,`runtime_config/` 的每一个文件你都亲手把它从一个具体问题里逼了出来。现在——也只有现在——把整棵树摊开看,它的每一根枝桠你都知道**为什么在那儿**。

```
apps/platform/src/od_platform/runtime_config/
├── __init__.py     ← 阶段 7: 对外公共 API(__all__ 契约, D6 主用 build_*)
├── base.py         ← 阶段 2 立(共享字段+验证); 阶段 5 长出 mask/快照; 阶段 6 长出反射/元数据
├── train.py        ← 阶段 3: YOLOTrainConfig(撞墙② task/experiment_name 拆分)
├── val.py          ← 阶段 3: YOLOValConfig(conf=0.001, task 不传 ultralytics)
├── infer.py        ← 阶段 3: YOLOInferConfig(复用红利, source 必填)
├── loaders.py      ← 阶段 4: YAMLLoader(撞墙③ fail-fast) + CLILoader
├── merger.py       ← 阶段 5: ConfigMerger(链表式溯源 + string_utils 报告 + mask + 审计)
├── generator.py    ← 阶段 6: ConfigGenerator(反射式自解释模板 + 文件安全 + main)
├── registry.py     ← 阶段 6: CONFIG_REGISTRY(name→配置类, 生成器 + build_* 共享)
└── api.py          ← 阶段 7: build_train/val/infer_config(管线门面 + dry_run)

configs/runtime/      ← 阶段 0: 运行配置 yaml 的家(odp-gen-config 产出)
docs/architecture/ADR-005-runtime_config-subsystem.md   ← 阶段 9
```

**注意这张树出现在最后一节,不是第一节。** 这是整个 ODPlatform 系列的方法:架构不是开场白,是问题逼出来的产物。你若把它倒过来——先甩树、再填坑——就回到了 D5 旧版讲义的老路。

## 三堵墙,回头看

| 墙 | 朴素做法 | 撞到的痛 | 逼出的结构 |
|---|---|---|---|
| **撞墙①**(阶段 1) | argparse 串 60 参数 | 字段三处定义、无验证、无溯源、模板手维护 | Pydantic BaseConfig + 四个缺口 |
| **撞墙②**(阶段 3) | task 一个字段两件事 | SSoT 失守 + 实验名锁死 + 字符串匹配地狱 | task(SSoT 硬校验)/ experiment_name 拆分 |
| **撞墙③**(阶段 4) | yaml 不存在自动生成继续跑 | CI 跑 5 小时用错值、改名拼错跑了别的 | fail-fast + 修复指引(显式 ≠ 隐式自动化) |

三堵墙都不是"新模式",是"把前几天学会的思路用对地方"——这也是 D5 比 D4 更节制的原因。

## 复用红利:这一天踩在前几天的地基上

D5 没有从零造任何已有的轮子。明确点一遍,免得当成新知识:

- **`paths.py` SSoT(D1 起 / D2 ADR-001)**:`RUNTIME_CONFIGS_DIR` / `runtime_config_path` 全走它,没有一处硬编码路径(阶段 0、4、6、7)。
- **`Task` 词汇 SSoT(D3 / ADR-002)**:task 字段的合法取值直接 `from od_platform.common.constants import Task`,没另立第二份字符串集(阶段 0、3)。
- **`string_utils`(D1 撞墙⑤)**:来源报告的中文标题对齐,正是当年那堵"`.center` 数字符不数显示列"的墙——一行 `pad_to_width(..., align="center")` 复用,没重写对齐(阶段 5)。**这是"撞一次墙、造一件工具、后面每次同类问题都白捡"最直接的兑现。**
- **注册表思想(D3 撞墙①)**:`CONFIG_REGISTRY` 一张表分发 name→配置类,掐灭"同一映射散多处"的漂移(阶段 6)。

> **值得说清的一点(关于两件"没用上"的工具)**:有两件 D1 的工具——`system_utils`(设备/环境信息)和 `performance_utils`(`@time_it` 计时)——**D5 一次都没用,这是故意的**。配置子系统只"描述跑法",不"执行训练";环境快照和计时是 D6 真开训时才有的需求。**别为了"把工具用全"硬塞进配置层**——那跟 D5 旧版"先搭满骨架"是同一种答案先行(D0 决定 3:别为不存在的需求写代码)。它俩的家在 D6,到那天自然会用上。

## 收尾

你现在手里的 `runtime_config/`,能做四件 legacy 做不到的事:**字段只有一份真相**(改一处不漂移)、**非法输入进门即拒**(不拖到训练崩)、**每个值都能溯源**(`50(CLI) ← 200(YAML) ← 100(DEFAULT)`)、**模板会自己解释自己**(`odp-gen-config train` 产出带注释/示例/提示)。更重要的是,它是被一个个问题逼出来的——所以你不只是会用它,你知道**为什么它长这样**。

下一天(D6),这套配置会第一次真正驱动训练:`build_train_config(...)` 拿到的 config,喂给 TrainService 的 `model.train(**config.to_ultralytics_kwargs())`——`system_utils` / `performance_utils` 也会在那天派上用场。

> **金句**:**"好架构不是开场白里的一张图,是问题一个个逼出来的结果。等你把每一块都撞出来,再回头看那张树——它的每一根枝桠,你都知道为什么在那儿。这,就是'撞墙法'给你的东西。"**
